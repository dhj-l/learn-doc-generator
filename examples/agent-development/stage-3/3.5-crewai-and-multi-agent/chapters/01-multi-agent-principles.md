## 💡 核心概念

### Multi-Agent 的三大要素

当我们需要多个 AI Agent 协同工作时，本质上是在模拟人类团队的运作方式。与人类团队类似，多 Agent 系统的设计也需要回答三个基本问题：团队里有哪些角色？每个人负责什么任务？彼此之间如何沟通？

这三个问题对应着 Multi-Agent 的三大核心要素：

```
角色定义（Role）    — 每个 Agent 擅长什么，具有什么样的专业知识和行为风格
任务分配（Task）    — 谁负责完成哪一部分工作，任务之间的先后顺序和依赖关系
通信机制（Message） — Agent 之间如何交换信息、传递结果、请求协作
```

**为什么这三点至关重要？** 如果角色定义不清晰，两个 Agent 可能会做重复的工作，或者互相推诿。如果任务分配不合理，关键环节可能没有人负责。如果通信机制设计不当，Agent 之间的信息传递会出现丢失或延迟，导致整个系统的输出质量下降。

在实际项目中，这三个要素往往是相互影响的。例如，一个 Agent 的角色决定了它适合完成什么类型的任务，而任务的依赖关系又决定了 Agent 之间需要什么样的通信机制。

### 四种编排模式

编排模式决定了 Agent 之间的组织方式和执行流程。不同的业务场景适合不同的编排模式，选择错误的编排模式可能是导致多 Agent 系统效率低下的首要原因。以下是四种最常用的编排模式：

**Pipeline（流水线）模式**是最直观的编排方式——Agent 按固定顺序依次执行，前一个 Agent 的输出作为后一个 Agent 的输入。这种模式适合有明显阶段性特征的流程，比如内容生产（选题→研究→写作→审核）和数据处理（提取→转换→加载）。优点是流程清晰、易于调试，缺点是整体速度受最慢环节的限制。

**Debate（辩论）模式**让多个 Agent 对同一个问题从不同角度进行分析和辩论，最终由一个裁判 Agent 或投票机制做出裁决。这种模式特别适合需要多方验证的决策场景，比如代码审查、方案评审、风险评估。辩论模式的核心优势在于能够暴露单一视角下可能忽略的问题。

**Voting（投票）模式**让多个独立的 Agent 分别对同一个问题做出回答，然后通过投票或评分机制选出最佳答案。这种模式常用于提高回答的准确性和可靠性，比如医疗诊断辅助、技术方案选型。投票模式的效果取决于 Agent 的多样性和投票机制的合理性——如果所有 Agent 的训练数据相似，投票的价值就会大打折扣。

**Division of Labor（分工协作）模式**将一个大任务拆分成多个独立的小任务，分配给不同的 Agent 并行处理。这种模式的核心优势是效率——多个 Agent 同时工作，整体完成时间大幅缩短。适合的场景包括：同时分析多个数据源、并行生成不同模块的代码、同时对多个用户请求做出响应。

```typescript
// 1. Pipeline（流水线）— 串行执行
// 研究员 → 写手 → 编辑
const pipeline = [researcher, writer, editor];

// 2. Debate（辩论）— 多个 Agent 对同一问题辩论
// 正方 Agent vs 反方 Agent → 裁判 Agent
const debaters = [proAgent, conAgent];
const judge = judgeAgent;

// 3. Voting（投票）— 多个 Agent 独立回答，取共识
const voters = [agent1, agent2, agent3];
const answer = majorityVote(voters);

// 4. Division of Labor（分工协作）— 并行处理
const workers = [agent1, agent2, agent3];
const results = await Promise.all(workers.map(a => a.execute(task)));
```

**如何选择编排模式？** 在实际项目中，编排模式的选择往往取决于任务的特性。如果任务是固定流程的，比如数据清洗和报告生成，Pipeline 模式是最自然的选择。如果需要确保决策质量，可以引入 Debate 或 Voting 模式，让多个 Agent 相互验证。如果追求效率，Division of Labor 模式可以让你在最短时间内完成多个独立任务。值得注意的是，这些模式并不是互斥的——你可以在同一个系统中组合使用多种模式，比如先用 Pipeline 定义主流程，再在某些环节内部使用 Voting 模式来提高关键决策的质量。

## 🔨 实战演练

### 练习：设计一个新闻聚合 Multi-Agent 系统

**场景描述：** 你正在为一家科技媒体设计一个自动新闻生产系统，需要多个 Agent 协作完成从选题到发布的完整流程。

**你的任务：** 使用本章学到的三大要素（角色、任务、通信）和编排模式，设计一个新闻聚合系统。思考：
1. 需要哪些 Agent？各自扮演什么角色？
2. 如何分配任务？采用哪种编排模式？
3. Agent 之间如何通信和传递数据？

<details>
<summary>🧑‍💻 参考答案</summary>

**设计方案：**

**Agent 角色：**
- 📡 **选题 Agent** — 监控科技新闻源，筛选热点话题
- 🔍 **研究员 Agent** — 深度调研选定话题，收集事实信息
- ✍️ **写手 Agent** — 根据调研结果撰写新闻稿
- 🎨 **编辑 Agent** — 审核内容质量、修正错误、优化可读性

**编排模式（Pipeline）：**
```
选题 Agent → 研究员 Agent → 写手 Agent → 编辑 Agent → 发布
```

**通信机制：**
每个 Agent 的输出作为下一个 Agent 的输入，通过共享的「文章对象」传递数据：
- 选题 Agent → 输出：`{ topic, sources, keywords }`
- 研究员 Agent → 输出：`{ research_notes, facts, interviews }`
- 写手 Agent → 输出：`{ draft, word_count }`
- 编辑 Agent → 输出：`{ final_draft, status }`

</details>

## ⚡ 进阶技巧

### 技巧一：何时使用 Multi-Agent 架构？

| 适合 Multi-Agent | 适合单一 Agent |
|-----------------|----------------|
| 任务可拆分为多个独立专业技能 | 任务简单、步骤固定 |
| 需要多轮审核和迭代 | 一次性生成即可 |
| 不同步骤需要不同的提示策略 | 使用同一套提示逻辑 |
| 输出需要多角度验证 | 单一正确输出即可 |

### 技巧二：常见的 Anti-Pattern（反面模式）

在实际项目中，我们观察到了一些常见的多 Agent 设计错误。了解这些反面模式可以帮助你在设计系统时避开这些坑：

- **过度 Agent 化** — 一个简单的翻译任务拆成 5 个 Agent，反而降低了效率。经验法则是：如果一个任务可以用 3-5 行提示词让单个 Agent 完成，就不需要拆成多个 Agent。
- **角色定义模糊** — Agent 的 role 和 goal 过于笼统，Agent 不知道自己的职责边界。比如「你是一个助手」这种定义几乎没有任何约束力，结果可能是多个 Agent 做重复的工作。
- **通信过载** — Agent 之间传递了完整的对话历史，导致 Token 爆炸。一个好的实践是只传递结构化摘要而非原始对话内容，比如让每个 Agent 输出一个包含关键信息的 JSON 对象，而不是长篇大论的文本。

## 🧠 知识检查点

<details>
<summary>点击展开答案</summary>

**Q1：Multi-Agent 系统的三大核心要素是什么？**

> A：角色定义（Role）——每个 Agent 擅长什么，有什么专业知识；任务分配（Task）——谁负责做什么，任务之间的依赖关系；通信机制（Communication）——Agent 之间如何交换数据和结果。

**Q2：四种编排模式分别适用于什么场景？**

> A：Pipeline（流水线）适用于有固定顺序的生产流程；Debate（辩论）适用于需要多角度分析的问题；Voting（投票）适用于需要提高准确率的决策；Division of Labor（分工协作）适用于可并行的独立任务。

**Q3：为什么说「过度 Agent 化」是一个常见的反面模式？**

> A：每个 Agent 的调用都会消耗 LLM Token 和增加延迟。将一个简单任务拆成过多 Agent 会引入不必要的通信开销和复杂度。经验法则：能用单个 Agent 解决的问题，就不要用多个。

</details>

## 🐛 常见错误

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| Agent 角色定义过于宽泛 | role 和 goal 写得太笼统 | 用「你是一个 X 领域的 Y 专家，擅长 Z」精确限定 |
| 下游 Agent 拿不到上游结果 | Task 的 context 依赖未正确配置 | 检查 Task 的 context 参数，确保引用了前置 Task |
| Agent 之间无限循环执行 | 没有设置 max_iter 上限 | 设置合理的最大迭代次数（3-5 次） |
| 多 Agent 系统 Token 消耗惊人 | 每个 Agent 都在重复传递完整上下文 | 精简 Agent 之间的通信内容，只传递必要的结构化数据 |
