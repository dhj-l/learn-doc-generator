## 💡 核心概念

### 概念一：Capstone 项目的本质

**生活类比：** Capstone 项目就像烹饪学校的「毕业考试」——前面你学过了刀工（提示工程）、火候（API 调用）、调味（工具使用）、摆盘（前端集成），现在需要独立完成一道完整的宴席菜品。老师不会告诉你每一步该做什么，而是给出要求和评分标准，由你来决定菜品的设计和实现。

这个项目最重要的不是"用到了所有技术"，而是**做出一个可用的、有实际价值的产品**。一个完成度 80% 的可用产品，远远好于一个"完美"但只完成了 40% 的半成品。

### 概念二：需求分析框架

在做任何编码之前，先使用以下框架明确需求：

```
功能需求（必须做）
├── 核心功能 1: [用户场景] → [系统行为]
├── 核心功能 2: [用户场景] → [系统行为]
└── ...

技术需求（必须满足）
├── 前端: Vue 3 / React + TypeScript ✅
├── Agent: LangGraph / Vercel AI SDK ✅
├── 工具: ≥ 3 个 MCP Server ✅
├── 部署: 云端可公开访问 ✅
└── 安全: 输入验证 + 权限控制 ✅

加分功能（有时间再做）
├── 单元测试覆盖 > 60%
├── CI/CD 自动部署
└── 使用统计面板
```

#### 需求分析实战模板

```typescript
// 需求文档模板
interface ProjectRequirement {
  name: string;
  description: string;
  priority: 'P0' | 'P1' | 'P2';  // P0=必须有, P1=建议有, P2=锦上添花
  effort: 'S' | 'M' | 'L' | 'XL'; // 预估工时
  dependencies: string[];           // 依赖的其他功能
}

// 示例：智能客服系统的需求
const requirements: ProjectRequirement[] = [
  {
    name: '知识库问答',
    description: '用户可以通过自然语言提问，系统从知识库中检索相关文档并生成回答',
    priority: 'P0',
    effort: 'XL',
    dependencies: ['向量数据库', 'RAG 流水线', '流式输出'],
  },
  {
    name: '多轮对话',
    description: 'Agent 能记住对话上下文，支持追问和澄清',
    priority: 'P0',
    effort: 'M',
    dependencies: ['记忆系统'],
  },
  {
    name: '工单自动创建',
    description: '当 Agent 无法解决问题时，自动创建工单并转接人工',
    priority: 'P1',
    effort: 'M',
    dependencies: ['知识库问答'],
  },
];
```

**💡 为什么需求分析如此重要？** 大多数项目失败不是因为技术做不到，而是需求不清晰导致的"范围蔓延"（Scope Creep）。花 30 分钟做需求分析，可以节省 3 小时的返工时间。一个清晰的需求文档是你的项目地图，没有地图就开始旅行，你大概率会迷路。

### 概念三：四种项目深度对比

**生活类比：** 选择 Capstone 项目就像选择健身计划。智能代码助手是「力量训练」（见效快、实用性强），AI 研究助手是「耐力训练」（需要更多时间、收益期长），智能客服系统是「瑜伽」（门槛低、适合入门），工作流自动化平台是「铁人三项」（全方位挑战、成就感最强）。最好的健身计划不是最难的，而是你能坚持做完的那个。

### 概念四：项目选择决策框架

```typescript
// 用这个简单的评分表来选择项目
interface DecisionMatrix {
  currentSkills: {
    vue: number;      // 1-5
    react: number;    // 1-5
    nodejs: number;   // 1-5
    langchain: number; // 1-5
    devops: number;   // 1-5
  };
  interests: {
    codeTools: number;
    research: number;
    customerService: number;
    workflow: number;
  };
  timeAvailable: number; // 小时
}

function recommendProject(you: DecisionMatrix): string {
  if (you.timeAvailable < 10) return 'C: 智能客服系统（最省时）';
  if (you.interests.workflow > 4) return 'D: 工作流自动化平台（最具挑战）';
  if (you.interests.codeTools > 4 && you.currentSkills.langchain > 3) return 'A: 智能代码助手';
  if (you.interests.research > 4) return 'B: AI 研究助手';
  return 'C: 智能客服系统（最稳妥）';
}
```

**💡 为什么提供了 4 个选项？** Capstone 项目的首要目标是"完成"而不是"完美"。不同的技术背景和可用时间决定了哪个项目最适合你。宁可选择一个简单项目完成 100%，也不要选择一个复杂项目只完成 30%。

## ⚡ 进阶技巧

### 技巧一：在需求阶段就考虑边界情况

好的需求文档不仅描述"正常流程"，还要定义"异常流程"：

```typescript
// 异常场景定义
const edgeCases = [
  {
    scenario: '用户输入为空',
    expectedBehavior: '提示用户输入内容，不调用 LLM',
  },
  {
    scenario: 'LLM API 返回错误',
    expectedBehavior: '显示友好错误消息，保留用户输入不丢失',
  },
  {
    scenario: '搜索结果为空',
    expectedBehavior: '告知用户未找到相关信息，建议修改关键词',
  },
  {
    scenario: '用户连续快速提交（10次/秒）',
    expectedBehavior: '前端防抖 + 后端限流，防止滥用',
  },
];
```

### 技巧二：使用 AI 辅助需求分析

你不必完全靠脑力做需求分析——让 AI 帮你评审：

```
推荐 Prompt 模板：

"我正在规划一个 AI Agent Capstone 项目，主题是 [项目名]。
我列出的核心需求是：
1. [需求1]
2. [需求2]
...

请帮我评审这份需求文档，指出：
1. 遗漏了哪些关键功能？
2. 哪些需求的优先级应该调整？
3. 有什么潜在的实现风险？

我的技术栈是：Vue 3 + Node.js + LangGraph"
```

### 技巧三：MVP（最小可行产品）思维

不要试图在第一版实现所有功能。按照以下优先级曲线制定计划：

```
迭代 1（前 40% 时间）→ 核心链路跑通
  ├── Agent 能回答一个问题 ✅
  ├── 前端能显示回复 ✅
  └── 能搜索知识库 ✅

迭代 2（中间 35% 时间）→ 体验完善
  ├── 流式输出 ✅
  ├── 多轮对话 ✅
  └── 错误处理 ✅

迭代 3（最后 25% 时间）→ 锦上添花
  ├── 部署上线 ✅
  ├── 美化 UI ✅
  └── 加分功能 ✅
```

## 🧠 知识检查点

<details>
<summary>🧠 Q1：P0（必须做）和 P1（建议做）的需求有什么区别？如何判断？</summary>

P0 需求：如果没有这个功能，产品就完全不可用。例如：客服系统不能回答问题 = 产品无意义。判断标准：删除这个功能，用户的主要使用场景还能成立吗？

P1 需求：有这个功能体验更好，但没有也能用。例如：客服系统的工单自动创建。判断标准：这个功能可以作为一个独立迭代版本发布吗？
</details>

<details>
<summary>🧠 Q2：为什么要先估算时间再开始项目？</summary>

原因有三：
1. **防止范围蔓延** — 明确 12 小时内能做什么、不能做什么
2. **指导取舍** — 时间不够时，知道该砍掉哪些 P2 功能
3. **保持动力** — 有明确的时间线，知道自己进度正常还是落后了
</details>

<details>
<summary>🧠 Q3：在选择项目时，"技术挑战"应该排在什么优先级？</summary>

技术挑战应该排在第三位：第一位是**实用性**（做出来有人用），第二位是**可行性**（以你的技术栈能完成），第三位才是挑战性。一个超出你当前能力 20% 的项目是最好的学习机会，但如果超出 100%，很可能半途而废。
</details>

<details>
<summary>🧠 Q4：需求分析阶段需要和部署方案结合考虑吗？</summary>

需要！需求直接决定了部署方案：
- 如果项目需要 WebSocket 长连接（如实时协作），Vercel Functions 不支持
- 如果项目需要上传大文件，Lambda 有 6MB 请求体限制
- 如果项目需要定时任务，需要额外的 Cron 服务

建议在需求文档中标注每个功能的"部署约束"。
</details>

## 🐛 常见错误

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| 选择了超出能力范围的项目 | 高估了自己的技术栈成熟度 | 使用决策矩阵评估，选择契合度最高的项目 |
| 功能列表写得太模糊 | 需求描述使用"高效"、"智能"等主观词汇 | 每个需求写成"当用户做 X 时，系统应该做 Y"的格式 |
| 没有定义"不做"的范围 | 未明确排除哪些功能，导致范围不断扩展 | 明确写出"本项目不做 X、Y、Z" |
| 时间估算过于乐观 | 只算了编码时间，忽略了调试和文档 | 编码:调试:文档 = 5:3:2 的比例分配时间 |
| 忽略了部署环境的限制 | 选择了 Vercel 但需要长连接 WebSocket | 在需求分析阶段就明确部署约束 |
| 前置知识未复习就开始了 | 对 LangGraph 不熟但直接开始编码 | 花 30 分钟快速复习关键知识点的代码示例 |
| 一个人做多人项目 | 选择了一个需要前后端 + 移动端的复杂项目 | Capstone 选择 2-3 人月的单人或双人项目 |

## 📝 本章小结

- ✅ **选择项目** — 使用决策矩阵"实用性 → 可行性 → 挑战性"做选择
- ✅ **需求分析** — P0 核心功能 + P1 重要功能 + P2 加分功能，三层清晰分层
- ✅ **时间规划** — 12-15 小时分配到需求:架构:实现:部署 = 1:2:6:2 的比例
- ✅ **MCP 工具** — 每个项目至少集成 3 个 MCP Server，其中至少 1 个自定义的
