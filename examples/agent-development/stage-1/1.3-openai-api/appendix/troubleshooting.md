# OpenAI API 常见错误排错指南

---

## 1. API Key 无效 (401)

**现象：** `401 Unauthorized` 或 `Incorrect API key`
**原因：** API Key 未设置、格式错误、或已过期
**解决方案：**
- 检查 `OPENAI_API_KEY` 环境变量是否已正确设置
- 确认 Key 格式正确（以 `sk-` 开头）
- 不同提供商使用不同的 Key 和 `baseURL`，确保两者配套

---

## 2. 模型不存在 (404)

**现象：** `model_not_found` 错误
**原因：** 模型名称拼写错误，或该账户没有访问该模型的权限
**解决方案：**
- 确认模型名称拼写正确（如 `gpt-4o` 而非 `gpt4o`）
- 国产模型的 model ID 与 OpenAI 不同，查阅对应文档获取正确的 ID
- 检查账户是否有该模型的访问权限（部分新模型需要单独申请）

---

## 3. 速率限制 (429)

**现象：** `Rate limit exceeded`，请求被拒绝
**原因：** 短时间内发送了太多请求，超过了 API 限制
**解决方案：**
- 降低并发请求数
- 使用 SDK 内置重试：`new OpenAI({ maxRetries: 3 })`
- 实现指数退避：失败后等待 `2^n` 秒再重试
- 升级账户 tier 获取更高的速率限制

---

## 4. 结构化输出 Schema 不匹配

**现象：** 输出的 JSON 不完全符合定义的 Schema
**原因：** 使用了不支持 Structured Outputs 的旧模型，或 Schema 定义有歧义
**解决方案：**
- 使用 `gpt-4o-2024-08-06` 或更新的模型
- 确保 Schema 中设置 `additionalProperties: false`
- 所有字段都列在 `required` 中
- 使用 `.describe()` 为每个字段提供清晰的语义描述

---

## 5. 国产模型调用失败

**现象：** 使用兼容 API 调用国产模型时返回各种错误
**常见原因：**
- `baseURL` 配置错误（注意是否有 `/v1` 后缀）
- API Key 格式与 OpenAI 不同（如 DeepSeek 以 `sk-` 开头但格式不同）
- 某些模型不支持某些参数（如 `response_format`、`tool_choice`）
**解决方案：** 查阅各模型提供商的文档，确认支持的参数和 endpoint

---

## 6. 流式输出中文乱码

**现象：** 流式接收的中文内容显示为乱码
**原因：** 终端编码不是 UTF-8，或使用了不合适的输出方法
**解决方案：** 确保终端编码为 UTF-8，使用 `process.stdout.write()` 而非 `console.log()`

---

## 7. Token 消耗超出预期

**现象：** API 账单远高于预估
**原因：** System Prompt 过长、对话历史不断累积、或设置了过大的 `max_tokens`
**解决方案：**
- 检查 `usage` 字段的 `prompt_tokens`，看哪些部分占了最多 Token
- 精简 System Prompt，删除冗余内容
- 对长对话使用摘要压缩
- 设置合理的 `max_tokens`，不要超过实际需要

---

## 8. 国产模型输出截断

**现象：** 模型回答突然中断，在句子中间断开
**原因：** 国产模型的 `max_tokens` 上限较低（如 4096 vs OpenAI 的 16384）
**解决方案：**
- 查阅模型文档确认输出 Token 上限
- 分多次调用，每次处理一个子任务
- 使用流式输出提前看到已生成的内容

---

## 9. 兼容接口返回格式与 OpenAI 不同

**现象：** 国产模型的返回结构缺少某些字段（如 `logprobs`、`finish_reason`）
**原因：** 兼容接口只实现了 OpenAI API 的子集
**解决方案：** 在代码中做 null 安全检查，不要假设所有字段都存在

---

## 10. 并发请求导致数据竞争

**现象：** 多个请求同时修改同一份对话历史，导致上下文错乱
**原因：** 单一线程中多个异步请求交叉写入
**解决方案：** 为每个对话会话维护独立的消息数组；使用锁或队列控制并发写入

---

## 11. 多模型网关模型选择错误

**现象：** 简单问答被路由到昂贵的模型，复杂任务被路由到便宜的模型
**原因：** 路由策略过于简单，没有根据任务类型匹配模型
**解决方案：** 实现基于 taskType（chat/code/analysis）的多级路由，配合预算约束

---

## 12. 超时设置不当导致请求失败

**现象：** 长时间运行的请求在中间被断开
**原因：** HTTP 客户端默认超时太短，或流式连接在空闲时被关闭
**解决方案：** 设置合理的 timeout（如 60 秒），流式请求使用 `stream: true` 并在每次收到数据时重置超时

---

## 13. 使用过时的模型 API 版本

**现象：** 新功能在文档上有但调用时报错
**原因：** 使用了旧的 API 版本，新功能需要新的 API 版本支持
**解决方案：** 在请求头中添加 `OpenAI-Beta` 指定版本，或升级 SDK 到最新版本

---

## 14. 函数调用（Function Calling）参数错误

**现象：** 模型调用了不存在的函数，或传递了错误的参数
**原因：** 函数定义中有语法错误，或参数描述不清导致模型误解
**解决方案：** 检查 `tools` 定义的 JSON Schema 是否合法；用 `.describe()` 清晰描述每个参数的用途

---

## 15. Embedding 返回结果异常

**现象：** Embedding 向量的值全是 0 或 NaN
**原因：** 输入文本为空，或包含了模型无法处理的特殊字符
**解决方案：** 检查输入文本是否为空，移除不可见控制字符；确保文本编码为 UTF-8

---

## 16. 批量请求（Batch API）状态卡住

**现象：** 批量请求提交后长时间处于 `in_progress` 状态
**原因：** Batch API 是异步队列，处理时间取决于队列长度和请求量
**解决方案：** 使用轮询（每 30 秒检查一次），设置合理的超时（如 1 小时）；大批量请求分多次提交

---

## 17. 多模态请求中图片无法识别

**现象：** 发送图片 URL 或 Base64 后模型返回错误
**原因：** 图片格式不被支持、图片过大、或 URL 无法访问
**解决方案：** 确认图片格式为 PNG/JPEG/WebP/GIF（非 SVG）；压缩图片到 20MB 以下；确保 URL 可公开访问

---

## 18. 成本追踪与实际账单不一致

**现象：** 自己统计的 Token 消耗和 OpenAI 账单对不上
**原因：** 未计入缓存命中/未命中的差异、不同模型的定价不同、或忽略了 `max_tokens` 预占的成本
**解决方案：** 使用 API 返回的 `usage` 字段精确计算，而非预估；区分 prompt 和 completion 的 Token 单价
