LangChain.js 开发中 18 个高频错误的原因分析与解决方案

📋 使用说明
本文档收录了 LangChain.js TypeScript 开发中最常遇到的错误和问题。每个条目包含错误信息、原因分析、解决步骤和示例代码，帮助你快速定位并解决问题。建议使用浏览器的 Ctrl+F 搜索错误关键字来快速定位。

🔧 环境与安装

❌ Node.js 版本过低导致 ERR_REQUIRE_ESM

错误信息：Error [ERR_REQUIRE_ESM]: Must use import to load ES Module

原因分析：LangChain.js 从 v0.1.0 起全面采用 ESM 模块系统，要求 Node.js 版本 >= 18。当你的 Node.js 版本过低（如 v14 或 v16），或者在 CommonJS 项目中直接 require() LangChain 模块时，就会触发此错误。

解决方案：

  1. 确认 Node.js 版本 >= 18.18.0，推荐使用 v20 LTS
  2. 如果项目需要 CommonJS，在 tsconfig.json 中配置 "module": "NodeNext" 并使用 .mts 扩展名
  3. 或在 package.json 中声明 "type": "module"
TypeScript
// tsconfig.json — 推荐配置
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "esModuleInterop": true,
    "strict": true,
    "outDir": "./dist"
  },
  "include": ["src"]
}

// package.json — 确保声明为 ESM 项目
{
  "name": "my-langchain-app",
  "type": "module",
  "engines": {
    "node": ">=18.18.0"
  }
}

❌ npm install 报错 peer dependency conflicts

错误信息：npm ERR! ERESOLVE unable to resolve dependency tree

原因分析：LangChain 生态由多个包组成（@langchain/core、@langchain/openai、langchain 等），它们之间有严格的版本对齐要求。当项目中已有其他依赖与 LangChain 的 peer dependency 冲突时，npm 会拒绝安装。

解决方案：

  1. 使用 --legacy-peer-deps 跳过 peer dependency 检查（快速方案）
  2. 锁定整个 LangChain 生态的版本范围，使用 overrides 字段统一管理
  3. 推荐使用 pnpm 或 yarn 管理依赖，它们对 monorepo 结构的兼容性更好
Bash
# 方案一：跳过 peer dependency 检查
npm install langchain @langchain/core @langchain/openai --legacy-peer-deps

# 方案二：在 package.json 中使用 overrides 强制统一版本
# "overrides": {
#   "@langchain/core": "^0.3.0"
# }

# 方案三（推荐）：使用 pnpm
npm install -g pnpm
pnpm add langchain @langchain/core @langchain/openai

❌ TypeScript 编译错误：模块解析失败

错误信息：TS2307: Cannot find module '@langchain/openai' or its corresponding type declarations

原因分析：TypeScript 的 moduleResolution 配置不正确。LangChain 的各个包使用 exports 字段映射子路径，需要 "NodeNext" 或 "Bundler" 模式才能正确解析。如果使用旧版的 "Node" 模式则无法识别。

解决方案：

  1. 在 tsconfig.json 中设置 "moduleResolution": "NodeNext"
  2. 确认 @types/node 版本 >= 18
  3. 删除 node_modules 和 tsconfig.tsbuildinfo 后重新安装
TypeScript
// tsconfig.json — 正确配置
{
  "compilerOptions": {
    "module": "NodeNext",
    "moduleResolution": "NodeNext",  // 关键：必须是 NodeNext 或 Bundler
    "target": "ES2022",
    "esModuleInterop": true,
    "strict": true,
    "skipLibCheck": true
  }
}

// 然后重新安装依赖
// rm -rf node_modules package-lock.json tsconfig.tsbuildinfo
// npm install

🔑 API 密钥与认证

❌ OpenAI API Key 未设置

错误信息：OpenAI API key not set. Set the OPENAI_API_KEY environment variable or pass apiKey to the constructor.

原因分析：LangChain 的 OpenAI 模型默认从环境变量 OPENAI_API_KEY 读取密钥。如果你没有在环境中设置此变量，也没有在代码中显式传入 apiKey 参数，就会触发此错误。

解决方案：

  1. 推荐：使用 .env 文件配合 dotenv 管理密钥
  2. 或在代码中直接传入 apiKey 参数（不推荐用于生产环境）
  3. 确保 .env 文件已加入 .gitignore，避免泄露密钥
TypeScript
// 方案一（推荐）：通过 .env 文件
// .env 文件内容：
// OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxx

import "dotenv/config"; // 在入口文件最顶部引入
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({ model: "gpt-4o" });

// 方案二：直接传入参数
const model2 = new ChatOpenAI({
  model: "gpt-4o",
  apiKey: "sk-xxxxxxxxxxxxxxxxxxxxxxxx", // 不推荐硬编码
});

❌ API Key 无效 — 401 Unauthorized

错误信息：AuthenticationError: 401 Incorrect API key provided: sk-***xxx

原因分析：传入的 API Key 格式不正确、已过期、被撤销，或者复制时多出了空格/换行符。有时在 Windows 系统中，.env 文件可能包含隐藏的 BOM 字符，导致密钥读取异常。

解决方案：

  1. 到 OpenAI 控制台重新生成一个 API Key
  2. 确认 .env 文件中的密钥没有多余的空格或引号
  3. 在代码中打印密钥的前缀和长度进行调试
TypeScript
// 调试：检查密钥是否正确加载
const key = process.env.OPENAI_API_KEY;
console.log(`Key prefix: ${key?.slice(0, 7)}...`);
console.log(`Key length: ${key?.length}`); // 通常应该是 51 或 56

// .env 文件正确格式（注意：不要加引号！）
// OPENAI_API_KEY=sk-proj-xxxxxxxxxxxxxxxxxxxx
//
// 错误示范：
// OPENAI_API_KEY="sk-proj-xxxxxxxxxxxxxxxxxxxx"  ← 不要加引号
// OPENAI_API_KEY = sk-proj-xxxxxxxxxxxxxxxxxxxx   ← 等号两边不要空格

❌ 速率限制 — 429 Too Many Requests

错误信息：RateLimitError: 429 You exceeded your current quota, please check your plan and billing details

原因分析：你的 API 调用频率或 Token 消耗速度超过了 OpenAI 账户的速率限制（Rate Limit）。免费账户和低级别账户的限制更为严格。在开发调试时，如果循环调用或并发请求过多，很容易触发此错误。

解决方案：

  1. 在 ChatOpenAI 构造函数中配置 maxRetries 和 maxConcurrency
  2. 使用 LangChain 内置的速率限制器（RateLimiter）
  3. 升级 OpenAI 账户的使用层级以获得更高的速率限制
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import { InMemoryRateLimiter } from "@langchain/core/utils/async_caller";

// 配置自动重试与并发控制
const model = new ChatOpenAI({
  model: "gpt-4o",
  maxRetries: 3,          // 失败后最多重试 3 次（自带指数退避）
  maxConcurrency: 2,      // 同时最多 2 个并发请求
});

// 或使用独立的速率限制器
const rateLimiter = new InMemoryRateLimiter({
  requestsPerSecond: 5,   // 每秒最多 5 次请求
});

// 在批量处理时添加延迟
for (const input of batchInputs) {
  await rateLimiter.acquire();
  const result = await model.invoke(input);
  // 处理结果...
}

🤖 模型调用

❌ 模型名称错误 — Model Not Found

错误信息：NotFoundError: 404 The model 'gpt-5-turbo' does not exist

原因分析：指定的模型名称拼写错误，或者使用了当前 API 提供商尚不可用的模型。OpenAI 的模型名称是区分大小写且经常更新的，错误的名称会直接返回 404。

解决方案：

  1. 查阅 OpenAI 官方文档确认当前可用的模型列表
  2. 确认你使用的 API 版本支持该模型
  3. 检查是否混用了不同提供商的模型名称（如把 Anthropic 的名称传给了 OpenAI）
TypeScript
import { ChatOpenAI } from "@langchain/openai";

// 常见的正确模型名称（截至 2025 年）
const model = new ChatOpenAI({
  model: "gpt-4o",           // GPT-4o 多模态模型
  // model: "gpt-4o-mini",   // GPT-4o Mini（经济型）
  // model: "gpt-4-turbo",   // GPT-4 Turbo
  // model: "o1",            // o1 推理模型
  // model: "o3-mini",       // o3-mini 推理模型
});

// 不要使用错误名称：
// model: "gpt-5"            ← 不存在
// model: "gpt4o"            ← 少了连字符
// model: "GPT-4o"           ← 大小写不对

❌ 请求超时 — Timeout

错误信息：TimeoutError: Request timed out after 60000ms 或 ConnectionError: Request failed with status code 408

原因分析：模型推理耗时超过了默认超时时间（通常为 60 秒）。这在使用大型模型处理长文本、复杂推理任务，或者网络状况不佳时容易出现。某些模型（如 o1 系列）的推理时间本身就较长。

解决方案：

  1. 增大 timeout 参数
  2. 减少单次请求的输入长度
  3. 对于长时间任务，考虑使用流式传输来保持连接活性
TypeScript
import { ChatOpenAI } from "@langchain/openai";

// 增大超时时间
const model = new ChatOpenAI({
  model: "o1",
  timeout: 300_000,  // 5 分钟超时
});

// 对于推理模型，建议配合流式传输
const streamingModel = new ChatOpenAI({
  model: "gpt-4o",
  timeout: 120_000,
  streaming: true,
});

// 流式调用，可以逐步看到输出，避免长时间无响应
const stream = await streamingModel.stream("请详细分析这段代码...");
for await (const chunk of stream) {
  process.stdout.write(chunk.content.toString());
}

❌ Token 超限 — Context Length Exceeded

错误信息：BadRequestError: 400 This model's maximum context length is 128000 tokens. However, your messages resulted in 150000 tokens.

原因分析：发送给模型的消息总长度（包括系统提示、历史消息和用户输入）超过了该模型的最大上下文窗口限制。例如 GPT-4o 的上下文窗口为 128K tokens，当累积的对话历史过长时就会触发。

解决方案：

  1. 使用 Tiktoken 在发送前估算 Token 数量
  2. 使用 trimMessages 工具自动裁剪历史消息
  3. 考虑使用更长上下文窗口的模型
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import {
  trimMessages,
  HumanMessage,
  AIMessage,
  SystemMessage,
} from "@langchain/core/messages";

// 配置消息裁剪器
const trimmer = trimMessages({
  maxTokens: 10000,                  // 保留最近 10000 tokens 的消息
  strategy: "last",                  // 从最新的消息开始保留
  tokenCounter: (msgs) => msgs.length * 4, // 简单估算；生产环境用 tiktoken
  includeSystem: true,               // 始终保留 SystemMessage
  allowPartial: false,               // 不允许截断单条消息
});

const model = new ChatOpenAI({ model: "gpt-4o" });

const messages = [
  new SystemMessage("你是一个有帮助的助手。"),
  ...historyMessages,  // 历史消息可能很长
  new HumanMessage("我的问题是什么？"),
];

// 裁剪后再发送
const trimmed = await trimmer.invoke(messages);
const response = await model.invoke(trimmed);

🕷️ Agent 与工具

❌ Agent 无限循环 — Max Iterations Exceeded

错误信息：GraphRecursionError: Recursion limit of 25 reached without hitting a stop condition

原因分析：Agent 在执行过程中反复调用同一个工具或在几个工具之间来回跳转，始终无法得到最终答案。这通常是因为工具的描述不明确、返回结果不符合预期，或者任务本身需要的步骤超过了默认的迭代上限（25 次）。

解决方案：

  1. 优化工具描述（description），让 Agent 能准确判断何时使用
  2. 在 createReactAgent 中配置 recursionLimit
  3. 为工具添加明确的错误提示，帮助 Agent 自我修正
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// 工具描述要清晰、具体，避免歧义
const searchTool = tool(
  async ({ query }) => {
    // 搜索逻辑...
    return JSON.stringify({ results: [...] });
  },
  {
    name: "web_search",
    description:
      "搜索互联网获取实时信息。当用户询问时事新闻、实时数据、" +
      "你不知道的信息时使用此工具。不要用于数学计算或代码生成。",
    schema: z.object({
      query: z.string().describe("搜索关键词，建议使用英文"),
    }),
  }
);

const model = new ChatOpenAI({ model: "gpt-4o" });

const agent = createReactAgent({
  llm: model,
  tools: [searchTool],
  // 不要在 prompt 中设置 recursionLimit，而是在调用时指定
});

const result = await agent.invoke(
  { messages: [{ role: "user", content: "今天天气如何？" }] },
  { recursionLimit: 15 }  // 将默认的 25 降低到 15
);

❌ 工具 Schema 不匹配 — Zod Validation Error

错误信息：ZodError: [ { "code": "invalid_type", "expected": "string", "received": "number", "path": ["temperature"] } ]

原因分析：LLM 返回的工具调用参数类型与 Zod schema 定义的类型不一致。例如 schema 要求 temperature 是 z.number()，但模型返回了字符串 "36.5"。这是 LLM 工具调用中最常见的问题之一。

解决方案：

  1. 在 Zod schema 中使用 .describe() 为每个字段添加清晰的描述
  2. 对于可能为多种类型的字段，使用 z.coerce 进行类型强制转换
  3. 在工具函数内部添加类型检查和转换逻辑
TypeScript
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// 错误示范：类型定义过于严格
const badTool = tool(
  async ({ temp }) => `温度是 ${temp}°C`,
  {
    name: "get_weather",
    description: "查询天气",
    schema: z.object({
      temp: z.number(),  // LLM 可能传入字符串 "36.5"
    }),
  }
);

// 正确示范：使用 coerce 和详细的 describe
const weatherTool = tool(
  async ({ city, unit }) => {
    // 即使类型防御如此，也建议在内部做检查
    const temp = 25; // 模拟查询结果
    return `${city} 当前温度: ${temp}°${unit === "celsius" ? "C" : "F"}`;
  },
  {
    name: "get_weather",
    description: "查询指定城市的当前天气信息",
    schema: z.object({
      city: z.string().describe("城市名称，如 'Beijing'、'Shanghai'"),
      unit: z
        .enum(["celsius", "fahrenheit"])
        .default("celsius")
        .describe("温度单位，默认摄氏度"),
    }),
  }
);

❌ 工具函数报错未捕获导致 Agent 崩溃

错误信息：ToolException: Error: connect ECONNREFUSED 127.0.0.1:5432 或整个 Agent 直接抛出未处理异常

原因分析：工具函数内部抛出了异常（如数据库连接失败、API 调用出错、文件不存在等），但没有被捕获处理。默认情况下，未捕获的工具异常会导致整个 Agent 执行中断。

解决方案：

  1. 在工具函数内部使用 try-catch 捕获所有异常
  2. 返回结构化的错误信息，让 Agent 有机会理解失败原因并重试
  3. 使用 returnDirect 或自定义错误处理策略
TypeScript
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// 错误示范：不捕获异常
const badTool = tool(
  async ({ query }) => {
    const res = await fetch(`https://api.example.com/search?q=${query}`);
    return await res.text(); // 如果 fetch 失败，整个 Agent 崩溃
  },
  { name: "search", description: "搜索", schema: z.object({ query: z.string() }) }
);

// 正确示范：捕获异常并返回有用信息
const safeTool = tool(
  async ({ query }) => {
    try {
      const res = await fetch(`https://api.example.com/search?q=${query}`);
      if (!res.ok) {
        return `搜索失败：服务器返回状态码 ${res.status}。请尝试换个关键词重试。`;
      }
      const data = await res.json();
      return JSON.stringify(data);
    } catch (error) {
      const msg = error instanceof Error ? error.message : String(error);
      return `搜索工具执行出错：${msg}。请检查网络连接后重试。`;
    }
  },
  {
    name: "safe_search",
    description: "安全搜索工具",
    schema: z.object({
      query: z.string().describe("搜索关键词"),
    }),
  }
);

💬 消息与上下文

❌ 消息格式错误

错误信息：TypeError: Cannot read properties of undefined (reading 'content') 或 ValidationError: Invalid message format

原因分析：传入 model.invoke() 的消息格式不符合 LangChain 的要求。常见问题包括：直接传入纯字符串数组而不是消息对象、角色名称拼写错误、缺少必需的 content 字段等。

解决方案：

  1. 使用 LangChain 提供的消息类（HumanMessage、AIMessage、SystemMessage）
  2. 如果使用 OpenAI 格式的对象，确保 role 和 content 字段都存在
  3. 使用 convertToOpenAIMessages 工具进行格式转换
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import {
  HumanMessage,
  AIMessage,
  SystemMessage,
} from "@langchain/core/messages";

const model = new ChatOpenAI({ model: "gpt-4o" });

// 错误示范
// await model.invoke("你好");                           ← 纯字符串在某些版本可能不工作
// await model.invoke([{ role: "user" }]);               ← 缺少 content
// await model.invoke([{ type: "human", text: "你好" }]); ← 字段名错误

// 正确示范：使用消息类（推荐）
await model.invoke([
  new SystemMessage("你是一个有帮助的助手。"),
  new HumanMessage("你好！"),
]);

// 正确示范：使用 OpenAI 格式对象
await model.invoke([
  { role: "system", content: "你是一个有帮助的助手。" },
  { role: "user", content: "你好！" },
]);

// 正确示范：链式构建对话
const chatHistory = [
  new HumanMessage("你好，请问你能做什么？"),
  new AIMessage("你好！我是一个 AI 助手，可以帮助你解答问题。"),
  new HumanMessage("那太好了，请给我介绍一下 LangChain。"),
];
const response = await model.invoke(chatHistory);

❌ 上下文溢出 — Context Window Full

错误信息：BadRequestError: 400 This model's maximum context length is 128000 tokens 或 Agent 在多轮对话后输出质量严重下降

原因分析：在多轮对话或 Agent 交互中，历史消息不断累积，最终超过了模型的上下文窗口。即使没有触发硬限制，过长的上下文也会导致模型注意力分散，输出质量下降。这在使用 Agent + 工具时尤为明显，因为工具的返回结果也会占用上下文。

解决方案：

  1. 实现消息摘要机制，将旧消息压缩为摘要
  2. 使用 trimMessages 限制历史长度
  3. 在 Agent 的系统提示中明确要求简洁回答
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import {
  trimMessages,
  HumanMessage,
  AIMessage,
  SystemMessage,
  type BaseMessage,
} from "@langchain/core/messages";

const model = new ChatOpenAI({ model: "gpt-4o" });

// 策略一：保留系统消息 + 最近 N 轮对话
const trimmer = trimMessages({
  maxTokens: 8000,
  strategy: "last",
  tokenCounter: (msgs) => msgs.length * 4,
  includeSystem: true,
  startOn: "human",   // 确保裁剪后第一条是用户消息
});

// 策略二：消息摘要压缩
async function summarizeHistory(
  messages: BaseMessage[]
): Promise {
  if (messages.length <= 6) return messages;

  const oldMessages = messages.slice(0, -4);  // 取较早的消息
  const recentMessages = messages.slice(-4);  // 保留最近 4 条

  const summaryResult = await model.invoke([
    new SystemMessage(
      "请将以下对话历史总结为简洁的摘要，保留关键信息。"
    ),
    new HumanMessage(
      oldMessages.map((m) => `${m.constructor.name}: ${m.content}`).join("\n")
    ),
  ]);

  return [
    new SystemMessage(`对话摘要：${summaryResult.content}`),
    ...recentMessages,
  ];
}

❌ 流式传输中断或丢数据

错误信息：Error: stream was destroyed 或流式输出中途截断，只有部分内容

原因分析：流式传输过程中连接被意外中断。常见原因包括：请求超时、客户端在流完成前关闭了连接、在 Express/Fastify 等框架中忘记正确处理流式响应、或在异步迭代中未正确使用 for await...of。

解决方案：

  1. 使用 model.stream() 获取 AsyncIterator，并用 for await...of 正确迭代
  2. 在 Web 框架中正确设置响应头和流式写入
  3. 添加错误处理和流完成检测

import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

const model = new ChatOpenAI({ model: "gpt-4o", streaming: true });

// 基础用法：正确迭代流
async function streamChat(userInput: string) {
  const stream = await model.stream([
    new HumanMessage(userInput),
  ]);

  try {
    for await (const chunk of stream) {
      const content = chunk.content;
      if (typeof content === "string") {
        process.stdout.write(content);
      }
    }
    console.log(); // 换行
  } catch (error) {
    console.error("流式传输中断:", error);
  }
}

📋 结构化输出

❌ 结构化输出 Schema 验证失败

错误信息：LLM output does not match the expected schema 或 ZodError: Expected string, received undefined at "name"

原因分析：使用 withStructuredOutput() 方法时，模型返回的 JSON 不符合定义的 Zod schema。常见原因有：schema 过于复杂导致模型生成困难、缺少 .describe() 说明、或使用了模型不支持的 Zod 类型（如 z.record()）。

解决方案：

  1. 简化 schema 结构，优先使用基础类型（string、number、enum）
  2. 为每个字段添加 .describe() 描述，帮助模型理解预期
  3. 使用 z.enum() 替代自由文本字段来减少错误
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import { z } from "zod";

// 错误示范：schema 太复杂且缺少描述
// const badSchema = z.object({
//   meta: z.record(z.unknown()),
//   nested: z.object({ deep: z.object({ value: z.any() }) }),
// });

// 正确示范：清晰、简洁的 schema
const movieSchema = z.object({
  title: z.string().describe("电影名称"),
  year: z.number().describe("上映年份，如 2024"),
  rating: z
    .number()
    .min(0)
    .max(10)
    .describe("评分，0-10 之间"),
  genre: z
    .enum(["动作", "喜剧", "剧情", "科幻", "恐怖", "动画", "其他"])
    .describe("电影类型"),
  summary: z.string().describe("一句话简介，不超过 50 字"),
});

const model = new ChatOpenAI({
  model: "gpt-4o",
  temperature: 0,  // 结构化输出时建议降低温度
});

// withStructuredOutput 会自动处理 JSON 解析和验证
const structuredModel = model.withStructuredOutput(movieSchema);

const result = await structuredModel.invoke(
  "请分析电影《盗梦空间》的基本信息"
);

console.log(result.title);   // "盗梦空间"
console.log(result.year);    // 2010
console.log(result.rating);  // 9.3
console.log(result.genre);   // "科幻"

🧠 记忆系统

❌ Checkpointer 未配置导致 Agent 无状态

错误信息：没有明确报错，但每次对话 Agent 都忘记了之前的上下文，表现得像"失忆"

原因分析：LangGraph 的 Agent 默认是无状态的。每次调用 agent.invoke() 都是一次独立的执行，不会自动保留对话历史。如果你没有配置 checkpointer（检查点存储），Agent 就无法在多次调用之间保持状态。

解决方案：

  1. 配置 MemorySaver（内存存储，适合开发测试）
  2. 在每次调用时传入相同的 thread_id 来关联同一对话
  3. 生产环境建议使用 @langchain/langgraph-checkpoint-postgres 等持久化方案
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { MemorySaver } from "@langchain/langgraph";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

const model = new ChatOpenAI({ model: "gpt-4o" });

const greetTool = tool(
  async ({ name }) => `你好，${name}！很高兴认识你。`,
  {
    name: "greet",
    description: "向指定的人打招呼",
    schema: z.object({ name: z.string() }),
  }
);

// 关键：创建 checkpointer
const checkpointer = new MemorySaver();

const agent = createReactAgent({
  llm: model,
  tools: [greetTool],
  checkpointer,  // 注入 checkpointer
});

// 关键：每次调用传入相同的 thread_id
const config = { configurable: { thread_id: "user-123" } };

// 第一轮对话
await agent.invoke(
  { messages: [{ role: "user", content: "我叫小明" }] },
  config
);

// 第二轮对话 — Agent 能记住你的名字
const result = await agent.invoke(
  { messages: [{ role: "user", content: "我叫什么名字？" }] },
  config  // 使用同一个 config
);

// Agent 会回答"你叫小明"（如果没有 checkpointer 则不知道）

❌ thread_id 冲突导致状态混乱

错误信息：没有明确报错，但对话中出现了不属于当前用户的上下文信息，或者 Agent 行为不一致

原因分析：多个用户或多个对话会话复用了同一个 thread_id，导致不同会话的状态发生了交叉。例如所有用户都使用硬编码的 thread_id: "default"，或者在多线程环境下 thread_id 生成逻辑有竞态条件。

解决方案：

  1. 为每个用户/会话生成唯一的 thread_id
  2. 使用 UUID 或用户 ID + 时间戳的组合
  3. 在 Web 应用中，将 thread_id 与用户会话绑定
TypeScript
import { ChatOpenAI } from "@langchain/openai";
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { MemorySaver } from "@langchain/langgraph";
import { randomUUID } from "node:crypto";

const model = new ChatOpenAI({ model: "gpt-4o" });
const checkpointer = new MemorySaver();

const agent = createReactAgent({
  llm: model,
  tools: [],
  checkpointer,
});

// 错误示范：所有会话使用同一个 thread_id
// const badConfig = { configurable: { thread_id: "default" } };

// 正确示范：为每个会话生成唯一 ID
function createSession(userId: string) {
  const sessionId = randomUUID();
  return {
    threadId: `${userId}-${sessionId}`,
    config: {
      configurable: {
        thread_id: `${userId}-${sessionId}`,
      },
    },
  };
}

// 使用示例
const user1Session = createSession("user-001");
const user2Session = createSession("user-002");

// 用户1的对话
await agent.invoke(
  { messages: [{ role: "user", content: "我的爱好是编程" }] },
  user1Session.config
);

// 用户2的对话（完全独立的状态）
await agent.invoke(
  { messages: [{ role: "user", content: "我的爱好是绘画" }] },
  user2Session.config
);

// 用户1询问时，只会知道"编程"，不会知道"绘画"
const result = await agent.invoke(
  { messages: [{ role: "user", content: "我的爱好是什么？" }] },
  user1Session.config
);

🔍 快速诊断清单

遇到问题时，按以下顺序逐步排查：

排查步骤 检查内容 常见命令
1. 环境检查 Node.js 版本、TypeScript 配置 node -v、npx tsc --showConfig
2. 依赖检查 包版本是否对齐、是否有重复安装 npm ls @langchain/core
3. 密钥检查 环境变量是否正确加载 console.log(process.env.OPENAI_API_KEY?.slice(0,7))
4. 网络检查 是否能访问 API 端点 curl https://api.openai.com/v1/models
5. 消息格式 消息对象是否使用正确的类构造 console.log(JSON.stringify(messages, null, 2))
6. 模型调用 模型名称、参数是否正确 在 OpenAI Playground 中先测试
💡 通用排错技巧
  • 始终在入口文件顶部添加 import "dotenv/config" 确保环境变量最先加载
  • 使用 console.dir(result, { depth: null }) 查看完整的对象结构
  • 开启 LangChain 的调试日志：process.env.LANGCHAIN_VERBOSE = "true"
  • 使用 LangSmith 进行请求追踪，可以直观看到每一步的输入输出
  • 缩小问题范围：先单独测试模型调用，再加入工具和 Agent
