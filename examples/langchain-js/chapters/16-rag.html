🔍 RAG 检索增强生成 — 让 AI 拥有专业知识库

将外部知识注入 LLM，打造能回答私有领域问题的智能系统

💡 核心概念

概念一：RAG 架构全景 — Indexing → Retrieval → Generation

RAG（Retrieval-Augmented Generation） 是当前最流行的 LLM 应用范式之一。 它的核心思想是：先从一个知识库中检索出与用户问题相关的文档片段，再将它们作为上下文 注入到 LLM 的提示中，让模型基于真实知识生成回答。

为什么要这样做？因为 LLM 的知识截止于训练数据的时间点，而且大模型不具备私有知识。 如果直接问"我们公司的退款政策是什么？"，模型要么胡编乱造，要么说"我不知道"。 RAG 解决了这个问题——它把公司的退款政策文档导入向量库，用户提问时先检索相关条款， 再让模型基于检索到的内容回答。

🏗️ RAG 三阶段架构

第一阶段 — 索引（Indexing）：
加载文档 → 文本分割 → 向量化 → 存储到向量数据库。
这是离线预处理阶段，文档准备好后只需做一次。

第二阶段 — 检索（Retrieval）：
用户提问 → 将问题向量化 → 在向量库中做相似性搜索 → 返回 Top-K 相关文档片段。
这是在线推理的第一步，速度必须快。

第三阶段 — 生成（Generation）：
将检索到的文档片段 + 用户问题 + 系统提示 拼接到一起 → 传给 LLM → 生成最终回答。
模型的任务是"用提供的资料来回答问题"。

概念二：LangChain v1.x 中的 RAG 实现模式

在 LangChain v1.x 中，最典型的 RAG 实现方式有两种：

模式 A：Agent + Retriever Tool
将向量存储的检索接口封装成一个 tool，然后传给 createAgent。 Agent 在思考过程中自主决定何时调用检索工具。这种方式适合需要复杂决策的场景—— 比如 Agent 可以先用检索工具获取知识，再结合记忆和推理给出最终答案。

模式 B：LCEL 管道
使用 RunnablePassthrough.assign() 将检索结果注入到提示模板中， 形成 context_retriever | prompt | model | parser 的处理管线。 这种方式简单直接，数据流清晰，适合确定性较强的 QA 场景。

TypeScript 
// ==========================================
// 模式 A：使用 Agent + Retriever Tool
// ==========================================
import { createAgent, tool } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { OpenAIEmbeddings } from "@langchain/openai";
import { z } from "zod";

// 1. 准备向量存储
const vectorStore = new MemoryVectorStore(new OpenAIEmbeddings());
// ... 先存入文档 ...

// 2. 将检索封装为工具
const searchKnowledgeBase = tool(
  async ({ query }) => {
    const results = await vectorStore.similaritySearch(query, 3);
    return results.map((d) => d.pageContent).join("\n\n---\n\n");
  },
  {
    name: "search_knowledge_base",
    description: "搜索知识库，获取与问题相关的文档内容",
    schema: z.object({ query: z.string() }),
  }
);

// 3. 创建 Agent
const agent = await createAgent({
  model: new ChatOpenAI({ model: "gpt-4o-mini" }),
  tools: [searchKnowledgeBase],
  systemPrompt: `你是一个知识助手。当用户提问时，使用 search_knowledge_base 工具
获取相关信息，然后基于检索到的内容回答问题。如果你不确定答案，请诚实地说不知道。`,
});

// ==========================================
// 模式 B：使用 LCEL 管道
// ==========================================
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { RunnablePassthrough } from "@langchain/core/runnables";
import { StringOutputParser } from "@langchain/core/output_parsers";

const prompt = ChatPromptTemplate.fromMessages([
  ["system", `你是一个知识问答助手。请基于以下资料回答问题。
如果资料中找不到相关信息，请说"资料中没有提到这个问题的答案"。

相关资料：
{context}`],
  ["human", "{question}"],
]);

// 将检索包装成可运行函数
const retrieveDocs = async (input: { question: string }) => {
  const docs = await vectorStore.similaritySearch(input.question, 3);
  return {
    ...input,
    context: docs.map((d) => d.pageContent).join("\n\n---\n\n"),
  };
};

// LCEL 管道
const ragChain =
  RunnablePassthrough.assign({ context: retrieveDocs })
  | prompt
  | new ChatOpenAI({ model: "gpt-4o-mini" })
  | new StringOutputParser();

const answer = await ragChain.invoke({
  question: "公司的退款政策是什么？",
});
⚠️ LangChain v1.x 注意事项

在 LangChain v1.x 中，createRetrievalChain() 是 v0.3+ 中的概念（对应 @langchain/langgraph 或 langchain/chains 中的实验性 API）。在 v1.x 稳定的 Agent API 中，推荐使用上述两种模式之一：将向量检索封装为 Agent 工具，或使用 LCEL 管道手动构建 RAG 管线。

概念三：RAG 提示工程 — 让模型"学会引用"

RAG 的成功不仅取决于检索质量，也取决于提示设计。一个好的 RAG 提示需要告诉模型：

  • 上下文的来源："基于以下资料回答问题"
  • 引用要求："请引用相关原文支持你的回答"
  • 不知道的处理："如果资料中没有相关信息，请诚实地回答不知道"
  • 格式约束："用中文回答，使用 Markdown 格式"
TypeScript 
// ==========================================
// 一个生产级的 RAG 系统提示模板
// ==========================================
const ragSystemPrompt = `你是一个基于知识库的问答助手。你的回答必须遵循以下规则：

## 回答规则

1. **基于检索内容**：你的回答必须基于以下"相关资料"中的内容，不要使用你自己的知识。
2. **引用来源**：在回答中引用相关原文时，用「引号」标注。
3. **不知道就说不知道**：如果检索到的资料不足以回答用户问题，
   请直接说"根据现有资料，我无法回答这个问题"。不要编造答案。
4. **保持简洁**：回答要直接、准确，不要添加不必要的背景信息。
5. **中文回答**：请使用中文回答用户问题。

## 相关资料

{context}

## 用户问题

{question}`;

// 当模型看到这个提示时，它会：
// 1. 优先使用 {context} 中的信息
// 2. 不会胡编乱造
// 3. 在必要时引用原文

概念四：查询变换 — 让检索更精准

在实际应用中，用户的原始提问往往不适合直接用于检索。比如用户问"它贵吗？" ——如果没有上下文，"它"指代不明，直接向量搜索效果会很差。 查询变换（Query Transformation） 就是在检索之前， 先用 LLM 对用户问题做一次"预处理"。

常见的查询变换策略包括：

策略描述示例
重写（Rewrite）将模糊问题改写为清晰、自包含的查询"它安全吗？" → "TypeScript 的类型系统安全吗？"
分解（Decomposition）将复杂问题拆解为多个子查询"请比较 React 和 Vue 的优缺点" → ["React 的优点是什么？", "React 的缺点是什么？", "Vue 的优点是什么？", "Vue 的缺点是什么？"]
假设回答（HyDE）先生成一个假设性回答，再用这个回答去检索问题 → LLM 生成伪答案 → 用伪答案检索 → 找到更相关的文档
扩展（Expansion）用同义词或相关概念扩展查询词"退款" → "退款 退货 取消订单 退款政策"

概念五：Self-RAG 与 HyDE — 进阶 RAG 范式

Self-RAG 是一种让 LLM 自我反思的 RAG 范式。它的核心流程是：

  1. 检索相关文档片段
  2. LLM 检查每个片段是否与问题相关（相关性判断）
  3. 只保留相关的片段
  4. 基于筛选后的片段生成回答
  5. LLM 再检查回答是否被检索内容充分支持（事实性检查）

HyDE（Hypothetical Document Embeddings） 是一种有趣的策略。 它的思路是：与其直接用问题去向量库搜索，不如先让 LLM 根据问题生成一个"假设性的理想文档"， 然后用这个假设文档的向量去搜索。因为假设文档和真实文档在语义空间上更接近， 往往能检索到更相关的内容。

TypeScript 
// ==========================================
// Self-RAG：带自我反思的检索增强生成
// ==========================================

// 步骤 1：检索
const rawDocs = await vectorStore.similaritySearch(question, 5);

// 步骤 2：用 LLM 判断相关性（相关性过滤）
const relevanceCheckPrompt = `判断以下文档是否与用户问题相关。
只回答 "相关" 或 "不相关"。

用户问题：{question}

文档内容：{document}`;

const relevanceChain = ChatPromptTemplate.fromTemplate(relevanceCheckPrompt)
  | model
  | new StringOutputParser();

const relevantDocs = [];
for (const doc of rawDocs) {
  const judgment = await relevanceChain.invoke({
    question,
    document: doc.pageContent,
  });
  if (judgment.includes("相关")) {
    relevantDocs.push(doc);
  }
}

// 步骤 3：基于筛选后的文档生成回答
// ... 正常 RAG 生成 ...

// ==========================================
// HyDE：假设文档嵌入检索
// ==========================================

// 步骤 1：根据问题生成假设性文档
const hydePrompt = `请写一段文本，内容是对以下问题的理想解答。
这段文本将用于相似度搜索，所以请写得像一篇百科知识文章，
包含具体的细节和解释。

问题：{question}`;

const hydeChain = ChatPromptTemplate.fromTemplate(hydePrompt)
  | model
  | new StringOutputParser();

const hypotheticalDoc = await hydeChain.invoke({ question });

// 步骤 2：用生成的假设文档去检索（而非用原问题）
const searchResults = await vectorStore.similaritySearch(hypotheticalDoc, 3);

// 步骤 3：基于检索到的真实文档生成最终答案
// 注意：这里使用的是真实文档，不是假设文档
const finalAnswer = await generateAnswer(question, searchResults);

🔨 实战演练

实战一：完整文档问答系统（LCEL 管道版）

场景描述：你有一个存放公司产品说明书的目录（Markdown 文件），需要构建一个文档 QA 系统， 让员工可以自然语言提问，系统自动从文档中找到答案。

你的任务：实现完整的 RAG 流水线：加载 → 分割 → 嵌入 → 存储 → 检索 → 生成。

🧑‍💻 先自己写，写完再展开看参考答案
TypeScript 
import { ChatOpenAI, OpenAIEmbeddings } from "@langchain/openai";
import { TextLoader } from "langchain/document_loaders/fs/text";
import { RecursiveCharacterTextSplitter } from "langchain/text_splitter";
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { RunnablePassthrough } from "@langchain/core/runnables";
import { StringOutputParser } from "@langchain/core/output_parsers";
import * as fs from "fs/promises";
import * as path from "path";

// ==========================================
// 第一阶段：索引（Indexing）
// ==========================================

async function buildKnowledgeBase(docsDir: string) {
  console.log("📚 正在构建知识库...");

  // 1. 加载文档
  const files = await fs.readdir(docsDir);
  const mdFiles = files.filter((f) => f.endsWith(".md"));

  const allDocuments = [];
  for (const file of mdFiles) {
    const loader = new TextLoader(path.join(docsDir, file));
    const docs = await loader.load();
    // 给每个文档添加文件名作为元数据
    docs.forEach((doc) => {
      doc.metadata.source = file;
    });
    allDocuments.push(...docs);
  }
  console.log(`   ✅ 加载了 ${allDocuments.length} 个文档`);

  // 2. 文本分割
  const splitter = new RecursiveCharacterTextSplitter({
    chunkSize: 500,
    chunkOverlap: 50,
    separators: ["\n## ", "\n### ", "\n\n", "\n", " ", ""],
  });

  const splitDocs = await splitter.splitDocuments(allDocuments);
  console.log(`   ✅ 分割为 ${splitDocs.length} 个文本块`);

  // 3. 嵌入并存储
  const embeddings = new OpenAIEmbeddings();
  const vectorStore = await MemoryVectorStore.fromDocuments(
    splitDocs,
    embeddings
  );
  console.log(`   ✅ 知识库构建完成！`);

  return vectorStore;
}

// ==========================================
// 第二阶段 + 第三阶段：检索 + 生成
// ==========================================

async function createRagQA(vectorStore: MemoryVectorStore) {
  // 创建检索器
  const retriever = vectorStore.asRetriever(3); // 返回 Top-3

  // RAG 提示模板
  const ragPrompt = ChatPromptTemplate.fromMessages([
    ["system", `你是一个专业的文档问答助手。请基于以下资料回答用户问题。

## 回答要求
- 优先使用资料中的信息回答问题
- 如果资料不足以回答，请说"资料中没有相关说明"
- 引用资料原文时，用「引号」括起来
- 回答要简洁、准确

## 相关资料
{context}`],
    ["human", "{question}"],
  ]);

  // 模型
  const model = new ChatOpenAI({
    model: "gpt-4o-mini",
    temperature: 0.2,
  });

  // 检索函数（用于 LCEL 管道）
  const formatRetrievedDocs = async (input: { question: string }) => {
    const docs = await retriever.invoke(input.question);
    const formatted = docs
      .map((d, i) => `[片段 ${i + 1}]（来自：${d.metadata.source || "未知"})\n${d.pageContent}`)
      .join("\n\n");
    return {
      ...input,
      context: formatted,
    };
  };

  // 构建 LCEL 管道
  const ragChain =
    RunnablePassthrough.assign({ context: formatRetrievedDocs })
    | ragPrompt
    | model
    | new StringOutputParser();

  return ragChain;
}

// ==========================================
// 主程序
// ==========================================

async function main() {
  // 构建知识库
  const vectorStore = await buildKnowledgeBase("./docs");

  // 创建 QA 链
  const qaChain = await createRagQA(vectorStore);

  // 提问
  const questions = [
    "产品的保修期是多久？",
    "如何重置管理员密码？",
    "支持哪些第三方集成？",
  ];

  for (const q of questions) {
    console.log(`\n❓ ${q}`);
    const answer = await qaChain.invoke({ question: q });
    console.log(`💡 ${answer}`);
    console.log("─".repeat(50));
  }
}

main().catch(console.error);

实战二：Agent + Retriever Tool 版文档 QA

场景描述：与实战一相同，但这次使用 Agent 模式——Agent 可以自主决定是否需要检索知识库， 并且可以结合多步推理和工具调用来回答更复杂的问题。

你的任务：使用 createAgent + retriever tool 实现 Agent 驱动的 RAG。

🧑‍💻 先自己写，写完再展开看参考答案
TypeScript 
import { createAgent, tool } from "langchain";
import { ChatOpenAI, OpenAIEmbeddings } from "@langchain/openai";
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { RecursiveCharacterTextSplitter } from "langchain/text_splitter";
import { TextLoader } from "langchain/document_loaders/fs/text";
import { z } from "zod";

// ==========================================
// 1. 构建知识库（与实战一相同）
// ==========================================

async function buildVectorStore() {
  const loader = new TextLoader("./docs/product-manual.md");
  const docs = await loader.load();

  const splitter = new RecursiveCharacterTextSplitter({
    chunkSize: 500,
    chunkOverlap: 50,
  });
  const splitDocs = await splitter.splitDocuments(docs);

  const embeddings = new OpenAIEmbeddings();
  const vectorStore = await MemoryVectorStore.fromDocuments(
    splitDocs,
    embeddings
  );

  return vectorStore;
}

// ==========================================
// 2. 创建检索工具
// ==========================================

async function createRetrieverTool(vectorStore: MemoryVectorStore) {
  return tool(
    async ({ query }) => {
      // 执行向量检索
      const results = await vectorStore.similaritySearch(query, 3);

      if (results.length === 0) {
        return "知识库中没有找到相关信息。";
      }

      // 格式化结果
      return results
        .map(
          (doc, i) =>
            `[结果 ${i + 1}]${doc.metadata.source ? ` (来自: ${doc.metadata.source})` : ""}\n${doc.pageContent}`
        )
        .join("\n\n");
    },
    {
      name: "search_documentation",
      description: `搜索产品文档知识库。当你需要查找产品功能、配置方法、
        故障排除、API 文档等信息时，使用此工具。输入应该是简洁的搜索查询。`,
      schema: z.object({
        query: z.string().describe("搜索关键词，如'如何重置密码'或'保修期'"),
      }),
    }
  );
}

// ==========================================
// 3. 创建 Agent
// ==========================================

async function createRagAgent(vectorStore: MemoryVectorStore) {
  const retrieverTool = await createRetrieverTool(vectorStore);

  const agent = await createAgent({
    model: new ChatOpenAI({
      model: "gpt-4o-mini",
      temperature: 0.3,
    }),
    tools: [retrieverTool],

async function buildMultiDocStore(docsDir: string) {
  // 使用 DirectoryLoader 加载目录下所有文档
  const loader = new DirectoryLoader(docsDir, {
    ".md": (filePath) => new TextLoader(filePath),
    ".txt": (filePath) => new TextLoader(filePath),
  });

  const splitter = new RecursiveCharacterTextSplitter({

  const embeddings = new OpenAIEmbeddings();
  const vectorStore = await MemoryVectorStore.fromDocuments(docs, embeddings);

  return vectorStore;
}

  // 检索工具

技巧一：查询重写（Query Rewriting）

在多轮对话中，用户的后续问题往往包含指代（"它"、"那个"、"第一个"）， 直接检索效果很差。查询重写在检索之前，先利用 LLM 和对话历史将模糊问题改写为自包含的查询。

TypeScript 
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { ChatOpenAI } from "@langchain/openai";
import { StringOutputParser } from "@langchain/core/output_parsers";

// ==========================================
// 查询重写组件
// ==========================================

const rewritePrompt = ChatPromptTemplate.fromMessages([
  ["system", `你是一个查询重写助手。你的任务是将用户的当前问题，
结合对话历史，改写成自包含的搜索查询。

规则：
1. 替换所有指代词（它、他、这个、那个、第一个等）为具体名词
2. 保持原问题的核心意图
3. 只输出改写后的查询，不要加任何解释
4. 如果问题已经自包含，保持原样`],
  ["human", `对话历史：
{chat_history}

当前问题：{question}

请输出改写后的搜索查询：`],
]);

const model = new ChatOpenAI({ model: "gpt-4o-mini", temperature: 0 });
const rewriteChain = rewritePrompt | model | new StringOutputParser();

// ==========================================
// 在 RAG 管线中使用查询重写
// ==========================================

async function rewriteQuery(
  question: string,
  chatHistory: string
): Promise {
  if (!chatHistory) return question; // 首轮无需重写

  const rewritten = await rewriteChain.invoke({
    chat_history: chatHistory,
    question,
  });

  console.log(`   🔄 查询重写: "${question}" → "${rewritten}"`);
  return rewritten.trim();
}

技巧二：查询分解（Query Decomposition）

当用户提一个复杂问题时，将其拆解为多个简单子查询，分别检索后再汇总结果。 这能显著提高复杂问题的回答质量。

TypeScript 
// ==========================================
// 查询分解组件
// ==========================================

const decomposePrompt = ChatPromptTemplate.fromMessages([
  ["system", `将以下复杂问题拆解为多个简单的子问题。
每个子问题应该独立可检索，覆盖原问题的不同方面。
用换行分隔，只输出子问题列表。`],
  ["human", "{question}"],
]);

const decomposeChain = decomposePrompt | model | new StringOutputParser();

// ==========================================
// 多查询检索 + 汇总
// ==========================================

async function multiQueryRetrieve(
  question: string,
  retriever: any
): Promise {
  const result = await decomposeChain.invoke({ question });
  const subQueries = result.split("\n").filter((q: string) => q.trim());

  console.log(`   🔗 分解为 ${subQueries.length} 个子查询：`);
  subQueries.forEach((q: string, i: number) => console.log(`      ${i + 1}. ${q}`));

  // 并行检索每个子查询
  const allResults = await Promise.all(
    subQueries.map((q: string) => retriever.invoke(q))
  );

  // 合并去重（基于文本内容）
  const seen = new Set();
  const uniqueDocs = allResults.flat().filter((doc: any) => {
    const key = doc.pageContent.substring(0, 50);
    if (seen.has(key)) return false;
    seen.add(key);
    return true;
  });

  return uniqueDocs;
}

// 2. "专业版有哪些功能？"
// 3. "免费版的价格是多少？"

技巧三：检索后重排（Re-ranking）

向量相似性搜索的 Top-K 结果不一定都是相关的。检索后重排是在生成最终答案之前， 用一个更精确的模型（如 Cross-encoder）或让 LLM 自己重新评估每个检索结果的相关性。

TypeScript 
// ==========================================
// LLM 重排器：让 LLM 评估每个文档的相关性
// ==========================================

const rerankPrompt = ChatPromptTemplate.fromMessages([
  ["system", `你是一个文档相关性评估器。对于每个文档，判断它与用户问题的相关性。
输出格式：每行一个分数（1-5），5 表示高度相关，1 表示完全不相关。

只输出数字列表，每行一个。`],
  ["human", `用户问题：{question}

文档列表：
{documents}

请为每个文档打分（1-5 分）：`],
]);

const rerankChain = rerankPrompt | model | new StringOutputParser();

async function retrieveAndRerank(
  question: string,
  retriever: any,
  topK: number = 5
) {
  // 1. 先检索出较多结果（扩大召回）
  const docs = await retriever.invoke(question, topK * 2);

  // 2. 让 LLM 评分
  const docTexts = docs.map((d: any, i: number) => `[${i + 1}] ${d.pageContent}`);
  const scores = await rerankChain.invoke({
    question,
    documents: docTexts.join("\n\n"),
  });

  // 3. 解析分数并排序
  const scoreList = scores
    .split("\n")
    .map((s: string) => parseInt(s.trim()))
    .filter((n: number) => !isNaN(n));

  const scored = docs.map((doc: any, i: number) => ({
    doc,
    score: scoreList[i] || 1,
  }));

  scored.sort((a: any, b: any) => b.score - a.score);

  // 4. 返回得分最高的 topK 个
  return scored.slice(0, topK).map((s: any) => s.doc);
}

// 使用方式：replace retriever.invoke() with retrieveAndRerank()

技巧四：多轮对话中的上下文压缩

在多轮 RAG 对话中，历史消息会不断累积，导致检索时上下文过长、噪声增多。 上下文压缩（Contextual Compression） 的思路是：在检索之后， 用 LLM 对检索结果进行"挤压"——只保留最相关的部分，去除与当前问题无关的信息。

TypeScript 
// ==========================================
// 上下文压缩
// ==========================================

const compressPrompt = ChatPromptTemplate.fromMessages([
  ["system", `你是一个文档压缩助手。你的任务是从以下文档中提取出与用户问题
最相关的部分。如果文档中有多个段落，只保留最能回答用户问题的那些。

要求：
1. 保留原文中的关键数据、数字、事实
2. 去除无关的示例、铺垫语言
3. 如果整个文档都不相关，输出"无相关信息"
4. 输出要简洁，直接给出压缩后的内容`],
  ["human", `用户问题：{question}

原始文档：
{document}

请输出压缩后的相关内容：`],
]);

const compressChain = compressPrompt | model | new StringOutputParser();

async function compressRetrievedDocs(
  question: string,
  docs: any[]
): Promise {
  const combined = docs.map((d) => d.pageContent).join("\n\n");

  const compressed = await compressChain.invoke({
    question,
    document: combined,
  });

  return compressed;
}

// 完整的多轮 RAG 管线（含查询重写 + 上下文压缩）
// ==========================================

async function advancedRagPipeline(
  question: string,
  chatHistory: string,
  vectorStore: MemoryVectorStore

  // Step 1: 查询重写
  const rewrittenQuery = await rewriteQuery(question, chatHistory);

  // Step 2: 检索
  const rawDocs = await vectorStore.similaritySearch(rewrittenQuery, 5);

  // Step 3: 上下文压缩
  const compressedContext = await compressRetrievedDocs(rewrittenQuery, rawDocs);

  // Step 4: 生成回答
  const answer = await ragChain.invoke({
    question: question, // 用原问题（保留用户原始意图）
    context: compressedContext,
  });

🧠 知识检查点

📝 自测题：检查你对本章内容的理解
1. RAG 的三阶段架构是什么？每个阶段的核心任务是什么？

Indexing（索引）：加载文档 → 文本分割 → 嵌入向量化 → 存储到向量数据库。这是离线预处理阶段。

Retrieval（检索）：用户提问 → 问题向量化 → 在向量库中做相似性搜索 → 返回 Top-K 相关文档片段。这是在线推理的第一步。

Generation（生成）：检索到的文档片段 + 用户问题 + 系统提示 → 传给 LLM → 生成最终回答。模型的任务是"用提供的资料来回答问题"。

2. 在 LangChain v1.x 中，实现 RAG 有哪两种主要模式？各有什么优缺点？

模式 A：Agent + Retriever Tool — 将检索封装为工具传给 createAgent。优点：Agent 可以自主决策何时检索、结合记忆和多步推理；缺点：Agent 调用有开销，适用于需要复杂决策的场景。

模式 B：LCEL 管道 — 使用 RunnablePassthrough.assign() 注入检索结果。优点：轻量、直观、数据流清晰、流式支持好；缺点：确定性管线，缺乏 Agent 的自主决策能力。适用于简单的 QA 场景。

3. 什么是查询变换？常见的查询变换策略有哪些？

查询变换是在检索之前，用 LLM 对用户问题做预处理，使检索更精准。常见策略包括：

  • 重写（Rewrite） — 将模糊问题改写为自包含查询，解决指代问题
  • 分解（Decomposition） — 将复杂问题拆解为多个子查询，分别检索后汇总
  • HyDE — 先生成一个假设性回答，用该回答去检索（假设文档与真实文档语义更接近）
  • 扩展（Expansion） — 用同义词和相关概念扩展查询词
4. RAG 的提示设计中，"不知道"的处理为什么重要？

因为 LLM 天生有"编造倾向"（幻觉/Hallucination）。如果检索到的资料不包含答案，模型可能会强行用其训练数据中的知识来回答，导致信息不准确。在提示中明确要求"如果资料中找不到相关信息，请说不知道"，能显著降低幻觉率。这是 RAG 系统从"可用"到"可靠"的关键之一。

5. Self-RAG 和 HyDE 有什么区别？

Self-RAG 是在检索后让模型自我反思：判断每个检索结果是否相关（相关性过滤），生成答案后再检查是否有充分的事实支持（事实性验证）。它是一个"检索→判断→生成→验证"的循环。

HyDE 是在检索前先让模型生成一个假设性的"理想文档"，然后用这个假设文档的向量去检索真实文档。它的出发点是：假设文档在语义空间上比原始问题更接近真实文档，因此能检索到更相关的内容。

简单说：Self-RAG 是"检后反思"，HyDE 是"检前假设"。

🐛 常见错误

❌ 错误 1：chunkSize 和 chunkOverlap 设置不合理

症状：检索到的文档片段要么太短缺失关键信息，要么太长包含大量噪声。

原因：chunkSize 太小（<200）导致每个片段信息量不足；太大（>2000）导致片段噪声过多。chunkOverlap 太小（<10）导致上下文断裂。

解决：一般推荐 chunkSize=500-1000，chunkOverlap=50-200。同时根据文档结构选择合适的分隔符（如 Markdown 标题）。

❌ 错误 2：检索结果未正确注入提示，模型仍依赖自有知识

症状：模型回答的内容在检索结果中并不存在，模型仍然使用了训练数据中的知识。

原因：提示模板中的 context 变量没有被正确注入，或者注入后模型被"允许"忽略它。常见于提示中缺少"请基于以上资料回答"的明确指令。

解决：在 system prompt 中明确写死规则："你的回答必须基于以下资料。不要使用你自己的知识。"同时检查 LCEL 管道中的数据流是否正确传递了 context 字段。

❌ 错误 3：多轮对话中指代未解析导致检索失效

症状：第二轮提问"它的价格是多少？"时，检索返回的内容与第一轮提到的产品无关。

原因：直接用原始问题检索，"它的价格"没有包含产品名，向量搜索无法找到相关文档。

解决：在检索之前先做查询重写（Query Rewriting），利用 LLM 和对话历史将"它"替换为具体名词，得到一个自包含的搜索查询。

❌ 错误 4：Agent Tool 的 description 写得太模糊

症状：Agent 在需要检索时没有调用检索工具，或者在不该调用时调用了。

原因：在 createAgent 模式中，Agent 根据工具的描述来决定是否调用。如果 description 写得太笼统（如"搜索知识库"），Agent 无法判断何时应该使用。

解决：写具体、可操作的 description。例如："当你需要查找产品文档中的功能说明、配置方法、故障排除步骤时使用此工具。不要用它来回答常识性问题。"

❌ 错误 5：未处理检索结果为空的边界情况

症状：当知识库中没有相关内容时，模型仍然给出了答案（可能编造）。

原因：提示中没有处理"资料不足"的情况，模型倾向于"尽力回答"而非"诚实说不知道"。

解决：在提示中明确加入："如果提供的资料不足以回答问题，请直接说'资料中没有相关信息'。"并在代码层面检查检索结果是否为空。
