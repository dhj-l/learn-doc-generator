🔀 LangGraph — 状态图编排与复杂工作流

用图结构编排多步骤 AI 工作流，实现条件分支、循环、人机协同等高级编排模式

🎯 本章目标

学完本章，你将能够：
  • 理解 LangGraph 的核心概念：StateGraph、Node、Edge 以及状态驱动的工作流模型
  • 掌握 addNode、addEdge、addConditionalEdges 的用法，构建有向无环图（DAG）工作流
  • 使用 createReactAgent 预置 Agent 快速搭建具备工具调用能力的 LangGraph Agent
  • 实现条件分支逻辑，让工作流根据中间结果动态选择执行路径
  • 理解 Human-in-the-loop（人机协同）模式，使用 interrupt 实现人工审批流程
  • 掌握 Checkpoint 持久化机制，让工作流具备可恢复能力
  • 学会子图合成（Subgraph Composition），将复杂流程模块化

📋 前置知识

⚠️ 前置要求：请确保你已完成 第 2 章：Agent 基础 的学习， 理解 Agent 的基本执行流程和 createAgent 的用法。 同时建议先了解 第 5 章：工具 Tools， 掌握 tool() 函数的定义和工具调用的基本模式。

💡 为什么需要 LangGraph？

在前面的章节中，我们使用的 createAgent 是一个"黑盒"——它内部有一个 Agent 执行循环 （思考→调用工具→获取结果→再思考→最终回答），但我们无法干预或定制这个循环的结构。

想象一下你正在构建一个智能文档处理系统，它需要：

  • 步骤 1：读取并解析文档内容
  • 步骤 2：根据文档类型选择不同的处理策略（合同、报告、邮件……）
  • 步骤 3：如果文档包含敏感信息，需要人工审核才可继续
  • 步骤 4：生成摘要并将结果存储到数据库

使用普通的 createAgent 很难实现这样的多步骤、条件分支和人工介入流程。 这就是 LangGraph 登场的地方。

💡 LangGraph 的核心思想

LangGraph 将 AI 工作流建模为有向图：图的节点（Node）代表 一个计算步骤或 Agent 操作，边（Edge）代表数据流动和执行顺序， 状态（State）在整个图中传递，记录每一步的输入、输出和中间结果。 你可以添加条件边（Conditional Edge）让执行流程根据当前状态动态决策。

🧱 核心概念

概念一：StateGraph 与状态定义

StateGraph 是 LangGraph 的核心类。你需要先定义一个状态类型（State）， 然后创建一个 StateGraph 实例来编排工作流。状态是一个数据结构， 它贯穿整个图的执行过程，每个节点都可以读取和写入状态。

使用 Annotation 来定义状态的结构。每个状态字段可以指定一个归约器（Reducer）， 决定当多个节点向同一字段写入时如何合并。

TypeScript 
import { StateGraph, Annotation } from "@langchain/langgraph";

// ========== 定义状态结构 ==========
// 使用 Annotation.Root 创建状态类型
const GraphState = Annotation.Root({
  // 简单字段：直接赋值
  input: Annotation,

  // 使用归约器：当多个节点写入 messages 时，使用 concat 合并
  messages: Annotation({
    reducer: (left, right) => left.concat(right),
    default: () => [],
  }),

  // 带默认值的字段
  stepCount: Annotation({
    reducer: (left, right) => left + right,
    default: () => 0,
  }),

  // 可选字段
  result: Annotation({
    default: () => null,
  }),
});

// GraphState 的类型可以推导为：
// {
//   input: string;
//   messages: string[];
//   stepCount: number;
//   result: string | null;
// }
💡 Reducer 的作用

归约器（Reducer）定义了"当图中有多个并行节点或循环节点都向同一个状态字段写入时，如何合并值"。 默认行为是覆盖（新值替换旧值）。如果你想累积（如追加消息列表）， 就需要提供自定义 reducer，如上面的 (left, right) => left.concat(right)。

概念二：Node（节点）

节点是图中的一个计算单元。每个节点是一个函数，它接收当前状态作为输入， 返回状态的部分更新。使用 addNode() 方法注册节点。

TypeScript 
// 节点是一个函数：接收 State，返回 Partial<State>
async function processInput(state: typeof GraphState.State) {
  console.log(`[processInput] 输入: ${state.input}`);
  return {
    messages: [`收到输入: ${state.input}`],
    stepCount: 1,
  };
}

async function analyzeContent(state: typeof GraphState.State) {
  console.log(`[analyzeContent] 分析中...`);
  const analysis = `已分析内容 (共 ${state.messages.length} 条消息)`;
  return {
    messages: [analysis],
    result: analysis,
    stepCount: 1,  // 由于 stepCount 的 reducer 是累加，这个 1 会加给 stepCount
  };
}

// 创建 StateGraph 并注册节点
const workflow = new StateGraph(GraphState)
  .addNode("process_input", processInput)
  .addNode("analyze", analyzeContent);

概念三：Edge（边）与 START / END

边定义了节点之间的连接关系和执行顺序。LangGraph 提供了几种边：

  • 普通边（addEdge） — 从一个节点到另一个节点的固定连接
  • 条件边（addConditionalEdges） — 根据状态动态选择下一个节点
  • 入口边（START） — 图的起点，用 START 常量表示
  • 出口边（END） — 图的终点，用 END 常量表示
TypeScript 
import { StateGraph, START, END } from "@langchain/langgraph";

// 接上面的 workflow
workflow
  // 从 START 到第一个节点
  .addEdge(START, "process_input")
  // process_input 执行完后自动到 analyze
  .addEdge("process_input", "analyze")
  // analyze 执行完后到达终点
  .addEdge("analyze", END);

// 编译图，生成可调用的应用
const app = workflow.compile();

// 执行
const result = await app.invoke({
  input: "这是一份需要处理的文档内容",
});

console.log(result);
// {
//   input: "这是一份需要处理的文档内容",
//   messages: ["收到输入: 这是一份需要处理的文档内容", "已分析内容 (共 1 条消息)"],
//   stepCount: 2,
//   result: "已分析内容 (共 1 条消息)",
// }

概念四：addConditionalEdges（条件边）

条件边是 LangGraph 最强大的特性之一。它允许你根据当前状态的值，动态决定下一步执行哪个节点。 你需要提供一个路由函数，该函数接收状态并返回一个节点名称字符串（或 END）。

TypeScript 
// 定义三个处理节点
async function classifyDocument(state: typeof GraphState.State) {
  const doc = state.input.toLowerCase();
  let docType = "unknown";
  if (doc.includes("合同") || doc.includes("协议")) docType = "contract";
  else if (doc.includes("报告") || doc.includes("总结")) docType = "report";
  else if (doc.includes("邮件") || doc.includes("email")) docType = "email";
  return { docType, messages: [`文档分类: ${docType}`] };
}

async function processContract(state: typeof GraphState.State) {
  return { messages: ["处理合同: 提取条款、签署方、金额..."] };
}

async function processReport(state: typeof GraphState.State) {
  return { messages: ["处理报告: 提取关键发现、数据指标..."] };
}

async function processEmail(state: typeof GraphState.State) {
  return { messages: ["处理邮件: 提取发件人、主题、优先级..."] };
}

// 路由函数：根据 docType 返回下一个节点名称
function routeByDocType(state: typeof GraphState.State): string {
  switch (state.docType) {
    case "contract": return "process_contract";
    case "report":   return "process_report";
    case "email":    return "process_email";
    default:         return END;  // 未知类型直接结束
  }
}

const workflow = new StateGraph(GraphState)
  .addNode("classify", classifyDocument)
  .addNode("process_contract", processContract)
  .addNode("process_report", processReport)
  .addNode("process_email", processEmail)
  .addEdge(START, "classify")
  // 条件边：从 classify 出发，使用 routeByDocType 路由函数
  .addConditionalEdges("classify", routeByDocType)
  .addEdge("process_contract", END)
  .addEdge("process_report", END)
  .addEdge("process_email", END);

const app = workflow.compile();

// 测试：处理一份合同
const result = await app.invoke({ input: "采购合同-2024-001" });
console.log(result.messages);
// ["文档分类: contract", "处理合同: 提取条款、签署方、金额..."]

概念五：createReactAgent（预置 ReAct Agent）

createReactAgent 是 LangGraph 提供的一个预构建 Agent 节点， 它将 ReAct（Reasoning + Acting）循环封装为一个标准的图节点。 你可以将它直接插入到你的 StateGraph 中，与其他自定义节点组合使用。

TypeScript 
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { ChatOpenAI } from "@langchain/openai";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// 定义工具
const searchDocs = tool(
  async ({ query }) => `知识库搜索结果: 关于"${query}"的文档...`,
  {
    name: "search_docs",
    description: "搜索内部知识库文档",
    schema: z.object({ query: z.string().describe("搜索关键词") }),
  }
);

const calculate = tool(
  async ({ expression }) => `计算结果: ${eval(expression)}`,
  {
    name: "calculate",
    description: "执行数学计算",
    schema: z.object({ expression: z.string().describe("数学表达式") }),
  }
);

// 创建 ReAct Agent 节点
const reactAgentNode = createReactAgent({
  llm: new ChatOpenAI({ model: "gpt-4o-mini" }),
  tools: [searchDocs, calculate],
  // systemPrompt 是可选的系统提示
  systemPrompt: "你是一个智能助手，可以使用搜索和计算工具帮助用户。",
});

// 现在 reactAgentNode 可以像普通节点一样加入图中
// 注意：createReactAgent 返回的是一个 Runnable，需要适配状态接口
⚠️ createReactAgent 的输入/输出格式

createReactAgent 返回的对象期望输入和输出中包含 messages 字段（消息数组）。 当你将它集成到自定义 StateGraph 时，确保你的状态中有一个 messages 字段， 或者使用 .bind() 适配。

概念六：Human-in-the-Loop（人机协同）与 Interrupt

在某些关键步骤，你可能希望暂停工作流，等待人工审批或输入后再继续。 LangGraph 通过 interrupt 机制实现这一点。你可以在节点函数中抛出 interrupt， 工作流会暂停并等待外部恢复。

TypeScript 
import { interrupt } from "@langchain/langgraph";

// 在节点中使用 interrupt 暂停工作流
async function reviewStep(state: typeof GraphState.State) {
  // 向审批者发送信息，等待人工响应
  const approval = await interrupt({
    type: "human_review",
    question: "是否批准以下处理结果？",
    data: state.result,
    options: ["approve", "reject", "modify"],
  });

  // 当工作流恢复后，approval 包含人工输入的值
  return {
    messages: [`审批结果: ${JSON.stringify(approval)}`],
    approval: approval,
  };
}

// 使用方式：
// const app = workflow.compile();
// 
// 第一次执行会在 reviewStep 处暂停
// const result1 = await app.invoke({ input: "..." });
// 
// 恢复执行：
// const result2 = await app.invoke(null, {
//   configurable: { thread_id: "..." },  // 需要 thread_id
// });

概念七：Checkpoint 持久化

LangGraph 支持在图的每个步骤自动保存状态快照（Checkpoint）。 这让工作流具备可恢复性——即使程序崩溃，你也可以从最后一个 Checkpoint 恢复执行。 此外，Checkpointer 也是 Human-in-the-loop 的基础——工作流暂停后，状态被保存， 你可以稍后恢复。

TypeScript 
import { MemorySaver } from "@langchain/langgraph";

// 创建 Checkpointer
const checkpointer = new MemorySaver();

// 编译图时传入 Checkpointer
const app = workflow.compile({ checkpointer });

// 执行时指定 thread_id
const result = await app.invoke(
  { input: "处理这份文档" },
  { configurable: { thread_id: "doc-process-001" } }
);

// 如果执行被中断或暂停，可以通过相同的 thread_id 恢复
const resumed = await app.invoke(
  null,  // 不传新输入，从暂停点继续
  { configurable: { thread_id: "doc-process-001" } }
);

// 查看历史状态
const stateHistory = await app.getStateHistory({
  configurable: { thread_id: "doc-process-001" },
});

概念八：Subgraph（子图）

当工作流变得非常复杂时，你可以将一部分节点封装为子图（Subgraph）。 子图本身就是一个完整的 StateGraph，可以像普通节点一样嵌入到父图中。 这使得你可以模块化地构建和维护复杂工作流。

TypeScript 
// ========== 定义子图 ==========
const SubState = Annotation.Root({
  text: Annotation,
  result: Annotation,
});

const subWorkflow = new StateGraph(SubState)
  .addNode("step1", async (state) => ({
    result: `子图步骤1: ${state.text}`,
  }))
  .addNode("step2", async (state) => ({
    result: `子图步骤2: ${state.result}`,
  }))
  .addEdge(START, "step1")
  .addEdge("step1", "step2")
  .addEdge("step2", END);

const subgraph = subWorkflow.compile();

// ========== 在父图中使用子图 ==========
const ParentState = Annotation.Root({
  input: Annotation,
  finalResult: Annotation,
});

const parentWorkflow = new StateGraph(ParentState)
  .addNode("preprocess", async (state) => ({
    // 预处理后将数据传给子图
  }))
  // 将子图作为一个节点
  .addNode("sub_process", subgraph)
  .addNode("postprocess", async (state) => ({
    finalResult: `最终结果: ${state.finalResult}`,
  }))
  .addEdge(START, "preprocess")
  .addEdge("preprocess", "sub_process")
  .addEdge("sub_process", "postprocess")
  .addEdge("postprocess", END);

🔨 实战一：多步骤文档处理工作流

场景描述：你需要构建一个文档处理流水线，接收原始文档文本， 依次执行：内容验证 → 格式清洗 → 信息提取 → 摘要生成。每个步骤独立封装为节点， 状态在节点间流动。

核心知识点：StateGraph 基础构建、多节点串联、状态传递。

🧑‍💻 先自己写，写完再展开看参考答案
TypeScript 
import { StateGraph, Annotation, START, END } from "@langchain/langgraph";
import { ChatOpenAI } from "@langchain/openai";

// ==========================================
// 1. 定义文档处理的状态结构
// ==========================================
const DocState = Annotation.Root({
  rawText: Annotation,                          // 原始文本
  validated: Annotation({ default: () => false }),  // 是否通过验证
  cleanText: Annotation({ default: () => "" }),     // 清洗后的文本
  extractedFields: Annotation>({     // 提取的字段
    default: () => ({}),
  }),
  summary: Annotation({ default: () => "" }),       // 生成的摘要
  errors: Annotation({
    reducer: (left, right) => left.concat(right),
    default: () => [],
  }),
});

// ==========================================
// 2. 定义各个处理节点
// ==========================================

// 节点 1：内容验证 —— 检查文档是否为空、格式是否有效
async function validateDocument(state: typeof DocState.State) {
  const text = state.rawText;

  if (!text || text.trim().length === 0) {
    return {
      validated: false,
      errors: ["文档内容为空，无法处理"],
    };
  }

  // 检查最小长度
  if (text.trim().length < 10) {
    return {
      validated: false,
      errors: ["文档内容过短，无法进行有效分析"],
    };
  }

  // 检查是否包含有效内容（简单示例）
  const hasContent = /[a-zA-Z\u4e00-\u9fa5]/.test(text);
  if (!hasContent) {
    return {
      validated: false,
      errors: ["文档不包含有效文本内容"],
    };
  }

  return {
    validated: true,
    messages: ["文档验证通过"],
  };
}

// 节点 2：格式清洗 —— 去除多余空格、特殊字符，统一格式
async function cleanDocument(state: typeof DocState.State) {
  if (!state.validated) {
    return { errors: ["跳过清洗步骤：文档未通过验证"] };
  }

  let cleaned = state.rawText;

  // 去除多余空白行
  cleaned = cleaned.replace(/\n{3,}/g, "\n\n");
  // 去除首尾空格
  cleaned = cleaned.trim();
  // 统一换行符
  cleaned = cleaned.replace(/\r\n/g, "\n");
  // 去除特殊控制字符（保留换行和制表符）
  cleaned = cleaned.replace(/[\x00-\x08\x0B\x0C\x0E-\x1F]/g, "");

  return {
    cleanText: cleaned,
    messages: ["文档格式清洗完成"],
  };
}

// 节点 3：信息提取 —— 使用 LLM 提取关键字段
async function extractFields(state: typeof DocState.State) {
  if (!state.cleanText) {
    return { errors: ["跳过提取步骤：清洗文本为空"] };
  }

  const model = new ChatOpenAI({ model: "gpt-4o-mini" });

  // 使用结构化输出提取信息
  const extractionPrompt = `从以下文档中提取关键信息：

文档内容：
${state.cleanText.slice(0, 2000)}  // 限制长度

请以 JSON 格式返回以下字段（如果不存在则设为 "未找到"）：
- title: 文档标题
- author: 作者
- date: 日期
- type: 文档类型
- keywords: 关键词（逗号分隔）`;

  const response = await model.invoke([
    { role: "system", content: "你是一个文档信息提取专家。只返回 JSON，不要包含其他内容。" },
    { role: "human", content: extractionPrompt },
  ]);

  try {
    // 尝试从回复中提取 JSON
    const content = response.content.toString();
    const jsonMatch = content.match(/\{[\s\S]*\}/);
    const fields = jsonMatch ? JSON.parse(jsonMatch[0]) : {};

    return {
      extractedFields: fields,
      messages: [`信息提取完成: 共提取 ${Object.keys(fields).length} 个字段`],
    };
  } catch (e) {
    return {
      errors: [`信息提取解析失败: ${e}`],
      extractedFields: { title: "提取失败" },
    };
  }
}

// 节点 4：摘要生成
async function generateSummary(state: typeof DocState.State) {
  const fields = state.extractedFields;
  const text = state.cleanText;

  const summaryText = `📄 **文档摘要**
━━━━━━━━━━━━━━━━━━
标题: ${fields.title || "未识别"}
作者: ${fields.author || "未识别"}
日期: ${fields.date || "未识别"}
类型: ${fields.type || "未识别"}
关键词: ${fields.keywords || "未识别"}

📝 文档长度: ${text.length} 字符
📊 处理状态: ✅ 完成`;

  return {
    summary: summaryText,
    messages: ["摘要生成完成"],
  };
}

// ==========================================
// 3. 组装工作流
// ==========================================
const docWorkflow = new StateGraph(DocState)
  .addNode("validate", validateDocument)
  .addNode("clean", cleanDocument)
  .addNode("extract", extractFields)
  .addNode("summarize", generateSummary)
  .addEdge(START, "validate")
  .addEdge("validate", "clean")
  .addEdge("clean", "extract")
  .addEdge("extract", "summarize")
  .addEdge("summarize", END);

const docApp = docWorkflow.compile();

⚡ 高级实战：Human-in-the-Loop 审批流程

场景描述：构建一个文档发布审批工作流。用户提交文档后， 系统自动进行内容审核和安全检查，如果发现问题则暂停等待人工审批， 根据审批结果决定是发布文档、退回修改还是直接拒绝。

核心知识点：interrupt 暂停机制、 Checkpointer 持久化、条件恢复、人工决策集成。

TypeScript 
import {
  StateGraph, Annotation, START, END, interrupt,
  MemorySaver,
} from "@langchain/langgraph";
import { ChatOpenAI } from "@langchain/openai";

// ==========================================
// 1. 定义审批状态
// ==========================================
const ApprovalState = Annotation.Root({
  docTitle: Annotation({ default: () => "未命名文档" }),
  docContent: Annotation({ default: () => "" }),
  author: Annotation({ default: () => "未知" }),

  // 自动审核结果
  riskLevel: Annotation({ default: () => "low" }),  // low | medium | high
  reviewReport: Annotation({ default: () => "" }),
  flaggedIssues: Annotation({
    reducer: (left, right) => left.concat(right),
    default: () => [],
  }),

  // 人工审批结果
  approvalStatus: Annotation({ default: () => "pending" }), // pending | approved | rejected | revision
  approverComment: Annotation({ default: () => "" }),

  // 发布信息
  publishedAt: Annotation({ default: () => "" }),
  publishUrl: Annotation({ default: () => "" }),

  // 执行追踪
  currentStep: Annotation({ default: () => "submitted" }),
  logs: Annotation({
    reducer: (left, right) => left.concat(right),
    default: () => [],
  }),
});

// ==========================================
// 2. 节点：提交文档
// ==========================================
async function submitDocument(state: typeof ApprovalState.State) {
  const now = new Date().toISOString();
  return {
    currentStep: "submitted",
    logs: [`[${now}] 文档已提交: "${state.docTitle}"，作者: ${state.author}`],
  };
}

// ==========================================
// 3. 节点：自动内容审核
// ==========================================
async function autoReview(state: typeof ApprovalState.State) {
  const model = new ChatOpenAI({ model: "gpt-4o-mini", temperature: 0 });

  const reviewPrompt = `你是一个文档审核员。请审核以下文档内容，输出 JSON 格式的审核报告。

文档标题: ${state.docTitle}
文档内容:
${state.docContent.slice(0, 3000)}

请检查以下方面：
1. 是否包含敏感信息（个人隐私、机密数据、不当言论）
2. 内容质量（是否清晰、完整、专业）
3. 合规性（是否符合一般发布规范）

输出格式（JSON）：
{
  "riskLevel": "low|medium|high",
  "issues": ["问题1", "问题2", ...],
  "summary": "审核总结"
}`;

  const response = await model.invoke([
    { role: "system", content: "你是一个严谨的文档审核专家。只输出 JSON。" },
    { role: "human", content: reviewPrompt },
  ]);

  let report;
  try {
    const match = response.content.toString().match(/\{[\s\S]*\}/);
    report = match ? JSON.parse(match[0]) : {
      riskLevel: "low", issues: [], summary: "自动审核完成（解析失败，使用默认值）",
    };
  } catch {
    report = { riskLevel: "low", issues: [], summary: "自动审核完成（使用默认值）" };
  }

  const now = new Date().toISOString();
  return {
    riskLevel: report.riskLevel || "low",
    flaggedIssues: report.issues || [],
    reviewReport: report.summary || "无审核摘要",
    currentStep: "reviewed",
    logs: [
      `[${now}] 自动审核完成`,
      `[${now}] 风险等级: ${report.riskLevel || "low"}`,
      ...(report.issues?.length
        ? [`[${now}] 发现问题: ${report.issues.join("; ")}`]
        : [`[${now}] 未发现问题`]),
    ],
  };
}

// ==========================================
// 4. 路由函数：根据风险等级决定是否需要人工审批
// ==========================================
function routeAfterReview(state: typeof ApprovalState.State): string {
  // high 风险：直接进入人工审批
  // medium 风险：进入人工审批
  // low 风险：自动通过
  if (state.riskLevel === "high" || state.riskLevel === "medium") {
    return "human_approval";
  }
  return "auto_publish";
}

// ==========================================
// 5. 节点：人工审批（含 interrupt）
// ==========================================
async function humanApproval(state: typeof ApprovalState.State) {
  const now = new Date().toISOString();

  // 使用 interrupt 暂停工作流，等待人工输入
  const approval = await interrupt({
    type: "document_approval",
    docTitle: state.docTitle,
    riskLevel: state.riskLevel,
    issues: state.flaggedIssues,
    reviewReport: state.reviewReport,
    // 给审批者提供的选项
    options: [
      { value: "approved", label: "✅ 批准发布", description: "文档通过审核，允许发布" },
      { value: "revision", label: "✏️ 退回修改", description: "文档需要修改后重新提交" },
      { value: "rejected", label: "❌ 拒绝发布", description: "文档不符合要求，拒绝发布" },
    ],
    question: `请审批文档「${state.docTitle}」的发布请求：`,
  });

  // 当工作流恢复时，approval 包含审批者的选择
  const decision = typeof approval === "string" ? approval : (approval as any)?.value || "revision";

  return {
    approvalStatus: decision,
    currentStep: `human_${decision}`,
    logs: [`[${now}] 人工审批结果: ${decision}`],
  };
}

// ==========================================
// 6. 路由函数：根据审批结果决定下一步
// ==========================================
function routeAfterApproval(state: typeof ApprovalState.State): string {
  switch (state.approvalStatus) {
    case "approved": return "auto_publish";
    case "revision": return "revision_feedback";
    case "rejected": return "rejection_notice";
    default:         return "rejection_notice";
  }
}

  const now = new Date().toISOString();
  const publishId = `pub-${Date.now()}-${Math.random().toString(36).slice(2, 6)}`;

  return {
    approvalStatus: "approved",
    publishedAt: now,
    publishUrl: `https://docs.example.com/published/${publishId}`,
    currentStep: "published",
    logs: [
      `[${now}] ✅ 文档已自动发布`,
      `[${now}] 📎 发布链接: https://docs.example.com/published/${publishId}`,

}

// 8. 节点：退回修改通知

  const now = new Date().toISOString();
  return {
    currentStep: "revision_requested",
    logs: [
      `[${now}] ✏️ 文档已退回修改`,
      `[${now}] 📝 请参考审核意见修改后重新提交`,

📝 自测题：检查你对本章内容的理解
1. StateGraph 中的 Reducer（归约器）的作用是什么？

Reducer 定义了当图中多个节点向同一个状态字段写入时如何合并值。默认行为是"覆盖"（新值替换旧值）。通过自定义 reducer，可以实现"追加"（如消息列表的 concat）、"累加"（如计数器求和）等累积行为。如果没有正确配置 reducer，在循环或并行节点中可能会丢失数据。

2. addConditionalEdges 和普通 addEdge 有什么区别？

普通边（addEdge）定义固定的执行顺序——节点 A 执行完后一定会去节点 B。条件边（addConditionalEdges）则根据当前状态的动态值决定下一个节点——你需要提供一个路由函数，接收 State 并返回目标节点名称（或 END）。条件边是实现分支逻辑（if/else、switch-case）的关键工具。

3. Human-in-the-loop 中的 interrupt 机制如何工作？

interrupt() 函数会在当前节点暂停图执行，保存当前状态到 Checkpointer，并向外部返回一个消息对象（包含问题、选项等）。外部系统（如 UI 界面）可以展示这些信息给人工操作者。当操作者做出决策后，调用 app.invoke(null, { configurable: { thread_id } }) 恢复执行，interrupt 会返回操作者的输入。整个过程需要 Checkpointer 配合以持久化暂停状态。

4. createReactAgent 与自定义 Node 的区别是什么？

createReactAgent 是一个预构建的 Agent 节点，它内部封装了完整的 ReAct 循环（思考→调用工具→观察结果→继续思考→最终回答）。而自定义 Node 只是一个简单的函数，接收状态返回更新。createReactAgent 适合需要工具调用和自主推理的场景；自定义 Node 适合纯粹的确定性计算步骤（如数据清洗、格式转换）。在实际应用中，两者经常组合使用：自定义 Node 做预处理/后处理，createReactAgent 做核心的 LLM 交互。

5. 子图（Subgraph）解决了什么问题？

子图解决了工作流复杂化后的模块化问题。当一张图包含 20+ 个节点时，管理和维护变得困难。子图允许你将一组相关的节点封装为一个独立的图，然后在父图中像使用普通节点一样引用它。好处包括：（1）关注点分离，每个子图职责单一；（2）子图可以独立测试和复用；（3）降低父图的理解和维护成本。

🐛 常见错误

❌ 错误 1：忘记安装 @langchain/langgraph 包

症状：导入 StateGraph、Annotation 时报 Cannot find module '@langchain/langgraph'。

原因：LangGraph 是一个独立的包，需要单独安装，它不在 langchain 或 @langchain/core 中。

解决：运行 npm install @langchain/langgraph。

❌ 错误 2：条件边路由函数返回了不存在的节点名

症状：执行时抛出类似 Node 'xyz' not found in graph 的错误。

原因：路由函数返回的字符串没有匹配到任何已注册的节点名称，包括可能拼写错误或忘记 addNode。

解决：确保路由函数返回的所有可能值都在图中通过 addNode 注册了。常见做法是将路由函数定义为返回联合类型：function router(state: State): "node_a" | "node_b" | typeof END。

❌ 错误 3：interrupt 使用但没有配置 Checkpointer

症状：调用 interrupt() 时抛出 Interrupt requires a checkpointer to be configured 错误。

原因：interrupt 机制依赖 Checkpointer 来保存暂停时的状态，没有 Checkpointer 无法工作。

解决：编译图时传入 Checkpointer 实例：workflow.compile({ checkpointer: new MemorySaver() })。

❌ 错误 4：状态字段的 Reducer 冲突

症状：图中某些节点的状态更新"丢失"了，或者出现了意料之外的值。

原因：默认的 reducer 是"覆盖"行为。如果多个节点向同一个字段写入，后面的会覆盖前面的。例如在循环中，如果你希望累积消息列表但没有配置 concat reducer，每次循环都会清空之前的消息。

解决：根据预期行为显式配置 reducer：累积用 (left, right) => left.concat(right)，求和用 (left, right) => left + right，保留最新值用默认覆盖。
