让 Agent 拥有持久化的对话上下文与跨会话记忆能力

💡 为什么 Agent 需要记忆

想象一下这个场景：你走进一家常去的咖啡店，店员却完全不认识你，每次都问你同样的问题 —— "请问您要点什么？""您的名字？""要加糖吗？"。这种体验让人非常沮丧，对吧？

没有记忆的 Agent 就像一个每天失忆的人，每次对话都是初次见面。

在前几章中，我们构建的 Agent 每次调用都是独立的 —— 它无法记住上一轮对话中用户说了什么，也无法积累任何关于用户的了解。这意味着：

  • 用户必须在每次提问时重复提供背景信息
  • Agent 无法根据对话历史做出个性化回应
  • 多步骤任务无法通过对话逐步推进
  • 用户体验极度碎片化，缺乏连贯性

记忆系统解决了这些问题。通过为 Agent 添加记忆能力，我们可以让它：

  • 记住对话上下文 — 知道之前聊了什么，避免重复询问
  • 积累用户偏好 — 了解用户的习惯、喜好和常用设置
  • 追踪任务进度 — 在多轮对话中逐步完成复杂任务
  • 提供个性化服务 — 根据历史记录给出更精准的建议

在 LangChain.js 中，记忆系统分为两个层次：短期记忆（Checkpointer）和长期记忆（Store）。它们各自解决不同场景下的记忆需求，接下来我们将逐一深入学习。

🔄 短期记忆 vs 长期记忆

在开始编码之前，我们先来理解两种记忆类型的核心区别。可以用一个简单的类比来帮助理解：

📝 类比：便签纸 vs 文件柜

想象你的办公桌上有两种记录工具：

  • 便签纸（短期记忆） — 你正在处理某个项目时，会在桌面的便签纸上快速记录相关信息。这个项目结束后，便签纸可能会被丢掉，也可能被收起来。每张便签纸对应一个特定的任务或会话。
  • 文件柜（长期记忆） — 办公室角落的文件柜里存放着所有历史资料。无论你换了多少个项目，文件柜里的信息始终可以查阅。它跨越所有会话，是持久化的知识存储。
特性 短期记忆（Checkpointer） 长期记忆（Store）
作用范围 单个会话（thread）内 跨所有会话
存储内容 完整的对话消息历史 键值对形式的结构化数据
生命周期 随会话存在，会话结束后可能清除 永久保存，直到手动删除
标识方式 thread_id namespace + key
典型用途 多轮聊天、任务跟进 用户偏好、历史摘要、知识库
LangChain 实现 MemorySaver / 数据库 Checkpointer InMemoryStore / 自定义 Store

简单来说：短期记忆帮你记住"这次聊了什么"，长期记忆帮你记住"关于这个人我知道什么"。在实际应用中，两种记忆通常配合使用，共同构建一个智能且贴心的 Agent 系统。

💾 Checkpointer 详解

Checkpointer 是 LangGraph 中实现短期记忆的核心机制。它的作用是在 Agent 执行的每个步骤中保存状态快照，使得对话可以在任意节点恢复或回溯。理解 Checkpointer 的工作原理对于构建可靠的对话系统至关重要。

Checkpointer 的工作原理

当 Agent 接收到一条消息并开始处理时，Checkpointer 会在以下关键节点保存状态：

  • 消息接收时 — 保存用户输入的消息
  • 模型调用前 — 记录即将发送给模型的完整上下文
  • 工具调用时 — 保存工具选择和参数信息
  • 工具返回时 — 记录工具执行的结果
  • 响应生成后 — 保存最终的 AI 回复

每个状态快照都与一个 thread_id 关联。thread_id 就像对话的唯一标识符，同一个 thread_id 下的所有消息和状态变化都属于同一个对话线程。当你再次使用相同的 thread_id 发起请求时，Checkpointer 会自动加载该线程的所有历史状态，Agent 就能"记住"之前发生的一切。

MemorySaver：内存存储

MemorySaver 是最简单的 Checkpointer 实现，它将所有状态保存在内存中。非常适合开发和测试阶段使用。

checkpointer-basics.ts TypeScript 
// 导入 MemorySaver —— 内存版 Checkpointer
import { MemorySaver } from "@langchain/langgraph";

// 创建 MemorySaver 实例
// 所有对话状态都保存在内存中，程序重启后会丢失
const checkpointer = new MemorySaver();

// 在创建 Agent 时传入 checkpointer
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({ modelName: "gpt-4o-mini" });

const agent = createAgent({
  model,
  tools: [],
  // 关键配置：指定 checkpointer
  checkpointer,
});

// 第一轮对话 —— 使用 thread_id 标识会话
const result1 = await agent.invoke(
  { messages: [{ role: "user", content: "我叫小明，我是一个前端工程师" }] },
  { configurable: { thread_id: "user-session-001" } }
);
console.log("第一轮回复:", result1.messages.at(-1)?.content);

// 第二轮对话 —— 使用相同的 thread_id，Agent 能记住之前的内容
const result2 = await agent.invoke(
  { messages: [{ role: "user", content: "你还记得我是谁吗？" }] },
  { configurable: { thread_id: "user-session-001" } }
);
console.log("第二轮回复:", result2.messages.at(-1)?.content);
// Agent 会回答："你是小明，一个前端工程师！"

Persistent Checkpointer：持久化存储

在生产环境中，你不可能依赖内存存储 —— 服务器重启意味着所有记忆消失。这时你需要一个持久化的 Checkpointer，将对话状态保存到数据库中。

LangGraph 提供了多种持久化 Checkpointer 的实现：

  • SqliteSaver — 基于 SQLite，适合小型应用和单机部署
  • PostgresSaver — 基于 PostgreSQL，适合生产环境的多实例部署
  • MongoDBSaver — 基于 MongoDB，适合需要灵活 schema 的场景
persistent-checkpointer.ts TypeScript 
// 生产环境：使用 SQLite 持久化存储
// 需要安装：npm install better-sqlite3
import { SqliteSaver } from "@langchain/langgraph-checkpoint-sqlite";

// 连接到 SQLite 数据库文件
const checkpointer = SqliteSaver.fromConnString("./checkpoints.db");

// 创建 Agent，使用持久化 Checkpointer
const agent = createAgent({
  model: new ChatOpenAI({ modelName: "gpt-4o-mini" }),
  tools: [],
  checkpointer, // 数据会保存到文件，重启后仍然可用
});

// 即使程序重启，使用相同的 thread_id 也能恢复对话历史
const result = await agent.invoke(
  { messages: [{ role: "user", content: "我们之前聊到哪了？" }] },
  { configurable: { thread_id: "persistent-session-001" } }
);
⚠️ 选择 Checkpointer 的建议

开发和测试阶段使用 MemorySaver，因为它零配置、启动快。但一旦进入生产环境，请务必切换到持久化方案。没有持久化的记忆系统就像一个优秀的店员每天下班后都会失忆 —— 第二天一切从头开始。

🛠️ 实战：带记忆的聊天 Agent

现在让我们构建一个完整的示例：一个能记住对话上下文的聊天 Agent。我们会模拟一个旅行顾问场景，用户可以与它进行多轮对话，Agent 能够记住之前讨论的目的地、预算等信息。

chat-agent-memory.ts TypeScript 
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { MemorySaver } from "@langchain/langgraph";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// ========== 第一步：创建记忆存储 ==========
const checkpointer = new MemorySaver();

// ========== 第二步：定义旅行相关工具 ==========
// 搜索航班的工具
const searchFlights = tool(
  async ({ destination, date }) => {
    // 模拟航班搜索结果
    return JSON.stringify({
      destination,
      date,
      flights: [
        { airline: "国航 CA1234", price: 1280, time: "08:00-11:30" },
        { airline: "东航 MU5678", price: 1150, time: "14:00-17:20" },
        { airline: "南航 CZ9012", price: 980, time: "19:00-22:15" },
      ],
    });
  },
  {
    name: "search_flights",
    description: "搜索指定目的地和日期的航班信息",
    schema: z.object({
      destination: z.string().describe("目的地城市"),
      date: z.string().describe("出发日期，格式 YYYY-MM-DD"),
    }),
  }
);

// 搜索酒店的工具
const searchHotels = tool(
  async ({ city, checkIn, checkOut, budget }) => {
    return JSON.stringify({
      city,
      checkIn,
      checkOut,
      hotels: [
        { name: "城市精品酒店", pricePerNight: 380, rating: 4.5 },
        { name: "商务连锁酒店", pricePerNight: 260, rating: 4.0 },
        { name: "青年旅舍", pricePerNight: 80, rating: 3.5 },
      ],
    });
  },
  {
    name: "search_hotels",
    description: "搜索指定城市的酒店信息",
    schema: z.object({
      city: z.string().describe("城市名称"),
      checkIn: z.string().describe("入住日期"),
      checkOut: z.string().describe("退房日期"),
      budget: z.string().describe("每晚预算范围"),
    }),
  }
);

// ========== 第三步：创建带记忆的 Agent ==========
const agent = createAgent({
  model: new ChatOpenAI({ modelName: "gpt-4o-mini" }),
  tools: [searchFlights, searchHotels],
  checkpointer, // 注入记忆能力
  systemMessage: `你是一个专业的旅行顾问助手。
请根据用户的偏好和历史对话信息，提供个性化的旅行建议。
记住用户之前提到的目的地、预算、出行日期等信息，避免重复询问。`,
});

// ========== 第四步：模拟多轮对话 ==========
const threadId = "travel-advisor-session";

// 第一轮：用户介绍旅行需求
console.log("=== 第一轮对话 ===");
const response1 = await agent.invoke(
  {
    messages: [
      { role: "user", content: "你好，我计划下个月去成都玩，预算大概3000元左右" }
    ],
  },
  { configurable: { thread_id: threadId } }
);
console.log("Agent:", response1.messages.at(-1)?.content);

// 第二轮：用户追问细节，Agent 应该记住目的地和预算
console.log("\n=== 第二轮对话 ===");
const response2 = await agent.invoke(
  {
    messages: [
      { role: "user", content: "帮我查一下7月15号的航班，顺便看看酒店" }
    ],
  },
  { configurable: { thread_id: threadId } }
);
console.log("Agent:", response2.messages.at(-1)?.content);
// Agent 会自动搜索成都的航班和酒店，无需再次询问目的地

// 第三轮：用户修改偏好
console.log("\n=== 第三轮对话 ===");
const response3 = await agent.invoke(
  {
    messages: [
      { role: "user", content: "预算提高到5000吧，帮我升级一下酒店推荐" }
    ],
  },
  { configurable: { thread_id: threadId } }
);
console.log("Agent:", response3.messages.at(-1)?.content);
// Agent 记得之前推荐的酒店，并根据新预算给出升级建议

这个示例展示了短期记忆的核心价值：在同一个会话中，Agent 能够自动积累和利用对话上下文。用户不需要在每条消息中重复说明目的地是成都、预算已经调整到5000元 —— Agent 通过 Checkpointer 自动保存的历史消息就能获取这些信息。

💡 理解 thread_id 的重要性

thread_id 是连接用户和对话记忆的桥梁。在真实应用中，你通常会为每个用户会话生成一个唯一的 thread_id（比如使用 UUID 或用户ID+时间戳的组合）。相同的 thread_id 意味着同一个对话，不同的 thread_id 则是全新的对话。

🏛️ 长期记忆（Store）

短期记忆解决了"这次对话聊了什么"的问题，但有些信息需要跨越多个会话持久存在。比如：

  • 用户的基本偏好（喜欢靠窗座位、素食主义者、偏好经济型酒店）
  • 用户的历史摘要（过去三个月去了哪些城市、消费水平如何）
  • 系统级别的知识（产品目录、常见问题库）

这些信息需要在不同的对话会话之间共享和持久化。这就是 Store（长期记忆存储）的用武之地。

BaseStore 与 InMemoryStore

LangChain 的 Store 系统基于 BaseStore 接口，使用命名空间（namespace）和键（key）的二级结构来组织数据。你可以把它想象成一个带有文件夹的键值存储系统。

store-basics.ts TypeScript 
// 导入 InMemoryStore —— 内存版长期记忆存储
import { InMemoryStore } from "@langchain/langgraph";

// 创建 Store 实例
const store = new InMemoryStore();

// 存储用户偏好 —— namespace 定义"文件夹"，key 定义"文件名"
await store.put(
  ["user_preferences", "user-001"],  // namespace: 分类路径
  "travel_prefs",                     // key: 具体的键名
  {
    favoriteCity: "成都",
    budgetRange: "3000-5000",
    preferWindowSeat: true,
    dietaryRestrictions: "素食",
    lastUpdated: new Date().toISOString(),
  }
);

// 读取用户偏好
const prefs = await store.get(["user_preferences", "user-001"], "travel_prefs");
console.log("用户偏好:", prefs);

// 搜索某个命名空间下的所有条目
const allUserPrefs = await store.search(["user_preferences", "user-001"]);
console.log("该用户的所有记忆:", allUserPrefs);

在 Agent 中使用 Store

要让 Agent 的工具能够访问长期记忆，需要在创建 Agent 时传入 store，然后在工具内部通过 runtime.store 访问。

agent-with-store.ts TypeScript 
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { InMemoryStore, MemorySaver } from "@langchain/langgraph";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// 创建长期记忆存储
const store = new InMemoryStore();
// 同时也可以搭配短期记忆
const checkpointer = new MemorySaver();

// 定义一个能读写长期记忆的工具
const savePreference = tool(
  async ({ key, value }, config) => {
    // 从 config 中获取 store（通过 runtime 注入）
    // 注意：在 LangGraph 中，store 通过 config 的 runtime 传入
    const userId = config.configurable?.user_id ?? "default";

    // 这里我们直接使用外部的 store 变量（简化示例）
    await store.put(
      ["user_preferences", userId],
      key,
      { value, savedAt: new Date().toISOString() }
    );

    return `已保存偏好设置：${key} = ${value}`;
  },
  {
    name: "save_preference",
    description: "保存用户的偏好设置到长期记忆中",
    schema: z.object({
      key: z.string().describe("偏好的名称，如 favorite_color"),
      value: z.string().describe("偏好的值，如 蓝色"),
    }),
  }
);

const getPreference = tool(
  async ({ key }, config) => {
    const userId = config.configurable?.user_id ?? "default";
    const result = await store.get(["user_preferences", userId], key);

    if (!result) {
      return `未找到名为 "${key}" 的偏好设置`;
    }
    return `偏好设置 ${key} 的值为: ${JSON.stringify(result)}`;
  },
  {
    name: "get_preference",
    description: "从长期记忆中读取用户的偏好设置",
    schema: z.object({
      key: z.string().describe("要查询的偏好名称"),
    }),
  }
);

// 创建同时具备短期和长期记忆的 Agent
const agent = createAgent({
  model: new ChatOpenAI({ modelName: "gpt-4o-mini" }),
  tools: [savePreference, getPreference],
  checkpointer, // 短期记忆：对话历史
  store,        // 长期记忆：持久化数据
});

⚙️ 实战：用户偏好记忆

让我们构建一个更完整的示例：一个能够记住用户偏好的个性化助手。这个助手可以在不同的对话会话中识别同一个用户，并自动调取其历史偏好。

preference-agent.ts TypeScript 
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { InMemoryStore, MemorySaver } from "@langchain/langgraph";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// ========== 创建记忆系统 ==========
const store = new InMemoryStore();
const checkpointer = new MemorySaver();

// ========== 定义记忆工具 ==========

// 保存用户偏好
const saveUserPreference = tool(
  async ({ category, key, value }, config) => {
    const userId = config.configurable?.user_id ?? "anonymous";
    const namespace = ["preferences", userId, category];

    // 读取该类别下已有的偏好
    const existing = await store.get(namespace, key);

    await store.put(namespace, key, {
      value,
      updatedAt: new Date().toISOString(),
      timesAccessed: existing?.timesAccessed ?? 0,
    });

    return `✅ 已保存偏好 [${category}/${key}]: ${value}`;
  },
  {
    name: "save_user_preference",
    description: "保存用户的偏好设置。category 是分类（如 饮食、旅行、工作），key 是具体偏好名，value 是偏好值。",
    schema: z.object({
      category: z.string().describe("偏好分类，如：饮食、旅行、工作、娱乐"),
      key: z.string().describe("具体偏好名称"),
      value: z.string().describe("偏好值"),
    }),
  }
);

// 查询用户偏好
const getUserPreferences = tool(
  async ({ category }, config) => {
    const userId = config.configurable?.user_id ?? "anonymous";
    const namespace = ["preferences", userId, category || "general"];

    const results = await store.search(namespace);

    if (results.length === 0) {
      return `该类别下暂无保存的偏好设置`;
    }

    // 格式化返回结果
    const formatted = results.map(
      (item) => `  - ${item.key}: ${item.value.value}`
    ).join("\n");

    return `用户偏好设置（${category}）:\n${formatted}`;
  },
  {
    name: "get_user_preferences",
    description: "查询用户在某个类别下的所有偏好设置",
    schema: z.object({
      category: z.string().describe("偏好分类名称"),
    }),
  }
);

// ========== 创建个性化助手 Agent ==========
const agent = createAgent({
  model: new ChatOpenAI({ modelName: "gpt-4o-mini" }),
  tools: [saveUserPreference, getUserPreferences],
  checkpointer,
  store,
  systemMessage: `你是一个贴心的个性化助手。你的核心能力是记住用户的偏好。

💡 namespace 设计最佳实践

Store 的 namespace 是一个字符串数组，建议按层级组织：["类型", "用户ID", "分类"]。这样既便于按用户隔离数据，也便于按分类搜索。例如 ["preferences", "user-001", "饮食"] 清晰地表达了"用户001的饮食偏好"这一语义。

🗜️ 上下文压缩

短期记忆虽然强大，但也有一个实际问题：对话越长，发送给模型的上下文就越大。当对话持续数十轮甚至上百轮时，累积的消息历史可能轻松超过模型的上下文窗口限制（token 上限），或者导致 API 调用成本飙升。

上下文压缩就是解决这个问题的关键技术。它的核心思想是：在对话变长时，自动将早期的消息压缩为摘要，只保留最近的消息原文。这样既保留了重要的历史信息，又控制了上下文大小。

summarizationMiddleware：自动压缩

LangChain 提供了 summarizationMiddleware，可以自动监控对话长度，并在超过阈值时触发压缩。

context-compression.ts TypeScript 
import { createAgent } from "langchain";
import { ChatOpenAI } from "@langchain/openai";
import { MemorySaver } from "@langchain/langgraph";
import { summarizationMiddleware } from "langchain/agents";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// ========== 创建用于压缩的模型 ==========
// 推荐使用较小的模型来做摘要，节省成本
const summaryModel = new ChatOpenAI({
  modelName: "gpt-4o-mini",
  temperature: 0, // 摘要需要确定性输出
});

// ========== 创建压缩中间件 ==========
const memoryMiddleware = summarizationMiddleware({
  // 使用哪个模型来生成摘要
  model: summaryModel,
  // 当消息数量超过这个阈值时触发压缩
  maxTokensBeforeSummary: 4000,
  // 压缩后保留最近多少条原始消息
  messagesToKeep: 6,
  // 自定义摘要提示词
  summaryPrompt: `请将以下对话历史压缩为简洁的摘要。
保留关键信息：用户的需求、重要决定、已确认的偏好、
任务进度和待办事项。忽略寒暄和重复信息。`,
});

// ========== 创建带压缩能力的 Agent ==========
const checkpointer = new MemorySaver();

const agent = createAgent({
  model: new ChatOpenAI({ modelName: "gpt-4o-mini" }),
  tools: [/* 你的工具列表 */],
  checkpointer,
  // 将压缩中间件加入 Agent
  middleware: [memoryMiddleware],
});

// 长对话场景 —— 前20轮正常对话
const threadId = "long-session-001";

for (let i = 1; i <= 20; i++) {
  await agent.invoke(
    {
      messages: [
        { role: "user", content: `这是第${i}轮对话消息，讨论话题${i}` }
      ],
    },
    { configurable: { thread_id: threadId } }
  );
}
// 当消息累积到一定量后，summarizationMiddleware 会自动：
// 1. 将早期消息压缩为一段摘要
// 2. 只保留最近 6 条原始消息
// 3. 以 [对话摘要] + [最近消息] 的形式发送给模型

手动上下文管理

除了自动压缩，有时你也需要手动管理上下文。例如，在某些场景下你只想保留特定类型的消息，或者需要在压缩前提取特定信息。

manual-context.ts TypeScript 
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage, AIMessage, SystemMessage } from "@langchain/core/messages";

// 手动管理上下文的策略
function compressMessages(messages: Array<any>, keepRecent: number = 6) {
  // 如果消息不多，不需要压缩
  if (messages.length <= keepRecent + 2) {
    return messages;
  }

  // 分离出需要压缩的旧消息和需要保留的新消息
  const oldMessages = messages.slice(0, -keepRecent);
  const recentMessages = messages.slice(-keepRecent);

  // 将旧消息转换为摘要文本
  const summaryText = oldMessages
    .map((msg) => {
      const role = msg instanceof HumanMessage ? "用户" : "助手";
      return `${role}: ${msg.content}`;
    })
    .join("\n");

  // 创建摘要消息
  const summaryMessage = new SystemMessage(
    `[以下是早期对话的摘要]\n${summaryText}\n[摘要结束]`
  );

  // 返回：摘要 + 最近消息
  return [summaryMessage, ...recentMessages];
}

// 在发送请求前手动压缩
const fullHistory = [
  new HumanMessage("你好，我想学习 TypeScript"),
  new AIMessage("好的！TypeScript 是 JavaScript 的超集..."),
  // ... 假设有很多轮对话
  new HumanMessage("那泛型怎么用？"),
  new AIMessage("泛型允许你创建可复用的组件..."),
  new HumanMessage("能给个实际例子吗？"),
  new AIMessage("当然，比如一个通用的 API 响应类型..."),
];

const compressed = compressMessages(fullHistory, 4);
console.log("压缩后消息数量:", compressed.length);
⚠️ 压缩的注意事项
  • 压缩是有损的 —— 细节信息可能在压缩过程中丢失
  • 不要压缩得太激进，保留足够的近期消息让模型理解当前上下文
  • 对于关键信息（如用户明确的偏好、合同条款等），建议额外保存到长期记忆
  • 压缩本身也会消耗 token，需要在压缩频率和成本之间找到平衡

完成以下问题，检验你对本章内容的理解程度：

问题 1：短期记忆的标识

在 LangGraph 中，用来标识一个对话线程（短期记忆的基本单位）的关键参数是什么？

查看答案

thread_id。它通过 configurable 传递，例如 { configurable: { thread_id: "my-thread" } }。相同的 thread_id 会加载该线程的所有历史状态。

问题 2：MemorySaver 的适用场景

为什么 MemorySaver 不适合在生产环境中使用？如果需要生产级别的持久化，应该选择什么方案？

查看答案

MemorySaver 将所有数据保存在内存中，服务器重启后数据会全部丢失。生产环境应该使用持久化的 Checkpointer，如 SqliteSaver（小型应用）、PostgresSaver（企业级应用）或 MongoDBSaver。

问题 3：Store 的 namespace 作用

在 InMemoryStore 中，namespace 参数的作用是什么？请设计一个适合多用户偏好存储的 namespace 结构。

查看答案

namespace 是一个字符串数组，用于组织和隔离数据，类似于文件系统的目录结构。多用户偏好存储的 namespace 示例：["preferences", userId, category]，例如 ["preferences", "user-001", "饮食"]。这样可以按用户和类别灵活查询数据。

问题 4：上下文压缩

当对话变得很长时，summarizationMiddleware 如何工作？它解决了什么问题？

查看答案

summarizationMiddleware 会在对话消息超过设定阈值时，自动将早期消息压缩为摘要，只保留最近几条原始消息。它解决了两个问题：（1）避免超出模型的上下文窗口限制；（2）降低 API 调用的 token 成本。压缩是有损的，建议同时将关键信息保存到长期记忆。

🐛 常见错误

❌ 错误 1：忘记传递 thread_id

症状：每次调用 Agent，它都无法记住之前的消息，表现得像全新对话一样。

原因：在 invoke 时没有通过 configurable 传递 thread_id，导致每次调用都创建了新的会话线程。

解决：确保每次调用都传递一致的 thread_id：

正确写法 TypeScript 
// ❌ 错误：没有 thread_id
await agent.invoke({ messages: [{ role: "user", content: "你好" }] });

// ❌ 错误：每次使用不同的 thread_id
await agent.invoke(
  { messages: [...] },
  { configurable: { thread_id: `session-${Date.now()}` } } // 每次都不同！

// ✅ 正确：使用固定的 thread_id
await agent.invoke(
  { messages: [...] },
  { configurable: { thread_id: "my-session-001" } } // 保持一致
);
❌ 错误 2：内存存储丢失数据

症状：开发时记忆功能正常，部署到服务器后所有记忆消失。

原因：使用 MemorySaver（内存存储）或 InMemoryStore 进入了生产环境，服务器重启或扩缩容时内存数据被清除。

解决：生产环境使用持久化方案，如 SqliteSaver、PostgresSaver，或配合数据库实现自定义 Store。

❌ 错误 3：上下文溢出（Token 超限）

症状：长对话进行到一定轮次后，API 返回 "maximum context length exceeded" 错误。

原因：没有配置上下文压缩，随着对话增长，累积的消息总 token 数超过了模型的上下文窗口限制。

解决：添加 summarizationMiddleware 自动压缩旧消息，或手动管理消息列表长度：

添加压缩中间件 TypeScript 
// 配置 summarizationMiddleware 防止上下文溢出
import { summarizationMiddleware } from "langchain/agents";

const agent = createAgent({
  model,
  tools,
  checkpointer,
  middleware: [
    summarizationMiddleware({
      model: new ChatOpenAI({ modelName: "gpt-4o-mini" }),
      maxTokensBeforeSummary: 4000, // 超过 4000 token 时触发压缩
      messagesToKeep: 6,            // 保留最近 6 条原始消息
    }),
  ],
});

📝 本章小结

本章我们深入学习了 LangChain.js 的记忆系统，这是构建智能对话 Agent 的关键能力。让我们回顾一下核心要点：

  • 记忆的本质 — 没有记忆的 Agent 每次对话都是初次见面，记忆让它能积累上下文、追踪进度、提供个性化服务
  • 短期记忆（Checkpointer） — 通过 MemorySaver 或持久化方案实现，以 thread_id 标识对话，保存完整的对话消息历史
  • 长期记忆（Store） — 通过 InMemoryStore 或自定义 Store 实现，以 namespace + key 组织数据，跨会话持久化存储
  • 上下文压缩 — 使用 summarizationMiddleware 自动压缩过长对话，防止 token 超限，同时降低 API 成本
  • 记忆配合使用 — 短期记忆记住"这次聊了什么"，长期记忆记住"关于这个用户我知道什么"，两者配合构建完整记忆体系
