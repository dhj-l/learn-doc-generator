第 1 章 · 概述与环境搭建

了解 LangChain 是什么、为什么需要它，以及如何从零搭建开发环境

📋 信息来源说明
本章内容基于 docs.langchain.com 官方文档站，采用 LangChain v1.x 全新 createAgent 架构。建议学习过程中对照官方文档确认关键 API。
🎯 本章目标
  • 理解 LangChain 是什么，以及它在 AI 开发中的定位
  • 了解 LangChain 生态系统的各个组成部分及其相互关系
  • 掌握 LangChain 的核心理念：Agent = Model + Harness
  • 从零搭建一个 TypeScript 开发环境
  • 成功运行你的第一个 LangChain 程序
  • 建立良好的项目结构习惯

🦜 什么是 LangChain

如果你曾经用过 ChatGPT 或者其他大语言模型（LLM），你一定体验过它们令人惊叹的对话能力。但当你想要在自己的应用中集成这些 AI 能力时，问题就来了：如何让 AI 不只是聊天，而是真正帮用户完成任务？

这就是 LangChain 诞生的原因。

LangChain 是一个开源框架，专门用于构建由大语言模型（LLM）驱动的应用程序。它提供了一整套工具和抽象，让你可以轻松地将 AI 模型与外部数据源、工具和工作流连接起来，从而构建出真正有用的智能应用。

💡 一个形象的类比

想象你在搭建一个智能助手，LangChain 就像一个完整的工具箱：

  • Agent（智能体）= 大脑 —— 负责理解用户需求、制定计划、做出决策
  • Tools（工具）= 双手 —— 负责执行具体任务，比如搜索网页、查询数据库、发送邮件
  • Memory（记忆）= 记事本 —— 负责记住之前的对话和上下文，让 AI 不会「健忘」
  • Model（模型）= 思考引擎 —— 底层的大语言模型，提供推理和生成能力

Agent = Model + Harness

在 LangChain 的核心理念中，Agent（智能体）并不是一个神秘的概念。它本质上就是：

Agent = Model（模型）+ Harness（控制框架）

这里的 Model 指的是底层的大语言模型，比如 GPT-4、Claude、Gemini 等。而 Harness（控制框架）则是 LangChain 提供的「外壳」，它负责：

  • 管理对话流程 —— 决定什么时候调用模型、什么时候调用工具
  • 解析输出 —— 将模型的自然语言输出转化为结构化的工具调用指令
  • 处理错误 —— 当工具调用失败时进行重试或回退
  • 维护状态 —— 追踪多轮对话的上下文和中间结果

换句话说，LangChain 帮你把一个「只会说话」的语言模型，变成了一个「能做事情」的智能助手。模型负责思考和表达，LangChain 负责协调和执行。这种分离设计使得你可以轻松切换底层模型（比如从 OpenAI 换成 Anthropic），而不需要重写整个应用逻辑。

LangChain v1.x 的新架构

LangChain 经历了多次重大迭代。在 v1.x 版本中，引入了全新的 createAgent 架构，这是一个重要变化。新架构的核心特点是：

  • 声明式创建 —— 使用 createAgent() 函数一步创建 Agent，无需手动组装各个组件
  • 内置工具调用 —— Agent 自动处理模型的工具调用请求，无需自己编写解析逻辑
  • 类型安全 —— 完整的 TypeScript 类型支持，开发时就能获得自动补全和类型检查
  • 灵活的中间件机制 —— 通过中间件可以拦截和增强 Agent 的行为，而不需要修改核心逻辑

这套新架构大幅降低了入门门槛，同时也为高级用户保留了足够的灵活性。接下来让我们详细看看 LangChain 的生态系统。

🌐 LangChain 生态系统

LangChain 并不是一个单独的包，而是一个由多个组件组成的生态系统。理解它们之间的关系，对于后续的学习至关重要。就像一座城市有不同功能的建筑一样，LangChain 生态也有不同分工的模块：

组件 包名 职责
LangChain Core @langchain/core 核心原语：消息、工具、输出解析器等基础抽象
LangChain langchain 主要的运行框架（Harness），包含 Agent 创建、链（Chain）等高级功能
模型集成 @langchain/openai、@langchain/anthropic 等 各大模型提供商的连接器，负责与具体的 LLM API 通信
LangGraph @langchain/langgraph 底层编排框架，用于构建复杂的多步骤工作流和有状态的 Agent
Deep Agents @langchain/deep-agents 开箱即用的 Agent 实现，提供「电池全包」的高级 Agent 功能
LangSmith 平台服务 可观测性平台：追踪、调试、评估你的 LLM 应用

包的依赖关系

这些包之间的依赖关系就像俄罗斯套娃一样层层嵌套：

  • @langchain/core 是最底层的基础包，定义了所有核心接口和类型
  • langchain 依赖 @langchain/core，在其基础上构建了 Agent、Chain 等高级抽象
  • @langchain/openai 等模型集成包依赖 @langchain/core，实现具体的模型连接
  • @langchain/langgraph 构建在 Core 之上，提供图结构的编排能力

这种模块化设计的好处是显而易见的：你只需要安装你实际使用的部分。如果只需要调用 OpenAI 的模型做简单对话，只需要 @langchain/core + @langchain/openai 就够了。只有当你需要构建复杂的 Agent 时，才需要引入 langchain 或 @langchain/langgraph。

📌 关于 LangSmith
LangSmith 是 LangChain 团队提供的可观测性平台（SaaS 服务），类似于给你的 AI 应用装上了一个监控摄像头。它可以记录每一次模型调用的输入输出、耗时、Token 消耗等信息。在开发调试阶段非常有用。它不是必须的，但强烈推荐在正式项目中使用。本教程中会偶尔提及它的使用方式。

如何选择组件

对于初学者来说，一个常见的困惑是：「我到底该安装哪些包？」这里给出一个简单的决策指南：

  • 刚入门学习 —— 安装 langchain + @langchain/core + 一个模型集成包（如 @langchain/openai），这三者足以覆盖本教程前 5 章的所有内容
  • 需要流式输出和高级编排 —— 再加上 @langchain/langgraph
  • 需要快速搭建 Agent —— 可以考虑 @langchain/deep-agents，它是对底层能力的高级封装

在本教程中，我们会逐步引入各个组件，不会一开始就堆砌所有依赖。就像学做菜一样，先把基础的炒鸡蛋学会，再挑战满汉全席。

🔧 环境搭建

好了，理论知识铺垫够了，是时候动手了！接下来我们一步一步搭建开发环境。

前置条件

在开始之前，请确保你的电脑上已经安装了以下工具：

工具 版本要求 说明
Node.js 20.0 或更高 JavaScript 运行时环境
npm 随 Node.js 自带 包管理器
TypeScript 5.0 或更高 我们将通过 npm 安装
代码编辑器 - 推荐 VS Code

你可以在终端中输入以下命令来检查 Node.js 版本：

Bash 
# 检查 Node.js 版本（需要 20.0 或更高）
node --version

# 检查 npm 版本
npm --version
v20.11.0 10.2.4

如果你还没有安装 Node.js，建议前往 nodejs.org 下载 LTS（长期支持）版本。安装时会自动包含 npm。

第一步：创建项目

打开终端，执行以下命令来创建一个新的项目目录并初始化 npm 项目：

Bash 
# 创建项目目录
mkdir my-langchain-app

# 进入项目目录
cd my-langchain-app

# 初始化 npm 项目（-y 表示使用默认配置）
npm init -y

执行完这些命令后，你会看到目录下生成了一个 package.json 文件。这就是你项目的「身份证」，记录了项目名称、版本和依赖信息。

第二步：安装依赖包

接下来，安装 LangChain 相关的核心包和其他必要的依赖：

Bash 
# 安装 LangChain 核心包 + OpenAI 集成 + Zod 类型验证 + TypeScript
npm install langchain @langchain/core @langchain/openai zod typescript

这条命令安装了五个包，每个都有明确的用途：

  • langchain —— 主框架，提供 Agent 创建等高级功能
  • @langchain/core —— 核心原语，定义消息、工具等基础类型
  • @langchain/openai —— OpenAI 模型连接器，支持 GPT-4 等模型
  • zod —— TypeScript 优先的类型验证库，用于定义工具参数和结构化输出
  • typescript —— TypeScript 编译器，让我们能用 TypeScript 写代码

第三步：配置 TypeScript

在项目根目录下创建 tsconfig.json 文件，这是 TypeScript 的配置文件：

JSON 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "esModuleInterop": true,
    "strict": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}

这些配置项的含义是：

  • target: "ES2022" —— 编译目标为 ES2022，支持最新的 JavaScript 特性
  • module: "NodeNext" —— 使用 Node.js 原生的模块系统
  • strict: true —— 启用严格的类型检查，这是好事，能帮你提前发现 bug
  • outDir: "./dist" —— 编译后的 JavaScript 文件输出到 dist 目录
  • rootDir: "./src" —— 源代码放在 src 目录下

第四步：获取 API 密钥

要使用 OpenAI 的模型（如 GPT-4），你需要一个 API 密钥。获取步骤如下：

  1. 访问 platform.openai.com 并注册账号
  2. 在控制台中导航到 API Keys 页面
  3. 点击「Create new secret key」生成一个新的密钥
  4. 立即复制并安全保存这个密钥，因为它只会显示一次
⚠️ 安全提醒

API 密钥相当于你的密码，绝对不能做以下事情：

  • 不要将密钥硬编码在源代码中
  • 不要将密钥提交到 Git 仓库
  • 不要在公开场合分享密钥

正确做法是使用环境变量来管理密钥。

第五步：设置环境变量

在项目根目录下创建一个 .env 文件来存储 API 密钥：

Bash 
# 创建 .env 文件
echo "OPENAI_API_KEY=sk-你的密钥放这里" > .env

然后安装 dotenv 包来加载环境变量：

Bash 
# 安装 dotenv 用于加载环境变量
npm install dotenv

# 同时安装 ts-node 以便直接运行 TypeScript 文件
npm install -D ts-node @types/node

别忘了在项目根目录创建 .gitignore 文件，将敏感文件排除在版本控制之外：

Bash 
# 创建 .gitignore
echo "node_modules/" > .gitignore
echo "dist/" >> .gitignore
echo ".env" >> .gitignore

至此，环境搭建完成！你的项目已经准备好迎接第一个 LangChain 程序了。

🚀 第一个 LangChain 程序

环境搭好了，让我们来写第一个程序。我们将从两个示例开始：先做一个简单的模型调用，然后构建你的第一个 Agent。

示例 1：简单模型调用

首先，创建 src 目录和我们的第一个文件：

Bash 
# 创建源代码目录
mkdir src

然后创建 src/01-simple-chat.ts 文件：

TypeScript 
// src/01-simple-chat.ts
// 第一个 LangChain 程序：简单的模型调用

// 加载环境变量（必须在其他导入之前）
import "dotenv/config";

// 从 langchain 导入 initChatModel 工厂函数
import { initChatModel } from "langchain/chat_models/universal";

async function main() {
  // 使用 initChatModel 创建模型实例
  // 第一个参数是模型名称，第二个参数是提供商名称
  const model = await initChatModel("gpt-4o-mini", "openai");

  // 调用模型，发送一条简单的消息
  const response = await model.invoke("用一句话介绍 LangChain 是什么？");

  // 输出模型的回复
  console.log("模型回复：");
  console.log(response.content);
}

// 运行主函数
main().catch(console.error);

运行这个程序：

Bash 
npx ts-node src/01-simple-chat.ts
模型回复： LangChain 是一个开源框架，用于构建由大语言模型驱动的应用程序，提供了将 AI 模型与外部工具和数据源集成的标准化接口。

恭喜！你刚刚完成了第一个 LangChain 程序。虽然这个例子很简单，但它展示了 LangChain 最基本的能力：连接大语言模型并获取回复。

💡 关于 initChatModel

initChatModel 是一个工厂函数，它根据你传入的模型名称和提供商，自动创建对应的模型实例。这种设计让你可以在不修改其他代码的情况下切换模型提供商。比如要把 OpenAI 换成 Anthropic，只需要改一行代码：

TypeScript 
// 切换到 Anthropic 的 Claude 模型
// 只需要安装 @langchain/anthropic 然后改这里即可
const model = await initChatModel("claude-sonnet-4-20250514", "anthropic");

示例 2：你的第一个 Agent

接下来，让我们构建一个真正有用的 Agent —— 一个可以查询天气的智能助手。这个例子展示了 Agent 的核心能力：根据用户的问题，自主决定是否需要调用工具来获取信息。

创建 src/02-first-agent.ts 文件：

TypeScript 
// src/02-first-agent.ts
// 第一个 Agent：天气查询助手

import "dotenv/config";
import { ChatOpenAI } from "@langchain/openai";
import { createAgent } from "langchain/agents";
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// 定义一个天气查询工具
// 这里使用模拟数据，实际项目中可以调用真实的天气 API
const getWeather = tool(
  // 工具的执行函数：接收参数，返回结果
  async ({ city }: { city: string }) => {
    // 模拟天气数据
    const weatherData: Record<string, string> = {
      "北京": "晴天，气温 25°C，空气质量良好",
      "上海": "多云，气温 28°C，湿度 65%",
      "广州": "阵雨，气温 30°C，注意带伞",
      "深圳": "阴天，气温 29°C，微风",
    };
    return weatherData[city] || `抱歉，暂时没有 ${city} 的天气数据`;
  },
  {
    // 工具的名称
    name: "get_weather",
    // 工具的描述：Agent 根据这段描述决定何时使用这个工具
    description: "查询指定城市的当前天气情况。当你需要回答关于天气的问题时使用此工具。",
    // 使用 Zod 定义参数的类型和验证规则
    schema: z.object({
      city: z.string().describe("要查询天气的城市名称"),
    }),
  }
);

async function main() {
  // 创建模型实例
  const model = new ChatOpenAI({
    modelName: "gpt-4o-mini",
    temperature: 0, // 使用 0 让输出更确定
  });

  // 使用 createAgent 创建 Agent
  // Agent 会自动理解工具的描述，并在需要时调用它们
  const agent = await createAgent({
    llm: model,
    tools: [getWeather],
  });

  // 向 Agent 提问
  console.log("用户提问：北京今天天气怎么样？\n");

  const result = await agent.invoke({
    messages: [{ role: "user", content: "北京今天天气怎么样？" }],
  });

  // 输出 Agent 的最终回复
  // result.messages 中包含完整的对话历史（包括工具调用记录）
  const lastMessage = result.messages[result.messages.length - 1];
  console.log("Agent 回复：");
  console.log(lastMessage.content);
}

main().catch(console.error);

运行这个 Agent：

Bash 
npx ts-node src/02-first-agent.ts
用户提问：北京今天天气怎么样？ Agent 回复： 北京今天天气很好！现在是晴天，气温 25°C，空气质量良好。是个适合外出的好天气！

让我们拆解一下这个程序中发生了什么：

  1. 用户提问 —— 「北京今天天气怎么样？」
  2. Agent 思考 —— 模型看到有一个 get_weather 工具，描述说「查询天气时使用」
  3. 调用工具 —— Agent 决定调用 get_weather("北京")
  4. 获取结果 —— 工具返回「晴天，气温 25°C，空气质量良好」
  5. 生成回复 —— Agent 将工具返回的数据组织成自然语言回复给用户

这就是 Agent 的魅力所在：它能自主判断什么时候需要「动手」（调用工具），什么时候直接「动嘴」（用自身知识回答）。你不需要写 if-else 来决定何时调用哪个工具，模型会根据上下文自动做出最优决策。

📌 工具描述的重要性

注意 getWeather 工具的 description 字段：「查询指定城市的当前天气情况。当你需要回答关于天气的问题时使用此工具。」这段描述不是给人看的，而是给模型看的。模型根据工具的名称和描述来决定在什么场景下使用哪个工具。因此，写好工具描述是构建高质量 Agent 的关键技巧之一。我们会在第 5 章详细讨论这个话题。

📁 项目结构最佳实践

随着项目的增长，良好的项目结构能帮你保持代码的整洁和可维护性。以下是一个推荐的项目目录结构：

Bash 
my-langchain-app/
├── src/
│   ├── agents/          # Agent 定义和配置
│   │   └── weather-agent.ts
│   ├── tools/           # 自定义工具
│   │   └── weather.ts
│   ├── models/          # 模型配置和工厂
│   │   └── index.ts
│   ├── prompts/         # 提示词模板
│   │   └── templates.ts
│   ├── utils/           # 工具函数
│   │   └── helpers.ts
│   ├── types/           # TypeScript 类型定义
│   │   └── index.ts
│   └── index.ts         # 程序入口
├── tests/               # 测试文件
│   └── agent.test.ts
├── .env                 # 环境变量（不提交到 Git）
├── .gitignore           # Git 忽略规则
├── package.json         # 项目配置
└── tsconfig.json        # TypeScript 配置

这个结构的核心思想是按职责分离：

  • src/agents/ —— 存放 Agent 的定义。每个 Agent 是一个独立的模块，方便复用和测试
  • src/tools/ —— 存放自定义工具。将工具独立出来，方便多个 Agent 共享同一套工具
  • src/models/ —— 集中管理模型的创建和配置。如果将来需要切换模型提供商，只需修改这一个地方
  • src/prompts/ —— 存放提示词模板。好的提示词是 Agent 表现优秀的关键，值得单独管理
  • src/utils/ —— 通用工具函数，如日志、错误处理等
  • src/types/ —— 自定义的 TypeScript 类型定义
💡 小贴士：从简单开始

对于本教程的前几个章节，你不需要一开始就创建这么完整的目录结构。先从简单的单文件开始（就像前面的示例 1 和示例 2 那样），等到代码量增长到你觉得混乱时，再按照这个结构进行重构。过早优化是万恶之源。

在 package.json 中，你可以添加一些常用的脚本来简化开发流程：

JSON 
{
  "scripts": {
    "start": "ts-node src/index.ts",
    "agent": "ts-node src/02-first-agent.ts",
    "build": "tsc",
    "test": "jest",
    "lint": "eslint src/"
  }
}

这样你就可以用 npm start 来运行程序，用 npm run build 来编译 TypeScript，用 npm test 来运行测试。比起每次输入长长的 npx ts-node ... 命令，这样方便多了。

🧠 知识检查点
Q1: LangChain 的核心理念「Agent = Model + Harness」中，Harness 的作用是什么？
A: Harness（控制框架）是 LangChain 提供的「外壳」，它负责管理对话流程、解析模型输出为工具调用指令、处理工具调用的错误和重试、以及维护多轮对话的状态。简而言之，Harness 把一个「只会说话」的模型变成了一个「能做事情」的 Agent。模型负责思考和表达，Harness 负责协调和执行。
Q2: 在 LangChain 生态系统中，@langchain/core 和 langchain 这两个包有什么区别？
A: @langchain/core 是最底层的核心包，定义了所有基础原语（如消息、工具、输出解析器等）的接口和类型。langchain 则是在 Core 之上的高级包，提供了 Agent 创建（createAgent）、链（Chain）等高级功能。可以理解为：@langchain/core 是地基，langchain 是建在地基上的房子。模型集成包（如 @langchain/openai）依赖 Core 而不依赖 langchain。
Q3: 为什么 API 密钥不能硬编码在源代码中？正确的做法是什么？
A: API 密钥如果硬编码在源代码中，一旦代码被提交到 Git 仓库（尤其是公开仓库），密钥就会泄露。恶意用户可以利用你的密钥产生大量费用或滥用你的账户。正确做法是将密钥存储在 .env 环境变量文件中，使用 dotenv 包在运行时加载，并确保 .env 文件在 .gitignore 中被排除。
Q4: 在 Agent 的工具定义中，description 字段为什么如此重要？如果描述写得不好会怎样？
A: description 字段是 Agent 决定「何时使用哪个工具」的唯一依据。模型会根据工具的名称和描述来判断当前场景下是否需要调用该工具。如果描述写得模糊或不准确，Agent 可能会在不该调用工具时调用（浪费时间），或者在应该调用时遗漏（回答不准确）。好的描述应该清晰说明工具的功能和适用场景。

🐛 常见错误

在搭建环境和运行第一个程序时，新手经常会遇到一些问题。这里列出最常见的几个，帮你少走弯路：

错误 1：API Key 未设置

Bash 
Error: OpenAI API key not found. Please set the OPENAI_API_KEY environment variable.

原因：没有正确设置 OPENAI_API_KEY 环境变量。

解决方案：

  1. 确认项目根目录下有 .env 文件，并且其中包含 OPENAI_API_KEY=sk-xxx
  2. 确认在代码最顶部（其他 import 之前）执行了 import "dotenv/config"
  3. 确认已安装 dotenv：npm install dotenv
  4. 确认密钥值前后没有多余的空格或引号

错误 2：模块找不到

Bash 
Error: Cannot find module '@langchain/openai'
Require stack:
- /path/to/your/project/src/index.ts

原因：忘记安装必要的 npm 包，或者包名拼写错误。

解决方案：

  1. 运行 npm install langchain @langchain/core @langchain/openai
  2. 检查 import 语句中的包名是否拼写正确（注意 @ 前缀和 / 分隔符）
  3. 确认 node_modules 目录存在（如果缺失，运行 npm install）

错误 3：Node.js 版本过低

Bash 
SyntaxError: Unexpected token '{'

原因：Node.js 版本低于 20，不支持某些现代 JavaScript 语法。

解决方案：

  1. 运行 node --version 检查当前版本
  2. 如果版本低于 20，请前往 nodejs.org 下载并安装最新 LTS 版本
  3. 如果你使用 nvm 管理 Node.js 版本，运行 nvm install 20 && nvm use 20

错误 4：TypeScript 编译配置问题

Bash 
TS1479: The current file is a CommonJS module whose imports will produce a require call...
Consider updating to the newer "module" setting.

原因：TypeScript 的模块系统配置与 Node.js 不匹配。

解决方案：确保 tsconfig.json 中设置了 "module": "NodeNext" 和 "moduleResolution": "NodeNext"。如果 package.json 中没有 "type": "module"，还需要将 TypeScript 文件的扩展名设为 .mts，或者在 package.json 中添加 "type": "module"。
