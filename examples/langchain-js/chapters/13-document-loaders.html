📄 Document Loaders & Text Splitters — 文档加载与智能分割

加载各种格式的文档，并将其智能分割成适合 LLM 处理的片段 进阶

🎯 本章目标

学完本章，你将能够：
  • 理解 Document 对象的结构（pageContent + metadata）及其作用
  • 掌握 TextLoader、PDFLoader、CheerioWebBaseLoader、JSONLoader、CSVLoader 等核心加载器
  • 使用 RecursiveCharacterTextSplitter 智能分割长文本
  • 理解 TokenTextSplitter 和 CharacterTextSplitter 的区别
  • 掌握 chunkSize 和 chunkOverlap 的调优策略

📋 前置知识

⚠️ 前置要求：请确保你已完成 第 11 章：Prompt Templates 的学习， 了解模板格式化的基本概念。同时熟悉 TypeScript 的异步编程（async/await）。 本章是 RAG（检索增强生成）专题的第一篇，后续章节将在此基础上构建完整的 RAG 系统。

💡 核心概念

概念一：Document 对象 — 文档的"护照"

在 LangChain 中，所有文档加载器返回的数据都是 Document 对象。 Document 是一个极其简单的数据结构，它只有两个字段：

字段类型说明
pageContentstring文档的文本内容
metadataRecord<string, any>文档的元数据（来源、页码、标题等）

你可以把 Document 想象成一本护照：pageContent 是你的照片和基本信息， 而 metadata 是签证页和出入境记录——记录了文档的"身份"和"来源"。 正是 metadata 的存在，使得 RAG 系统能够追踪每个片段来自哪个文件、哪一页， 从而在回答问题时引用准确的来源。

TypeScript 
import { Document } from "@langchain/core/documents";

// 创建一个 Document 对象
const doc = new Document({
  pageContent: "LangChain 是一个用于构建 LLM 应用的框架。",
  metadata: {
    source: "intro.md",
    author: "LangChain Team",
    page: 1,
    created: "2024-01-15",
  },
});

console.log(doc.pageContent); // "LangChain 是一个用于构建 LLM 应用的框架。"
console.log(doc.metadata.source); // "intro.md"

// Document 对象也可以直接序列化为 JSON
console.log(JSON.stringify(doc, null, 2));
/* {
 *   "pageContent": "LangChain 是一个用于构建 LLM 应用的框架。",
 *   "metadata": { "source": "intro.md", "author": "LangChain Team", ... }
 * } */
💡 为什么 metadata 如此重要？

当 RAG 系统从向量数据库中检索到某个文档片段时，metadata 告诉你这个片段来自哪里。比如用户问"去年的营收是多少"，系统检索到了对应的片段，通过 metadata 中的 source 和 page 字段，最终回答可以附上引用："根据 2023 年度财报第 12 页……"。没有 metadata，你就无法做来源追溯。

概念二：文档加载器 — 从各种来源读取数据

LangChain 提供了丰富的文档加载器，每种加载器对应一种数据源。 加载器的职责只有一个：读取原始数据 → 转为 Document 对象数组。

加载器来源安装命令
TextLoader纯文本 / Markdown 文件内置在 @langchain/community
PDFLoaderPDF 文档（逐页加载）npm install pdf-parse
CheerioWebBaseLoader网页 HTML 内容npm install cheerio
JSONLoaderJSON / JSONL 文件内置
CSVLoaderCSV 表格文件内置
TypeScript 
// ==========================================
// TextLoader — 加载纯文本/Markdown 文件
// ==========================================
import { TextLoader } from "@langchain/community/document_loaders/fs/text";

const textLoader = new TextLoader("./data/readme.md");
const docs = await textLoader.load();
// docs[0].pageContent  → 文件的完整文本内容
// docs[0].metadata     → { source: "./data/readme.md" }

// ==========================================
// PDFLoader — 加载 PDF（逐页拆分）
// ==========================================
import { PDFLoader } from "@langchain/community/document_loaders/fs/pdf";

const pdfLoader = new PDFLoader("./data/report.pdf");
const pdfDocs = await pdfLoader.load();
// pdfDocs.length  → PDF 页数
// pdfDocs[0].metadata → { source: "./data/report.pdf", page: 1, ... }

// ==========================================
// CheerioWebBaseLoader — 加载网页
// ==========================================
import { CheerioWebBaseLoader } from "@langchain/community/document_loaders/web/cheerio";

const webLoader = new CheerioWebBaseLoader("https://js.langchain.com/docs/introduction/");
const webDocs = await webLoader.load();
// webDocs[0].pageContent → 网页的文本内容（去除了 HTML 标签）

// ==========================================
// JSONLoader — 加载 JSON 文件
// ==========================================
import { JSONLoader } from "@langchain/community/document_loaders/fs/json";

const jsonLoader = new JSONLoader("./data/data.json");
const jsonDocs = await jsonLoader.load();
// 每个 JSON 对象中的字符串字段会被提取为独立的 Document

// ==========================================
// CSVLoader — 加载 CSV 文件
// ==========================================
import { CSVLoader } from "@langchain/community/document_loaders/fs/csv";

const csvLoader = new CSVLoader("./data/products.csv");
const csvDocs = await csvLoader.load();
// 每一行 CSV 记录变为一个 Document
💡 为什么需要不同的加载器？

不同文件格式的内部结构天差地别——PDF 要解析页面布局和文本流，网页要去除 HTML 标签提取正文，JSON 要处理嵌套结构。LangChain 把这种复杂性封装在各自的加载器中，你只需要调用 loader.load()，就能获得统一的 Document 数组，后续的向量化、检索流程完全不用关心原始格式。

概念三：文本分割器 — 将长文档切成"一口能吃下"的片段

LLM 有上下文窗口限制（如 GPT-4 的 128K token），但并不意味着你可以直接把一本 500 页的书塞进去。 大多数场景下，RAG 系统需要将文档分割成较小的块（chunk），每个块包含足够的信息但又不会浪费上下文空间。 这就是 TextSplitter 的职责。

分割器分割依据适用场景
RecursiveCharacterTextSplitter递归地按分隔符列表分割通用文档（推荐首选）
CharacterTextSplitter按固定字符分割简单文本、日志文件
TokenTextSplitter按 token 数量分割需要精确控制 token 用量时
TypeScript 
import { RecursiveCharacterTextSplitter } from "@langchain/textsplitters";
import { TokenTextSplitter } from "@langchain/textsplitters";
import { CharacterTextSplitter } from "@langchain/textsplitters";

// ==========================================
// RecursiveCharacterTextSplitter（推荐）
// ==========================================
// 按优先级递归分割：\n\n → \n → 句号 → 感叹号 → 空格 → 字符
const recursiveSplitter = new RecursiveCharacterTextSplitter({
  chunkSize: 500,      // 每块约 500 字符
  chunkOverlap: 50,    // 相邻块重叠 50 字符
  separators: ["\n\n", "\n", "。", ".", "！", "？", " "],
});

const chunks = await recursiveSplitter.splitDocuments(docs);
// chunks[0].pageContent → 第一块内容
// chunks[0].metadata    → 保留了原始文档的 metadata

// ==========================================
// TokenTextSplitter — 按 token 分割
// ==========================================
const tokenSplitter = new TokenTextSplitter({
  chunkSize: 1000,     // 每块约 1000 token
  chunkOverlap: 100,   // 重叠 100 token
  encodingName: "cl100k_base", // OpenAI 使用的编码器
});

const tokenChunks = await tokenSplitter.splitDocuments(docs);

// ==========================================
// CharacterTextSplitter — 按字符分割
// ==========================================
const charSplitter = new CharacterTextSplitter({
  separator: "\n\n",   // 按段落分割
  chunkSize: 300,      // 每块约 300 字符
  chunkOverlap: 30,
});

const charChunks = await charSplitter.splitDocuments(docs);

概念四：chunkSize 和 chunkOverlap 的调优策略

chunkSize 和 chunkOverlap 是分割器的两个核心参数， 它们的取值直接影响 RAG 系统的检索效果和 LLM 的回答质量。

参数详解与调优建议

参数含义调优建议
chunkSize每个 chunk 的目标大小500-1000 字符（约 100-200 token）适合问答场景；1500-3000 字符适合摘要/分析场景
chunkOverlap相邻 chunk 之间的重叠字符数通常为 chunkSize 的 10%-20%。例如 chunkSize=1000，则 overlap=100~200

调优策略速查

  • 小 chunk（~500 字符）：适合精确检索，每个 chunk 聚焦一个点，但可能缺乏上下文
  • 大 chunk（~2000 字符）：保留更多上下文，但可能包含无关信息，降低检索精度
  • 重叠不足：关键信息可能恰好被分割边界切断，导致上下不连贯
  • 重叠过多：产生大量冗余数据，增加向量存储和 LLM 推理的成本
TypeScript 
// chunkSize / chunkOverlap 的直观演示
const sampleText = `
LangChain 是一个强大的 LLM 应用框架。
它提供了丰富的工具和抽象。
开发者可以用它构建复杂的 AI 应用。
其中，文档加载器和文本分割器是 RAG 的基础组件。

RAG（检索增强生成）通过检索相关文档来增强 LLM 的回答。
这个模式广泛应用于问答系统、客服机器人等场景。
`;

const splitter = new RecursiveCharacterTextSplitter({
  chunkSize: 50,
  chunkOverlap: 10,
  separators: ["\n\n", "\n", "。", "！", "？"],
});

const result = await splitter.createDocuments([sampleText]);
console.log(`分割成 ${result.length} 块`);
result.forEach((doc, i) => {
  console.log(`--- 块 ${i + 1} (${doc.pageContent.length} 字符) ---`);
  console.log(doc.pageContent);
});
/* 输出示例：
 * --- 块 1 (49 字符) ---
 * LangChain 是一个强大的 LLM 应用框架。
 * 它提供了丰富的工具和抽象。
 * --- 块 2 (46 字符) ---
 * 它提供了丰富的工具和抽象。
 * 开发者可以用它构建复杂的 AI 应用。
 * ...
 */
💡 为什么需要 chunkOverlap？

想象你在一本书中查找"特朗普的贸易政策"。如果分割边界恰好把"特朗普的贸易政策对环境的影响"从中间切断，第一块以"特朗普的贸易政策"结尾，第二块以"对环境的影响"开头——那检索"贸易政策"时只能找到第一块，但它缺少"对环境的影响"这一关键信息。chunkOverlap 确保边界处的信息在相邻块中都有出现，避免了这种"信息断裂"。

⚡ 进阶技巧

技巧一：针对不同文档类型选择分割策略

不同的文档类型有不同的结构特点，选择合适的分割策略可以显著提升 RAG 的检索效果。

文档类型推荐分割器关键 separator说明
MarkdownRecursiveCharacterTextSplitter\n## 、\n### 按标题层级分割，保留语义结构
代码文件RecursiveCharacterTextSplitter\n\n、\n按函数/类定义分割，避免切断代码语句
PDF 报告PDFLoader + TokenTextSplitter按 token 分割PDF 页面已有自然边界，用 token 控制每块大小
JSON 数据JSONLoader + RecursiveCharacterTextSplitter\n先将 JSON 转为文本，再用 separator 分割
CSV 表格CSVLoader每行一条记录CSVLoader 本身已按行分割，无需额外分割
长篇文学作品TokenTextSplitter按 token 分割文学作品没有固定结构，用 token 确保每块大小一致
TypeScript 
// ==========================================
// Markdown 分割策略：保留标题层级
// ==========================================
const mdSplitter = new RecursiveCharacterTextSplitter({
  chunkSize: 400,
  chunkOverlap: 40,
  separators: ["\n## ", "\n### ", "\n#### ", "\n\n", "\n", " "],
});

// ==========================================
// 代码文件分割策略：按函数/类分割
// ==========================================
const codeSplitter = new RecursiveCharacterTextSplitter({
  chunkSize: 600,
  chunkOverlap: 80,
  separators: ["\n\n", "\n", ";", "}", "{"],
});

// ==========================================
// 学术论文分割策略：按段落 + token 控制
// ==========================================
import { TokenTextSplitter } from "@langchain/textsplitters";

const paperSplitter = new TokenTextSplitter({
  chunkSize: 512,     // 512 tokens 约等于 1 个段落
  chunkOverlap: 64,   // 10% 重叠
  encodingName: "cl100k_base",
});

// ==========================================
// 自定义 metadata 增强
// ==========================================
// 分割后可以给每个 chunk 添加额外 metadata
const rawDocs = await loader.load();
const splitDocs = await splitter.splitDocuments(rawDocs);

const enrichedDocs = splitDocs.map((doc, index) => ({
  ...doc,
  metadata: {
    ...doc.metadata,
    chunkIndex: index,           // 片段序号
    totalChunks: splitDocs.length, // 总片段数
    charCount: doc.pageContent.length, // 字符数
  },
}));
// 这样在检索时可以根据 chunkIndex 重建文档的原始顺序

技巧二：增量加载与流式处理

对于大文件（如 GB 级别的日志或数据集），一次性加载到内存可能不现实。 LangChain 的某些加载器和分割器支持增量处理。

TypeScript 
// ==========================================
// 使用 lazyLoad 增量加载大文件
// ==========================================
// 某些加载器支持 lazyLoad 方法，返回 AsyncIterable
// 适用于大文件的流式处理

import { TextLoader } from "@langchain/community/document_loaders/fs/text";

const loader = new TextLoader("./large-file.txt");

// 逐块处理，避免内存溢出
for await (const batch of loader.lazyLoad()) {
  // batch 是一批 Document 对象
  const chunks = await splitter.splitDocuments(batch);
  // 将 chunks 存入向量数据库或做其他处理
  console.log(`处理了 ${chunks.length} 个片段`);
}

技巧三：Metadata 的传递与扩增

分割后的 chunks 会自动继承原始文档的 metadata。但你也可以利用这一点， 在加载阶段就为文档附加丰富的元信息。

TypeScript 
// ==========================================
// 加载时附加自定义 metadata
// ==========================================
import { PDFLoader } from "@langchain/community/document_loaders/fs/pdf";

const loader = new PDFLoader("./report-2024.pdf", {
  // 自定义 metadata 会在每个页面中保留
  // 也可以通过 splitDocuments 后的 map 操作添加
});

const rawDocs = await loader.load();

// 为每页文档添加业务相关的 metadata
const enrichedDocs = rawDocs.map((doc, index) => {
  return new Document({
    pageContent: doc.pageContent,
    metadata: {
      ...doc.metadata,
      // 业务字段
      year: 2024,
      department: "engineering",
      documentType: "annual_report",
      // 内容特征
      hasCharts: doc.pageContent.includes("chart"),
      wordCount: doc.pageContent.split(/\s+/).length,
    },
  });
});

// 分割时 metadata 会自动传递到每个 chunk
const chunks = await splitter.splitDocuments(enrichedDocs);
// chunks[0].metadata 包含原始的所有字段 + chunkIndex 等

🧠 知识检查点

📝 自测题：检查你对本章内容的理解
1. Document 对象的两个核心字段是什么？为什么 metadata 在 RAG 系统中非常重要？

两个核心字段是 pageContent（文档文本内容）和 metadata（文档元数据）。metadata 在 RAG 系统中非常重要，因为它记录了文档的来源（如文件名、页码、URL 等），使得系统在检索到某个片段后能够追溯到原始出处，在回答问题时附上可靠的引用信息。

2. RecursiveCharacterTextSplitter 是如何决定在哪里分割文本的？

它按照传入的 separators 数组优先级依次尝试分割。默认顺序是 ["\n\n", "\n", " ", ""]，即先尝试按段落分割，不行再按行分割，再不行按空格，最后按字符。这种递归策略确保尽可能在语义自然的位置分割，尽量避免在句子中间切断。

3. chunkSize 和 chunkOverlap 的典型取值范围是多少？如果 chunkOverlap 设置为 0 会有什么风险？

chunkSize 的典型范围是 500-2000 字符（约 100-500 token），具体取决于应用场景。chunkOverlap 通常设为 chunkSize 的 10%-20%。如果 chunkOverlap 为 0，则分割边界处的信息可能会被丢失——例如一个关键术语恰好位于第一块的末尾和第二块的开头，两部分都缺少完整上下文，导致检索时都无法找到这个术语。

4. 如果你想从网页中提取内容，但只想提取某个特定 <div> 区域的内容，应该用什么加载器？如何配置？

使用 CheerioWebBaseLoader，在构造函数中传入 selector 选项来指定 CSS 选择器。例如：new CheerioWebBaseLoader(url, { selector: "main" }) 只提取 <main> 标签内的内容。如果不指定 selector，则默认提取整个 <body> 的文本。

🐛 常见错误

❌ 错误 1：忘记安装必需的依赖

症状：使用 PDFLoader 时报错 Cannot find module 'pdf-parse'，或使用 CheerioWebBaseLoader 时报 Cannot find module 'cheerio'。

原因：某些加载器需要额外的第三方包来处理文件格式。

解决：根据加载器文档安装对应的依赖包：npm install pdf-parse cheerio。TextLoader、JSONLoader、CSVLoader 不需要额外依赖。

❌ 错误 2：chunkSize 设置过大导致 LLM 溢出

症状：LLM 调用时报错 context length exceeded，或者回答质量急剧下降。

原因：chunkSize 设置过大（如 10000 字符），导致检索到的多个 chunk 拼接后超过了 LLM 的上下文窗口。

解决：根据目标 LLM 的上下文限制设置合理的 chunkSize。对于 GPT-4，建议单个 chunk 不超过 2000 token（约 8000 字符）；并控制检索时返回的 chunk 数量。

❌ 错误 3：分割后丢失了文档结构信息

症状：检索到的 chunk 内容不完整，例如代码块被从中间截断，或者表格数据被分开。

原因：使用的 separator 列表没有考虑到文档的特殊结构，分割器在代码块或表格内部进行了切割。

解决：为特定类型的文档自定义 separators 列表。例如 Markdown 代码块用 \n``` 作为分隔符，表格用 \n|。或者考虑使用更保守的 chunkSize 和更大的 overlap。

❌ 错误 4：网页加载器因反爬机制返回空内容

症状：CheerioWebBaseLoader 加载网页成功，但 pageContent 为空或包含"请验证你不是机器人"等提示。

原因：目标网站有反爬虫机制（如 Cloudflare、CAPTCHA）或需要 JavaScript 渲染。

解决：对于需要 JS 渲染的 SPA 网站，考虑使用 PuppeteerWebBaseLoader 或 PlaywrightWebBaseLoader。对于反爬虫，可以设置 User-Agent 请求头或使用代理。
