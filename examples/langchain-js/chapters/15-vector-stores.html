🗄️ Vector Stores & Retrievers — 向量存储与智能检索

将文本语义向量化存储，并通过相似度、MMR 等多种策略精准检索

🎯 本章目标

学完本章，你将能够：
  • 理解向量存储（VectorStore）的核心接口：fromDocuments、addDocuments、similaritySearch
  • 掌握 MemoryVectorStore（内存向量库）的完整用法
  • 了解 Chroma、PineconeStore 等生产级向量数据库的基本接入方式
  • 掌握 Retriever 接口及 asRetriever()、k 参数、scoreThreshold 的配置
  • 理解 MultiQueryRetriever 和 MMR（最大边际相关性）检索的差异
  • 学会使用元数据过滤（Metadata Filtering）来精确定位文档

📋 前置知识

⚠️ 前置要求：请确保你已完成 第 14 章：Embedding 向量化 的学习， 理解文本嵌入（Embedding）的基本概念和 OpenAI Embeddings 的用法。 同时建议复习 第 13 章 中 Document 对象的 结构（pageContent + metadata）以及文本分割的基本方法。

💡 核心概念

概念一：VectorStore 接口——向量数据库的统一抽象

VectorStore（向量存储）是 LangChain 对各种向量数据库的抽象接口。 你可以把它想象成一个"超强语义搜索引擎"——传统搜索引擎靠关键词匹配， 而向量搜索引擎靠语义相似度匹配。当你搜索"人工智能的伦理问题"时， 即使文档中从未出现这几个字，但如果有文档讨论了"AI 的道德困境"，它也会被找到。

在 LangChain.js 中，所有向量存储都实现了 VectorStore 接口， 它定义了一组标准的操作方法：

方法用途说明
fromDocuments(docs, embeddings, config?)从文档数组创建向量库并自动索引最常用的初始化方式（类静态方法）
addDocuments(docs)向已有向量库添加新文档增量更新时使用
deleteDocuments(ids?)删除指定 ID 的文档部分向量库支持
similaritySearch(query, k, filter?)按语义相似度检索最匹配的 k 个文档核心查询方法
similaritySearchVector(queryVector, k, filter?)直接用向量进行相似度检索进阶用法，跳过文本嵌入步骤
asRetriever(k, filter?)将向量库包装为 Retriever 接口用于 LCEL 链式调用
TypeScript 
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { OpenAIEmbeddings } from "@langchain/openai";
import { Document } from "@langchain/core/documents";

// ==========================================
// 从文档列表创建内存向量存储
// ==========================================
const embeddings = new OpenAIEmbeddings({
  model: "text-embedding-3-small",
});

const docs = [
  new Document({
    pageContent: "LangChain 是一个用于构建 LLM 应用的开源框架",
    metadata: { topic: "langchain", language: "zh" },
  }),
  new Document({
    pageContent: "向量数据库将文本语义转换为高维向量进行存储",
    metadata: { topic: "vector-db", language: "zh" },
  }),
];

const vectorStore = await MemoryVectorStore.fromDocuments(docs, embeddings);

// ==========================================
// 相似度搜索
// ==========================================
const results = await vectorStore.similaritySearch("什么是向量数据库", 2);

console.log(results);
// [
//   Document { pageContent: "向量数据库将文本语义转换为高维向量进行存储", metadata: {...} },
//   Document { pageContent: "LangChain 是一个用于构建 LLM 应用的开源框架", metadata: {...} }
// ]
💡 工作原理

当你调用 fromDocuments() 时，向量库内部会调用 Embeddings 模型将每篇文档的文本转为向量（embedding vector）， 然后将这些向量连同原始文本和元数据一起存储。当查询时，similaritySearch() 会将查询文本也转为向量， 然后计算查询向量与所有存储向量的余弦相似度（cosine similarity），返回最匹配的 k 个结果。

概念二：MemoryVectorStore — 轻量级内存向量库

MemoryVectorStore 是 LangChain.js 内置的内存级向量存储实现。 它在应用程序的内存中用数组保存所有向量，适合以下场景：

  • 学习和原型开发——不需要安装任何外部数据库
  • 小规模应用——文档总数量在几千以内
  • 测试和 CI 环境——无需额外的基础设施依赖
  • 无状态的 Serverless 函数——每次冷启动重新加载

当然，它也有局限性：所有数据存储在内存中，进程退出后数据丢失； 不适用于大规模生产环境；不支持持久化存储和分布式查询。

TypeScript 
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { OpenAIEmbeddings } from "@langchain/openai";

const embeddings = new OpenAIEmbeddings();

// ==========================================
// 方式一：fromDocuments —— 创建即填充
// ==========================================
const store1 = await MemoryVectorStore.fromDocuments(
  [
    { pageContent: "Python 是一种动态类型的编程语言", metadata: { lang: "zh" } },
    { pageContent: "TypeScript 是 JavaScript 的超集", metadata: { lang: "zh" } },
  ],
  embeddings
);

// ==========================================
// 方式二：new + addDocuments —— 分步创建
// ==========================================
const store2 = new MemoryVectorStore(embeddings);
await store2.addDocuments([
  { pageContent: "Node.js 是一个 JavaScript 运行时环境", metadata: { lang: "zh" } },
]);

概念三：生产级向量数据库——Chroma / PineconeStore

对于生产环境，你需要使用持久化的向量数据库。LangChain.js 支持多种主流方案：

向量数据库特点适合场景包名
Chroma开源、轻量、支持本地/客户端模式中小型项目，快速原型到生产的过渡langchain/vectorstores/chroma
Pinecone全托管的云原生向量数据库大规模生产应用，不需要运维@langchain/pinecone
Weaviate开源、支持混合搜索（向量+关键词）需要综合搜索能力的场景langchain/vectorstores/weaviate
Supabase (pgvector)基于 PostgreSQL 的向量扩展已有 Postgres 基础设施的团队@langchain/community
TypeScript 
// ==========================================
// Chroma 接入示例
// ==========================================
import { Chroma } from "@langchain/community/vectorstores/chroma";
import { OpenAIEmbeddings } from "@langchain/openai";

const chromaStore = await Chroma.fromDocuments(docs, embeddings, {
  collectionName: "my_collection",
});

// ==========================================
// Pinecone 接入示例
// ==========================================
import { PineconeStore } from "@langchain/pinecone";
import { Pinecone } from "@pinecone-database/pinecone";

const pinecone = new Pinecone({
  apiKey: process.env.PINECONE_API_KEY!,
});
const pineconeIndex = pinecone.Index("my-index");

const pineconeStore = await PineconeStore.fromDocuments(docs, embeddings, {
  pineconeIndex,
});
⚠️ 注意

以上代码仅为演示各数据库的接入方式。实际使用 Chroma 前需要确保 Chroma 服务正在运行（或使用内存模式）； 使用 Pinecone 则需要提前在 Pinecone 控制台创建 Index 并获取 API Key。 本章后续示例均使用 MemoryVectorStore 以便于学习。

概念四：Retriever 接口——标准化的检索抽象

Retriever（检索器）是 LangChain 中另一个核心抽象。如果说 VectorStore 关注的是 "如何存储和搜索向量"，那么 Retriever 关注的是"如何以统一的方式获取相关文档"。

Retriever 的核心优势在于它与 LCEL 管道的无缝集成——任何 Retriever 都可以直接用在 | 运算符中，成为 RAG 管线的数据源。

方法/属性说明
invoke(query)接收查询字符串，返回匹配的 Document 数组
getRelevantDocuments(query)同上（早期方法，仍可用但推荐 invoke）
asRetriever(k?, filter?)将 VectorStore 包装为 Retriever
TypeScript 
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { OpenAIEmbeddings } from "@langchain/openai";

const vectorStore = await MemoryVectorStore.fromDocuments(docs, embeddings);

// ==========================================
// 创建 Retriever
// ==========================================

// 基础检索器 —— 默认返回前 4 条
const retriever = vectorStore.asRetriever();

// 指定 k 值 —— 只返回最相关的 3 条
const retrieverK3 = vectorStore.asRetriever({
  k: 3,
});

// 带分数阈值的检索器 —— 只返回相似度 ≥ 0.8 的结果
const retrieverThreshold = vectorStore.asRetriever({
  k: 10,
  searchType: "similarity",
  searchKwargs: {
    scoreThreshold: 0.8,  // 只返回余弦相似度 ≥ 0.8 的文档
  },
});

// ==========================================
// 使用 Retriever 检索
// ==========================================
const results = await retrieverK3.invoke("什么是语义搜索？");
// 返回 Document[]，最多 3 条

// 用于 LCEL 管道
const chain = retrieverK3.pipe(someLambda);
const result = await chain.invoke("我的查询");
💡 scoreThreshold 的作用

当你设置 scoreThreshold 时，LangChain 会先按相似度检索最多 k 条结果， 然后过滤掉相似度低于阈值的文档。这在需要控制检索质量时非常有用—— 比如当数据库中没有足够相关的文档时，与其返回低质量的匹配，不如返回空结果让 LLM 自行处理。

概念五：MultiQueryRetriever — 多角度查询增强

MultiQueryRetriever 是一种高级检索策略。它的核心思想是： 用 LLM 从不同角度改写用户的原始查询，生成多个相关问法， 然后分别检索并合并结果。这样可以弥补用户单一问法可能遗漏的相关文档。

TypeScript 
import { MultiQueryRetriever } from "langchain/retrievers/multi_query";
import { ChatOpenAI } from "@langchain/openai";

// 构建一个 MultiQueryRetriever
const multiRetriever = MultiQueryRetriever.fromLLM({
  llm: new ChatOpenAI({ model: "gpt-4o-mini", temperature: 0 }),
  retriever: vectorStore.asRetriever({ k: 2 }),
  verbose: true,  // 打印生成的多个查询
});

// 用户只问了一个问题
const docs = await multiRetriever.invoke("苹果公司的总部在哪里？");

// LLM 可能会自动生成多个变体查询：
// 1. "苹果公司的总部在哪里？"
// 2. "Apple Inc. 的总部地址是什么？"
// 3. "库比蒂诺的苹果总部位置"
// 4. "苹果公司的主办公地点在哪儿？"
//
// 每个查询分别检索，去重合并后返回

适用场景：当用户的查询可能含糊不清、或者同一个概念有多种表达方式时， MultiQueryRetriever 可以显著提高召回率（Recall）。代价是增加了 LLM 调用次数和响应延迟。

概念六：MMR（最大边际相关性）检索

普通的相似度检索（similaritySearch）往往会返回非常相似的结果—— 比如搜索"深度学习"，可能会得到三篇都是关于卷积神经网络的文档，而忽略了 关于循环神经网络或生成对抗网络的内容。这就是"信息冗余"问题。

MMR（Maximum Marginal Relevance，最大边际相关性） 检索在考虑与查询的相关性的同时，也考虑结果之间的多样性。 它通过一个平衡参数 lambda（也称为 fetchK）来控制：

  • λ = 1：完全按相似度排序（退化为普通相似度搜索）
  • λ = 0：完全按多样性排序（结果尽可能不相似）
  • λ = 0.5：平衡相关性和多样性（常用值）
TypeScript 
// ==========================================
// MMR 检索（通过 Retriever 配置）
// ==========================================
const mmrRetriever = vectorStore.asRetriever({
  k: 4,                   // 最终返回 4 条结果
  searchType: "mmr",      // 使用 MMR 检索
  searchKwargs: {
    fetchK: 20,           // 先取 20 条候选
    lambda: 0.5,          // 相关性 vs 多样性的平衡参数
  },
});

const mmrDocs = await mmrRetriever.invoke("机器学习框架比较");

// ==========================================
// MMR 检索（通过 VectorStore 直接调用）
// ==========================================
const mmrResults = await vectorStore.maxMarginalRelevanceSearch(
  "机器学习框架比较",
  {
    k: 4,
    fetchK: 20,
    lambda: 0.5,
  }
);
💡 MMR 的工作流程

MMR 分两步进行： ① 首先用普通相似度搜索取 fetchK 个候选结果（比如 20 条）； ② 然后从中挑选出 k 个（比如 4 条）最终结果，在每一步都选择"与查询相关且与已选结果最不相似"的文档。 这样保证了最终结果既相关又多样化。

概念七：Metadata Filtering — 元数据过滤

在实际应用中，你的向量库中可能包含来自多个来源、多种类型、不同时间段的文档。 元数据过滤允许你在进行语义检索的同时，根据文档的元数据字段 缩小搜索范围，实现"先筛选，后检索"的效果。

不同向量数据库支持不同的过滤语法。在 MemoryVectorStore 中， 过滤通过简单的 JavaScript 对象实现（精确匹配）。对于 Chroma、Pinecone 等， 过滤语法会更复杂（支持比较运算、逻辑组合等）。

TypeScript 
// ==========================================
// MemoryVectorStore 的元数据过滤
// ==========================================

// 方式一：在 similaritySearch 时过滤
const filteredResults = await vectorStore.similaritySearch(
  "为什么天空是蓝色的？",
  5,
  { language: "zh", category: "science" }  // 只搜索中文科学类文档
);

// 方式二：在 asRetriever 时预设过滤
const filteredRetriever = vectorStore.asRetriever({
  k: 3,
  filter: { author: "张三", year: 2024 },
});

// ==========================================
// Chroma 的过滤语法（支持比较运算）
// ==========================================
import { Chroma } from "@langchain/community/vectorstores/chroma";

const chromaFiltered = await chromaStore.similaritySearch(
  "神经网络",
  5,
  { "year": { "$gte": 2023 } }  // 只返回 2023 年及以后的文档
);

// ==========================================
// Pinecone 的过滤语法
// ==========================================
const pineconeFiltered = await pineconeStore.similaritySearch(
  "transformer 架构",
  5,
  { "category": { "$eq": "deep-learning" } }  // 只返回深度学习类文档
);
⚠️ 过滤语法因库而异

不同向量数据库的过滤语法各不相同：MemoryVectorStore 支持简单的对象键值匹配； Chroma 支持 $eq、$ne、$gt、$gte、$lt、$lte、$in 等运算符； Pinecone 也有自己的一套过滤表达式。建议查阅各数据库的官方文档了解详细语法。

🔨 实战演练

实战一：MemoryVectorStore 完整存储与检索流程

场景描述：你需要构建一个小型知识库，包含 3+ 篇关于人工智能技术的中文文章。 用户可以通过自然语言提问来搜索相关内容。

你的任务：使用 MemoryVectorStore 完成文档存储、检索的完整流程， 并与 LLM 配合实现"知识库问答"。

// ==========================================

// ==========================================
// 3. 相似度搜索测试
// ==========================================
async function searchSimilarity(query: string, k: number = 3) {
  console.log(`\n🔍 搜索: "${query}" (k=${k})`);
  console.log("-".repeat(60));

  const results = await vectorStore.similaritySearch(query, k);

  results.forEach((doc, i) => {
    console.log(`\n📄 结果 ${i + 1} [相似度排名 ${i + 1}]`);
    console.log(`   标题: ${doc.metadata.title}`);
    console.log(`   分类: ${doc.metadata.category}`);
    console.log(`   内容: ${doc.pageContent.substring(0, 100)}...`);
  });

  return results;
}

// ==========================================
// 4. 结合 LLM 实现知识库问答
// ==========================================
async function answerWithKnowledge(query: string) {
  // 从向量库检索相关文档
  const relevantDocs = await vectorStore.similaritySearch(query, 2);
  const context = relevantDocs.map((d) => d.pageContent).join("\n\n---\n\n");

  // 创建带上下文的提示模板
  const prompt = ChatPromptTemplate.fromMessages([

技巧一：不同检索策略对比

选择一个合适的检索策略对于构建高效的 RAG 系统至关重要。下面我们比较 四种主要检索策略的差异和适用场景。

策略核心思想优点缺点适用场景
相似度检索（Similarity） 按向量余弦相似度排序取 Top-K 简单、快速、结果精确匹配查询意图 结果可能高度冗余，缺乏多样性 查询意图明确，需要精确匹配的场景
MMR 检索 平衡相关性和多样性 结果多样、覆盖不同方面 略慢于普通相似度（需要多一步排序） 需要全面了解某个主题的不同方面
MultiQuery 检索 LLM 生成多个查询并分别检索 召回率高、覆盖不同问法 额外 LLM 调用、延迟较高 用户查询含糊不清或有多重含义
得分阈值检索（Score Threshold） 设定最低相似度门槛，低于门槛的过滤掉 检索质量可控、避免低质量匹配 阈值难以确定，太高容易结果为空 对检索质量标准严格的场景

技巧二：自定义分数阈值调优

scoreThreshold 的值取决于你的 Embedding 模型和向量库的规模。 通常建议先做一次不加阈值的检索，观察返回结果的相似度分数分布，再选择合适的阈值。 对于 OpenAI text-embedding-3-small 模型，余弦相似度一般在 0.6~0.95 之间。

TypeScript 
// ==========================================
// 观察相似度分数分布
// ==========================================
async function inspectSimilarityScores(store: MemoryVectorStore, query: string) {
  // 注意：similaritySearchWithScore 会返回 [Document, number] 对
  // 数值就是余弦相似度分数
  const resultsWithScore = await store.similaritySearchWithScore(query, 10);

  console.log(`查询: "${query}"`);
  console.log("相似度分数分布：");
  resultsWithScore.forEach(([doc, score], i) => {
    console.log(`  #${i + 1}: score=${score.toFixed(4)} | ${doc.pageContent.substring(0, 50)}...`);
  });

  // 建议阈值：
  if (resultsWithScore.length >= 3) {
    const threshold = resultsWithScore[2][1]; // 取第 3 名的分数作为参考
    console.log(`\n💡 建议 scoreThreshold ≈ ${threshold.toFixed(4)}（第 3 名的分数）`);
  }
}

技巧三：将 Retriever 集成到 LCEL 管道

Retriever 实现了 Runnable 接口，因此可以直接用在 LCEL 管道中。 这是构建 RAG 系统的基础模式。

TypeScript 
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { OpenAIEmbeddings } from "@langchain/openai";
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";
import { Document } from "@langchain/core/documents";

// ==========================================
// 构建 RAG 管道
// ==========================================

// 1. 准备向量存储
const vectorStore = await MemoryVectorStore.fromDocuments(docs, embeddings);
const retriever = vectorStore.asRetriever({ k: 3 });

// 2. 定义提示模板
const ragPrompt = ChatPromptTemplate.fromMessages([
  ["system", "你是一个知识问答助手。请根据以下参考资料回答问题。\n\n{context}"],
  ["human", "{question}"],
]);

// 3. 定义模型和解析器
const model = new ChatOpenAI({ model: "gpt-4o-mini" });
const parser = new StringOutputParser();

// 4. 构建 RAG 管道
//    输入: { question: "..." }
//    → 检索文档 → 格式化上下文 → 注入提示 → LLM → 解析输出
const ragChain = {
  // 先检索并将文档格式化为上下文字符串
  context: async (input: { question: string }) => {
    const docs = await retriever.invoke(input.question);
    return docs.map((d) => d.pageContent).join("\n\n");
  },
  // 透传问题
  question: (input: { question: string }) => input.question,
}
  .pipe(ragPrompt)  // 注入到提示模板
  .pipe(model)      // 调用 LLM
  .pipe(parser);    // 提取文本

// 5. 使用
const answer = await ragChain.invoke({
  question: "什么是向量数据库？",
});
console.log(answer);

🧠 知识检查点

📝 自测题：检查你对本章内容的理解
1. VectorStore 的 fromDocuments() 和 addDocuments() 有什么区别？

fromDocuments() 是一个类静态方法，它会创建一个新的向量存储实例，同时将传入的文档全部向量化并存入。而 addDocuments() 是一个实例方法，用于向已有的向量存储实例中增量添加新的文档。简单说：fromDocuments 是"创建并填充"，addDocuments 是"追加"。

2. Retriever 和 VectorStore 的关系是什么？为什么需要 Retriever 这个抽象？

Retriever 是 VectorStore 的上层抽象。VectorStore 负责向量数据的存储和搜索，而 Retriever 提供了一个标准化的检索接口（invoke(query)），屏蔽了底层向量库的差异。更重要的是，Retriever 实现了 Runnable 接口，可以无缝集成到 LCEL 管道中。任何 VectorStore 通过 asRetriever() 变成 Retriever 后，就可以和其他 Runnable 组件用 | 运算符组合。

3. MMR 检索和普通相似度检索的核心区别是什么？

普通相似度检索只考虑结果与查询的相关性，可能导致结果高度冗余（比如搜"机器学习框架"，返回的三个结果都是关于 TensorFlow 的）。MMR 则同时考虑两个因素：① 结果与查询的相关性；② 结果之间的多样性。通过 lambda 参数平衡两者，λ 越大越重视相关性，λ 越小越重视多样性。MMR 会先取 fetchK 个候选，再从中挑选出既相关又互不相似的 k 个最终结果。

4. MultiQueryRetriever 在什么场景下最有用？它有什么代价？

MultiQueryRetriever 最适用于以下场景：① 用户查询含糊不清或有多种可能的解读方式；② 同一个概念有多种不同的表述方式（比如"AI""人工智能""大模型"）；③ 需要尽可能高的召回率（Recall）的场景。它的代价是每次检索需要额外调用 LLM 生成多个查询变体，增加了延迟和成本。此外，多个查询的检索结果可能会有大量重复，需要去重处理。

5. 元数据过滤（Metadata Filtering）是如何工作的？它和语义检索是什么关系？

元数据过滤是在语义检索之前或之后施加的一层结构化筛选条件。比如先通过 filter: { year: { $gte: 2023 } } 选出 2023 年以后的文档，再在这些文档中进行语义相似度搜索。它和语义检索是互补关系：语义检索解决"内容是否相关"的问题，元数据过滤解决"属性是否匹配"的问题。两者结合可以实现精确搜索——既要在语义上相关，又要在属性上匹配特定条件。

🐛 常见错误

❌ 错误 1：未安装 Embeddings 模型导致运行时错误

症状：调用 fromDocuments() 时抛出 Error: OpenAI API key not found 或模型加载失败。

原因：MemoryVectorStore 本身不需要外部依赖，但它依赖 Embeddings 模型将文本转化为向量。如果使用的 OpenAIEmbeddings 没有配置 API Key，就会失败。

解决：确保设置了 OPENAI_API_KEY 环境变量，或者使用其他 Embeddings 实现（如 FakeEmbeddings 用于测试）。对于离线测试，可以使用 @langchain/community 中的本地嵌入模型。

❌ 错误 2：scoreThreshold 设置过高导致检索结果为空

症状：设置了 scoreThreshold: 0.95，但检索结果始终为空。

原因：阈值设置过高，没有文档的相似度评分达到该值。不同 Embedding 模型和不同数据集产生的相似度分数分布差异很大。

解决：先用不带阈值的检索观察分数分布，然后根据实际数据选择合适的阈值。通常建议从 0.7 开始尝试，逐步调高。

❌ 错误 3：MemoryVectorStore 的数据在程序重启后丢失

症状：重新运行程序后，之前存入的向量数据都不见了。

原因：MemoryVectorStore 将数据存储在内存中，进程退出后数据自然消失。这不是 bug，而是它的设计特性。

解决：如果需要在程序重启后保留数据，应该使用持久化向量数据库（如 Chroma、Pinecone），或者将文档源文件和 Embedding 结果缓存到文件系统/数据库中，启动时重新加载。

❌ 错误 4：不同向量数据库的过滤语法混淆

症状：在 MemoryVectorStore 中使用了 Chroma 的 $gte 过滤语法，导致过滤失效或报错。

原因：不同向量数据库的元数据过滤语法不同。MemoryVectorStore 只支持简单的键值精确匹配，而 Chroma 和 Pinecone 支持更复杂的比较运算符。

解决：查阅你所使用的向量数据库的官方文档，确认其支持的过滤语法。在使用 asRetriever() 时，filter 参数的格式取决于底层 VectorStore 的实现。
