# MCP 常见错误排错指南

## 1. Server 连接失败
**现象：** 启动 MCP Server 后客户端无法连接
**原因：** 传输协议配置错误或端口被占用
**方案：** 检查传输协议参数，确认端口未被占用，检查防火墙设置

## 2. 客户端 listTools 返回空数组
**现象：** 成功连接但获取不到任何工具
**原因：** 工具注册在 connect() 之后才执行
**方案：** 在 connect() 之前完成所有 server.tool() 调用

## 3. 工具调用返回 "Tool not found"
**现象：** 工具名正确但调用失败
**原因：** 工具名大小写不匹配，或带了 namespace 前缀
**方案：** 使用 listTools() 返回的原始工具名调用

## 4. Tool 参数验证失败
**现象：** 调用工具时返回参数错误
**原因：** LLM 生成的参数不符合 Zod Schema
**方案：** 提供清晰的参数描述和名称，避免使用过于复杂的嵌套 Schema

## 5. 多个 Server 工具名冲突
**现象：** 连接多个 Server 时出现同名工具
**原因：** 不同 Server 提供了同名的工具
**方案：** 在 Host 端添加 namespace 前缀，如 `fs__read_file`

## 6. 工具返回内容被截断
**现象：** 大文件的返回内容不完整
**原因：** 工具返回的 content 没有做分页或限制
**方案：** 对大内容做分页返回，或设置最大返回长度

## 7. 错误信息丢失
**现象：** 工具报错但 LLM 看不到具体原因
**原因：** isError: true 但没有提供有意义的错误信息
**方案：** 在返回的 text 中包含具体的错误原因和解决方案建议

## 8. 文件路径跨平台问题
**现象：** Windows 上路径正常，Linux 上报错
**原因：** 路径分隔符差异
**方案：** 统一使用 `path.resolve()` 处理路径，避免硬编码分隔符

## 9. Resource URI 重复
**现象：** 不同 Server 有相同 URI 路径的 Resource
**原因：** 没有在 URI 中加 namespace 前缀
**方案：** 使用 Server 名称作为 URI 前缀，如 `fs://config` 和 `db://config`

## 10. 同步阻塞导致响应卡死
**现象：** Agent 在等待工具返回时停止响应
**原因：** 工具内部有同步阻塞操作
**方案：** 工具内部使用异步操作，实现合理的超时机制

## 11. SSE 连接频繁断开
**现象：** 生产环境中 SSE 连接不稳定
**原因：** 缺少心跳检测和重连机制
**方案：** 实现心跳包（ping/pong），客户端实现指数退避重连

## 12. 大文件传输导致内存溢出
**现象：** 读取大文件时 Server 进程崩溃
**原因：** 没有做流式读取或大小限制
**方案：** 使用流式读取（createReadStream），设置最大文件大小限制

## 13. 并发请求导致数据不一致
**现象：** 多个 Agent 同时调用同一个工具导致数据紊乱
**原因：** 工具内部有共享的可变状态
**方案：** 工具保持无状态设计，使用数据库事务或锁机制

## 14. 跨域请求被 CORS 拦截
**现象：** 浏览器端 Host 无法连接远程 MCP Server
**原因：** Server 未设置 CORS 头
**方案：** 在 HTTP Server 中添加 CORS 中间件，允许 Host 的域名

## 15. Resource 返回的数据格式不符合预期
**现象：** LLM 读取 Resource 内容时解析失败
**原因：** Resource 返回的 mimeType 与实际内容不匹配
**方案：** 正确设置 mimeType（JSON 用 application/json，文本用 text/markdown）

## 16. Dynamic ResourceTemplate URI 解析失败
**现象：** 带参数的 Resource URI 无法匹配
**原因：** URI 模板定义与实际传入的 URI 格式不一致
**方案：** 检查 ResourceTemplate 的 URI 模式，确保占位符与实际参数匹配

## 17. Streamable HTTP 模式超时
**现象：** 长连接一段时间后自动断开
**原因：** 没有设置 keepalive 超时时间
**方案：** 在 HTTP 服务器中配置 keepAliveTimeout 和 headersTimeout

## 18. tool 函数中未捕获异常
**现象：** 工具调用返回 500 内部错误
**原因：** tool 处理函数中没有 try/catch 包裹
**方案：** 在 tool 处理函数中使用 try/catch，将错误转换为 isError 格式返回

## 19. 客户端关闭后 Server 未清理资源
**现象：** 断开连接后 Server 进程残留
**原因：** 没有监听关闭事件
**方案：** 在 Server 端监听 close 事件，清理数据库连接和文件句柄

## 20. Prompt 模板中使用了不受支持的参数类型
**现象：** server.prompt() 注册的模板调用时报错
**原因：** Prompt 参数包含了函数等非序列化类型
**方案：** 确保 Prompt 参数只使用字符串、数字、布尔等基础类型
