# 记忆系统排错指南

> 涵盖 Agent 记忆系统开发中的 18 个常见问题的现象、原因和解决方案。

---

## 1. 对话太长导致 Token 超限

**现象：** Agent 运行一段时间后 API 返回 400 错误提示 `prompt is too long`，或响应速度明显变慢。

**原因：**
- 没有对历史消息做压缩或截断，messages 数组持续增长
- 每次工具调用返回的大量数据不断累积
- 模型 context window 有限

**方案：**
- 使用滑动窗口策略，只保留最近 20-30 轮对话
- 定期对旧消息做 LLM 摘要压缩
- 监控当前 token 数，接近限制时触发压缩
- 示例实现：
```typescript
const MAX_TOKENS = 100_000;
if (estimateTokens(messages) > MAX_TOKENS) {
  messages = await compressMessages(messages);
}
```

---

## 2. 记忆检索不准确

**现象：** 检索到的记忆与当前问题不相关，或返回了大量无关结果。

**原因：**
- Embedding 模型质量不够好（如使用通用模型而非领域特定模型）
- topK 设置过大，返回了过多低相似度结果
- 相似度阈值（score threshold）过低或未设置

**方案：**
- 使用更高质量的 Embedding 模型（如 `text-embedding-3-large` 或 `bge-large`）
- 设置合理的相似度阈值（如 > 0.75）
- 减小 topK（建议 3-5）
- 对检索结果做重排序（reranking），提高准确率

---

## 3. 记忆存储了无关信息

**现象：** 向量数据库中充斥着大量无意义的内容（如 "好的"、"我知道了"），导致检索结果质量下降。

**原因：**
- 没有重要性评估机制，所有对话都被存入记忆
- 短期记忆和长期记忆没有分开存储

**方案：**
- 实现记忆重要性评分，只存储 `importance >= 5` 的内容
- 在存储前过滤掉无信息量的回复（如问候语、确认性回复）
- 建立短期缓存区，只有被多次引用的内容才进入长期记忆

---

## 4. 跨会话记忆无法访问

**现象：** 用户第二次会话时，Agent 完全不记得之前的对话内容。

**原因：**
- 记忆只存储在内存中（进程级），进程重启后丢失
- 没有根据 userId 隔离存储
- 向量数据库没有持久化配置

**方案：**
- 使用持久化存储（ChromaDB 持久化模式、PostgreSQL、Redis）
- 检索时始终传入 `userId` 作为过滤条件
- 使用 Docker 部署数据库实例确保持久化

---

## 5. 记忆冲突（新旧记忆矛盾）

**现象：** Agent 同时检索到用户之前说的 "喜欢安静" 和最近说的 "喜欢热闹"，在回答中自相矛盾。

**原因：**
- 没有对记忆添加时间戳
- 检索时没有按时间排序
- 没有实现记忆冲突检测

**方案：**
- 存储时添加时间戳字段
- 检索结果按时间降序排列（最新的优先）
- 实现冲突检测：当同一实体的记忆相互矛盾时，保留置信度高的或最新的
- 在返回给 LLM 时标注记忆的时间："2024年1月：用户表示喜欢安静；2024年6月：用户表示喜欢热闹"

---

## 6. 记忆检索延迟过高

**现象：** 每次检索记忆需要 500ms 以上，严重影响 Agent 响应速度。

**原因：**
- 向量数据库中没有建立索引
- 数据集过大（百万级）
- 数据库与服务不在同一区域

**方案：**
- 为向量列创建索引（如 HNSW、IVF）
- 使用 Redis 做高频访问的缓存层
- 将数据库部署在与 Agent 服务相同的区域
- 使用 Approximate Nearest Neighbor（ANN）替代精确搜索

---

## 7. 工作记忆在任务中断后丢失

**现象：** Agent 在执行多步骤任务时，某一步需要用户确认，用户回复后 Agent 忘记了之前的步骤状态。

**原因：**
- 工作记忆只存在 messages 数组中，没有持久化
- 系统重启或部署更新导致进程内存丢失

**方案：**
- 使用 Redis 存储工作记忆（设置 TTL 自动过期）
- 序列化任务状态为 JSON 存储在数据库中
- 在每条 user 消息中附带当前任务上下文

---

## 8. 摘要压缩丢失关键细节

**现象：** 使用 LLM 压缩对话后，Agent 无法回答关于被压缩部分的细节问题。

**原因：**
- 压缩 prompt 不够好，丢失了关键信息
- 压缩比过大，每 10 轮对话压缩成一句话

**方案：**
- 优化压缩 prompt，明确要求保留：用户偏好、决策理由、具体数字、日期、承诺
- 控制压缩比，每 5-8 轮压缩为一个段落而非一句话
- 分层压缩：保留最近几轮原始消息，只压缩更早的部分

---

## 9. 知识图谱关系提取不准确

**现象：** LLM 从对话中提取的三元组（实体-关系-实体）大量错误或无关。

**原因：**
- 提取 prompt 没有给出足够具体的例子
- LLM 把假设性内容也当成了确定事实

**方案：**
- 在提取 prompt 中加入 Few-shot 示例
- 要求 LLM 标注置信度（0-1），只存储高置信度的三元组
- 添加实体类型约束（如只提取 PERSON、ORG、LOCATION 等）

---

## 10. 同义记忆重复存储

**现象：** 用户多次表达了类似的意思（如 "我喜欢吃辣的"和"我偏爱麻辣口味"），向量数据库中存在多条重复记忆。

**原因：**
- 没有在存储前做相似度去重检测
- Embedding 模型没有将同义句映射到相近位置

**方案：**
- 存入前检索已有记忆，相似度 > 0.9 则更新而非新增
- 实现记忆合并函数，将相似记忆合并
- 设置每日/每周的批量去重任务

---

## 11. 用户身份混淆

**现象：** Agent 把用户 A 的记忆展示给了用户 B，导致隐私泄露。

**原因：**
- 检索时没有按 userId 过滤
- 向量数据库的 metadata 过滤条件缺失
- 多租户隔离实现有 bug

**方案：**
- 所有记忆存储和检索都强制加上 `userId` 过滤
- 使用向量数据库的 metadata 过滤功能：`collection.query({ where: { userId } })`
- 为每个用户建立独立的 collection 或数据库

---

## 12. 长期记忆写入性能瓶颈

**现象：** 高频写入记忆时（如每轮对话都写入），系统响应变慢。

**原因：**
- 每次对话都同步写入向量数据库
- Embedding 生成 API 调用耗时

**方案：**
- 使用写缓冲区：批量写入而非逐条写入
- 异步写入：将记忆存储放入消息队列
- 只有高重要性（importance ≥ 7）的记忆才立即写入

---

## 13. 记忆回放导致上下文污染

**现象：** 检索出的记忆内容被混入当前对话，导致 Agent 回答带了不相关的历史信息。

**原因：**
- 所有检索到的记忆都被无差别地放入 context
- 没有对记忆做相关性过滤

**方案：**
- 只将相似度 > 0.8 的记忆注入 context
- 对检索到的记忆按相关度排序，只取 top 3
- 在 System Prompt 中说明：「以下是可能相关的历史记忆，请参考它们来回答」

---

## 14. 对话历史与检索记忆的信息矛盾

**现象：** 当前对话中用户刚说的内容与从长期记忆中检索到的内容矛盾，Agent 不知道以哪个为准。

**原因：**
- 没有设定信息优先级规则
- 短期记忆（当前对话）和长期记忆（历史记录）权重相同

**方案：**
- 设定优先级：当前对话 > 最近记忆 > 早期记忆
- 显式标注信息来源和时间
- in Prompt 中强调：`以用户当前的说法为准，引用历史记忆时标明时间`

---

## 15. Embedding 模型更新后向量空间迁移

**现象：** 更换了 Embedding 模型后，新旧向量无法在同一空间进行语义检索。

**原因：**
- 不同 Embedding 模型输出的向量维度或分布不同
- 新老模型没有在同一语义空间

**方案：**
- 更换 Embedding 模型后，重新为所有文档生成向量
- 使用兼容性好的模型（如 `text-embedding-ada-002` 到 `text-embedding-3-small`）
- 维护模型版本号，检索时使用同版本的向量

---

## 16. 记忆系统中敏感数据未脱敏

**现象：** 记忆存储中保存了用户的密码、身份证号等敏感信息。

**原因：**
- 存储前没有对内容做敏感信息检测
- 没有建立数据分类分级机制

**方案：**
- 存储前用正则表达式检测敏感数据模式
- 对敏感字段做脱敏处理（如 `password: "***"`）
- 遵循数据最小化原则，只存储必要信息
- 定期清理老化和不必要的记忆

---

## 17. 记忆蒸馏效率低下

**现象：** 每天/每周的批量蒸馏任务耗时过久，处理数万条记忆。

**原因：**
- 全量蒸馏，每次处理所有记忆而非增量
- 没有预处理过滤掉低价值记忆

**方案：**
- 只蒸馏新产生的记忆（增量蒸馏）
- 蒸馏前过滤：只处理 importance ≥ 5 的记忆
- 分批次、多线程处理
- 使用更便宜的模型（如 Claude Haiku）做蒸馏

---

## 18. 记忆召回率低导致 Agent 表现差

**现象：** 用户重复提到之前说过的事情，Agent 完全不记得，影响用户体验。

**原因：**
- 检索策略单一（只靠向量相似度）
- 没有结合关键词搜索
- 记忆缺少元数据标签

**方案：**
- 采用混合检索（向量 + BM25 关键词）
- 为记忆添加多标签分类（如 `tags: ["偏好", "任务", "个人信息"]`）
- 使用 Multi-Query 策略：将用户问题改写为多个查询再分别检索
- 增加检索的 topK 后做重排序

---

## 🔑 关键检查清单

| 检查项 | 说明 |
|--------|------|
| ☐ 记忆分级 | 短期/工作/长期记忆分开管理 |
| ☐ 重要性评估 | 只存储高分值记忆 |
| ☐ 时间戳标注 | 每条记忆带时间，检索按时间排序 |
| ☐ 用户隔离 | 所有操作绑定 userId |
| ☐ 去重机制 | 相似记忆合并或更新 |
| ☐ 压缩策略 | 超出 token 限制时自动压缩 |
| ☐ 敏感过滤 | 存储前脱敏敏感数据 |
| ☐ 持久化配置 | 数据库配置为持久化模式 |
