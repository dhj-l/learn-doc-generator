# RAG 系统排错指南

> 涵盖 RAG 管道开发中的 18 个常见问题的现象、原因和解决方案。

---

## 1. 检索不到相关文档

**现象：** 用户提出明确的问题，但检索模块返回空结果或完全不相关的内容。

**原因：**
- 分块策略不当，块太小或边界切碎了语义
- Embedding 模型质量不佳
- 仅使用了向量检索，没有结合关键词搜索
- Query 与文档的语言不一致（中英文混查）

**方案：**
- 尝试不同的分块策略（推荐递归分块，chunkSize 500-1000）
- 更换更高质量的 Embedding 模型（如 `bge-large-zh`、`text-embedding-3-large`）
- 采用混合检索（向量 + BM25）
- 检查 query 和文档的语言是否匹配

---

## 2. 回答出现幻觉（Hallucination）

**现象：** LLM 的回答中包含文档中没有的信息，或与文档内容矛盾。

**原因：**
- Prompt 中没有强调「仅根据提供文档回答」
- 检索到的文档不相关，LLM 自行补充知识
- LLM 过度自信，编造了不存在的细节

**方案：**
- 在 System Prompt 中强调：「你只能基于以下提供的文档内容回答。如果文档中没有相关信息，请直接说不知道。」
- 提高检索质量，确保注入生成阶段的文档是相关的
- 使用 Faithfulness 评估指标监控幻觉率
- 增加验证步骤：让 LLM 在回答前先输出引用的文档片段

---

## 3. 回答不够详细

**现象：** 检索到了相关文档，但 LLM 的回答过于简略，只给出了概要。

**原因：**
- topK 太小，注入的上下文不足
- chunkSize 太小，每个块包含的信息量有限
- LLM 没有充分使用文档中的细节

**方案：**
- 增大 topK（从 3 增加到 5-8）
- 增大 chunkSize（从 500 增加到 800-1000）
- 在 Prompt 中要求「请从文档中提取所有相关信息，详细回答」
- 使用 Multi-hop RAG 多次检索补充信息

---

## 4. 检索延迟太高

**现象：** 每次检索耗时超过 1 秒，整体回答延迟太大。

**原因：**
- 向量数据库没有建索引
- 数据集规模大但没有使用 ANN 算法
- 数据库服务与应用服务不在同一区域

**方案：**
- 建立 HNSW 或 IVF 索引
- 使用 Approximate Nearest Neighbor 替代精确搜索
- 将数据库部署在同区域
- 增加 Redis 缓存层，缓存高频查询结果
- 减小 topK 以降低搜索空间

---

## 5. Token 成本太高

**现象：** RAG 系统的 API 调用费用远超预期。

**原因：**
- 每次检索返回过多文档，context 过长
- 没有对检索结果做精简
- Embedding 模型调用频率过高

**方案：**
- 限制注入生成阶段的文档数量（topK ≤ 5）
- 对检索结果做摘要精简后再注入
- 使用 Prompt Caching（Claude API 支持）
- 用便宜的模型（Claude Haiku）做重排和摘要
- 对文档做预过滤，只检索与 query 相关的部分

---

## 6. 分块切碎了语义

**现象：** 一个完整的实体、概念或逻辑段落被分割到不同的块中，导致检索到的块语义不完整。

**原因：**
- chunkSize 设置过小，切断了句子或段落
- 分隔符选择不当，在语义边界中间切断
- 没有设置 chunkOverlap

**方案：**
- 增大 chunkSize（至少 500 tokens）
- 使用语义分块（Semantic Chunking）而非固定长度分块
- 设置合适的 chunkOverlap（至少 chunkSize 的 10%）
- 使用递归分块，优先在段落边界分割

---

## 7. 混合检索权重不合理

**现象：** 混合检索的结果不如单一检索，语义搜索和 BM25 的互补效果没有体现。

**原因：**
- 语义搜索和 BM25 的融合权重设置不合理
- 使用的融合算法（如 RRF）参数不当

**方案：**
- 使用 Reciprocal Rank Fusion（RRF），常数 k 设置为 60
- 根据场景调整权重：代码搜索提高 BM25 权重；开放问答提高语义权重
- 通过实验确定最佳权重比
- 考虑使用学习型融合模型替代固定权重

---

## 8. Embedding 维度不匹配

**现象：** 存储时向量维度为 1536，检索时生成的是 768 维，导致查询失败。

**原因：**
- 开发和生产环境使用了不同的 Embedding 模型
- 模型版本升级后向量维度发生变化
- 不同语言的模型输出维度不同

**方案：**
- 在配置文件中锁定 Embedding 模型版本
- 构建向量前检查维度一致性
- 模型升级后需要重新生成所有向量
- 在向量数据库中记录向量的 model 版本号

---

## 9. 文档加载格式错误

**现象：** 加载 PDF、Word 等文档时，文字乱码、格式丢失或提取出大量无关内容。

**原因：**
- PDF 是扫描版图片而非文字版
- 文档编码格式（UTF-8、GBK）未正确处理
- 文档中有表格、图表等无法正确提取的内容

**方案：**
- 扫描版 PDF 先做 OCR（如 Tesseract、Azure OCR）
- 统一使用 UTF-8 编码
- 对不同文档类型使用专门的加载器（`PDFLoader`、`DocxLoader`）
- 对复杂表格使用专门的表格解析工具

---

## 10. 检索结果重复

**现象：** topK 结果中有多条内容高度相似的文档块，浪费了上下文窗口。

**原因：**
- 相邻的文档块内容重叠过大
- 没有对检索结果做去重
- chunkOverlap 过大导致多个块包含相同内容

**方案：**
- 检索后对结果做相似度去重（相似度 > 0.9 的只保留一条）
- 减小 chunkOverlap（不超过 chunkSize 的 10%）
- 使用 Maximal Marginal Relevance（MMR）算法选择多样化的结果

---

## 11. RAG 评估指标低

**现象：** Faithfulness、Relevancy 等评估指标低于预期目标。

**原因：**
- 检索质量差导致上下文不相关
- Prompt 未充分约束生成行为
- 评估数据集与领域不匹配

**方案：**
- 先诊断检索质量：检查 Context Precision 和 Context Recall
- 优化检索策略（混合检索、重排序）
- 迭代优化 Prompt 模板
- 构建与业务场景匹配的评估数据集
- 使用 RAGAS 等框架做系统性评估

---

## 12. Query 改写后效果反而差

**现象：** 使用 LLM 对用户 query 进行改写（扩展或优化）后，检索结果变差。

**原因：**
- Query 改写引入了噪声或改变了原意
- 改写后的 query 与文档的语料分布不匹配
- 过度扩展导致查询模糊

**方案：**
- 简化 Query 改写逻辑，只做最小的优化（如拼写纠正）
- 保留原始 query 的同时增加改写版本（Multi-Query）
- 对改写结果做 A/B 测试
- 使用 Query 分解（Decomposition）替代扩展

---

## 13. 多语言混用导致检索失败

**现象：** 文档库包含中英文混合内容，用户用中文提问但检索到英文文档，或反之。

**原因：**
- Embedding 模型对多语言支持不佳
- 没有考虑查询语言与文档语言的匹配

**方案：**
- 使用多语言 Embedding 模型（如 `multilingual-e5-large`）
- 在 metadata 中标记文档语言，检索时优先匹配语言
- 检测 query 语言，检索前做语言过滤
- 对中文文档使用中文 Embedding 模型

---

## 14. 输入文档过长导致分块后上下文丢失

**现象：** 单个文档包含多个主题，分块后每个块无法独立回答问题。

**原因：**
- 文档没有先做章节分割
- 块之间没有保留必要的上下文引用

**方案：**
- 先按文档结构（章节、标题）做粗分割，再对每个章节做细分割
- 保留文档元数据（标题、章节号）作为上下文
- 在 Prompt 中注入文档层级信息：「以下内容来自《XXX》第 3 章」

---

## 15. 向量数据库扩容困难

**现象：** 文档从 10 万增加到 100 万后，检索速度急剧下降。

**原因：**
- 没有使用支持分布式扩展的向量数据库
- 索引类型不适合大规模数据

**方案：**
- 使用支持分布式部署的向量数据库（Milvus、Weaviate、Qdrant）
- 切换到更高效的索引（HNSW）
- 实现数据分片（Sharding）
- 使用量化（Product Quantization）压缩向量

---

## 16. 实时更新时索引不一致

**现象：** 文档更新后，查询仍然返回旧的文档内容。

**原因：**
- 文档更新后没有同步更新向量数据库
- 缓存过期时间过长

**方案：**
- 实现文档变更监听（Change Data Capture），自动触发索引更新
- 为每个文档块记录版本号，检索时过滤旧版本
- 设置合理的缓存 TTL
- 使用增量索引而非全量重建

---

## 17. 重排序增加延迟但精度提升不明显

**现象：** 引入了重排序（Rerank）步骤后延迟增加了 500ms，但准确性提升不到 5%。

**原因：**
- 初筛（retrieval）质量已经很好了，重排序收益有限
- 重排序模型与领域不匹配
- topK 过大，重排序了大量无关文档

**方案：**
- 评估重排序的实际收益，如果 ROI 低则移除
- 使用领域微调的重排序模型
- 减小传入重排序的候选数量（从 20 减到 10）
- 仅在精度要求高的场景启用重排序

---

## 18. 系统测试环境与生产环境表现不一致

**现象：** 在测试集上 RAG 系统表现良好，上线后用户反馈效果差。

**原因：**
- 测试数据分布与真实用户查询分布不一致
- 测试集过小或过时
- 生产环境文档库不同（版本不一致）

**方案：**
- 构建反映真实用户分布的测试集
- 持续收集生产环境的 query 日志，定期更新测试集
- 建立线上 A/B 测试机制
- 确保测试环境和生产环境的文档库版本一致

---

## 🔑 关键检查清单

| 检查项 | 说明 |
|--------|------|
| ☐ 分块参数 | chunkSize 500-1000，chunkOverlap 50-200 |
| ☐ 检索策略 | 混合检索（向量 + BM25）优于单一检索 |
| ☐ Embedding 维度 | 存储和检索使用同一模型 |
| ☐ 用户隔离 | 多租户时 metadata 过滤 |
| ☐ 去重机制 | MMR 或相似度去重 |
| ☐ 评估监控 | Faithfulness、Relevancy 周期评估 |
| ☐ 缓存策略 | 高频 query 结果缓存 |
| ☐ 文档版本 | CDC 机制保持索引同步 |
