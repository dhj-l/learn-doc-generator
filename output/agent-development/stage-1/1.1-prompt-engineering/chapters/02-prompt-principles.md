## 💡 核心概念

### 概念一：为什么 Prompt 设计如此重要？

想象你在一个餐厅里点餐：

```
❌ 差的点餐方式：
"给我来点吃的。"
→ 服务员不知道你要什么，可能给你端来任何东西

✅ 好的点餐方式：
"我要一份黑椒牛排，七分熟，配薯条和蔬菜沙拉，
 酱汁另外放，不要放香菜。"
→ 服务员清楚知道你要什么，准确执行
```

**Prompt 就是你对 AI 的「点餐」**。你的指令越清晰，AI 的输出就越符合你的期望。

一个真实的对比：

```
❌ 糟糕的 Prompt：
"帮我写一段代码"

✅ 好的 Prompt：
"用 TypeScript 写一个函数，接收一个用户数组（每个用户有 name 和 age 属性），
返回所有年龄大于 18 岁的用户姓名列表。
要求：使用 Array.filter 和 Array.map，添加类型注释，包含单元测试。"
```

> **💡 核心洞察**
>
> Prompt Engineering 的本质不是「讨好 AI」，而是**精确表达你的需求**。就像写需求文档一样——越精确，实现效果越好。

### 概念二：Prompt 设计的六大原则

#### 原则 1：清晰性（Clarity）

**「不要让模型猜你在想什么」**

```typescript
// ❌ 模糊 — 模型需要猜测你的意图
const vague = "优化这段代码";

// ✅ 清晰 — 明确告诉模型要优化什么
const clear = `请从以下三个维度优化这段代码：
1. 时间复杂度：当前是 O(n²)，尝试优化到 O(n log n)
2. 内存使用：减少不必要的中间变量
3. 可读性：添加有意义的变量名和注释`;
```

**清晰性的关键技巧：**
- 使用具体动词：「列出」「比较」「解释」「创建」而不是「处理」「搞定」「弄一下」
- 避免歧义词：「一些」→ 指定数量，「很快」→ 指定时间，「更好」→ 指定比较标准
- 一次只问一件事：不要在一句话里塞三个不同的任务

#### 原则 2：具体性（Specificity）

**「细节越多，结果越准」**

```typescript
// ❌ 不够具体
const vague = "写一个 API 接口";

// ✅ 高度具体
const specific = `用 Express.js + TypeScript 创建一个 RESTful API 端点：
- 路由：POST /api/users
- 请求体：{ name: string, email: string, age: number }
- 验证规则：email 必须合法格式，age 必须在 0-150 之间
- 成功响应：201, 返回创建的用户对象（含 id）
- 错误响应：400, 返回 { error: string, details: string[] }
- 数据库：使用 SQLite，定义 users 表
- 包含请求参数验证中间件`;
```

#### 原则 3：结构化（Structure）

**「用格式引导模型的输出结构」**

```typescript
// 使用 Markdown 标题、编号列表、XML 标签来组织 Prompt

const structured = `
## 任务
分析以下代码的安全漏洞

## 输入代码
\`\`\`javascript
${codeSnippet}
\`\`\`

## 输出格式
请按以下结构输出分析结果：

### 漏洞概述
（一句话总结发现的安全问题数量和严重程度）

### 详细分析
对每个漏洞：
1. **漏洞名称**：xxx
2. **严重程度**：高/中/低
3. **位置**：第 x 行
4. **描述**：xxx
5. **修复建议**：xxx（附代码示例）

### 总体评分
安全性评分：x/10`;
```

#### 原则 4：分步骤（Step-by-Step）

**「把复杂任务拆解成小步骤」**

```typescript
// ❌ 一步到位 — 对复杂任务效果差
const oneShot = "帮我分析这个用户数据集并生成报告";

// ✅ 分步引导 — 逐步推进，每步都可控
const stepByStep = `
请按以下步骤分析用户数据集：

步骤 1：数据概览
- 统计总用户数、字段数量、数据类型
- 识别缺失值和异常值

步骤 2：用户画像
- 按年龄段分组统计
- 按地区分组统计
- 计算活跃度指标

步骤 3：趋势分析
- 注册用户增长趋势（按月）
- 用户留存率计算
- 关键指标的环比变化

步骤 4：生成报告
- 总结关键发现（不超过 5 条）
- 给出可执行的建议（不超过 3 条）
- 输出格式：Markdown 表格 + 摘要文字

请逐步执行，每步完成后等我确认再进行下一步。`;
```

#### 原则 5：给示例（Examples）

**「一个好示例胜过千言万语」**

```typescript
// 不给示例，模型可能理解偏差
const noExample = "把用户评论分类为正面、负面或中性";

// 给出示例，模型准确率大幅提升
const withExamples = `
把用户评论分类为正面、负面或中性。

## 分类示例

评论："这个产品太好用了，强烈推荐！"
分类：正面
理由：明确的正面评价和推荐

评论："产品质量一般，但价格还行"
分类：中性
理由：有正面也有负面，总体中立

评论："等了一个月才收到，而且还是坏的，差评！"
分类：负面
理由：物流问题和质量问题，明确的不满

## 待分类评论
"${userComment}"

请按上述格式输出分类结果。`;
```

#### 原则 6：设边界（Constraints）

**「告诉模型什么不该做」**

```typescript
const withConstraints = `
你是一个技术文档翻译助手。

## 任务
将以下英文技术文档翻译成简体中文

## 约束条件
- ✅ 专有名词保留英文原文（如 API、REST、GraphQL）
- ✅ 代码块不翻译
- ✅ 技术术语首次出现时用「中文（English）」格式
- ❌ 不要添加原文中没有的解释
- ❌ 不要省略任何内容
- ❌ 不要改变原文的格式结构
`;
```

### 概念三：Prompt 四要素框架

基于以上六大原则，我们可以提炼出一个**通用的 Prompt 设计框架**：

```
┌─────────────────────────────────────────┐
│  角色（Role）                            │
│  "你是一个 xxx 专家"                     │
├─────────────────────────────────────────┤
│  任务（Task）                            │
│  "请完成 xxx"                            │
├─────────────────────────────────────────┤
│  约束（Constraints）                     │
│  "要求 xxx / 不要 xxx"                   │
├─────────────────────────────────────────┤
│  格式（Format）                          │
│  "按以下格式输出：xxx"                    │
└─────────────────────────────────────────┘
```

```typescript
// 四要素 Prompt 模板
function buildPrompt(
  role: string,         // 角色定义
  task: string,         // 核心任务
  constraints: string,  // 约束条件
  format: string        // 输出格式
): string {
  return `
# 角色
${role}

# 任务
${task}

# 约束条件
${constraints}

# 输出格式
${format}
`.trim();
}

// 使用示例
const prompt = buildPrompt(
  // 角色
  '你是一个资深前端工程师，精通 React 和 TypeScript，有 10 年大型项目经验。',
  // 任务
  '审查以下 React 组件代码，指出性能问题和代码质量问题。',
  // 约束
  `- 只关注 React 特有的问题，忽略代码风格问题
   - 每个问题必须附带具体的修复代码
   - 按严重程度排序（高→低）
   - 不超过 5 个问题`,
  // 格式
  `每个问题按以下格式输出：
  **问题 N：[问题名称]**
  - 严重程度：🔴 高 / 🟡 中 / 🟢 低
  - 位置：[代码行号]
  - 问题描述：[一句话说明]
  - 修复方案：\`\`\`tsx [修复代码] \`\`\``
);
```

## 🧠 知识检查点

<details>
<summary>点击展开答案</summary>

**Q1：Prompt 设计的六大原则是什么？**

> A：清晰性（Clarity）、具体性（Specificity）、结构化（Structure）、分步骤（Step-by-Step）、给示例（Examples）、设边界（Constraints）。

**Q2：为什么「设边界」比「只描述期望」更重要？**

> A：因为 LLM 有「讨好倾向」——它倾向于生成它认为你想要的内容，而这些内容可能超出或偏离你的需求。明确的边界（「不要做什么」）比单纯的描述（「要做什么」）更能约束模型的输出范围，减少意外行为。

**Q3：四要素框架是什么？在什么场景下使用？**

> A：四要素是「角色 + 任务 + 约束 + 格式」。适用于大多数专业场景的 Prompt 设计。角色设定上下文和专业水平，任务明确要做什么，约束定义质量和边界，格式控制输出结构。对于简单任务可以省略部分要素，但复杂任务建议全部包含。

**Q4：给示例（Few-shot）时，示例数量和质量哪个更重要？**

> A：质量更重要。1-3 个高质量、覆盖不同情况的示例，比 10 个低质量或重复的示例效果更好。示例应该：（1）覆盖典型和边界情况；（2）格式完全一致；（3）难度递进。

## 🐛 常见错误

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| 模型回答太长 | 没有指定长度限制 | 添加约束：「回答不超过 200 字」 |
| 模型回答太笼统 | Prompt 太笼统 | 添加具体维度和示例 |
| 模型忽略了部分指令 | 多个指令混在一起 | 用编号列表分条列出指令 |
| 模型输出格式不稳定 | 没有给出格式要求 | 使用 XML 标签或 Markdown 模板定义格式 |
| 中英文混杂 | 没有明确语言要求 | 添加约束：「全部使用简体中文回答，专有名词除外」 |

- ✅ **Prompt 设计六大原则** — 清晰、具体、结构化、分步骤、给示例、设边界
- ✅ **四要素框架** — 角色 + 任务 + 约束 + 格式，可复用的 Prompt 设计模板
- ✅ **XML 标签** — 组织复杂 Prompt 的利器
