## 💡 核心概念

### 8.1 什么是沙箱？

**用一个类比来理解：**

> 你收到了一封来自陌生人的电子邮件附件。虽然你想打开看看里面是什么，但你担心它可能含有病毒。于是你把它放进一个**隔离箱**——一个透明的、完全密封的容器——你在里面打开它，仔细观察它的行为。如果它真的是病毒，它跑不出来，你的电脑安然无恙。如果不是，你取出结果即可。
>
> Deep Agents 的 **Sandbox（沙箱）** 就是这个隔离箱。当 Agent 需要执行代码时——比如运行用户提交的 Python 脚本、编译代码片段、安装 npm 包——沙箱提供了一个**安全的隔离环境**。代码在沙箱中运行，不会影响到外面的系统。即使代码是有恶意的，它也"跑不出来"。

**沙箱的核心价值有四个：**

1. **安全隔离** —— 恶意代码无法访问宿主系统的文件、网络和进程。即使代码包含 `rm -rf /` 这样的危险命令，它影响的也只是沙箱内部，不会触及你的系统。这是沙箱最核心的价值——它确保了即使 Agent 被提示注入攻击诱导去执行恶意代码，这些代码也只能在沙箱的"笼子"里活动，无法伤害宿主系统。

2. **资源限制** —— 限制沙箱内代码的 CPU 使用率、内存上限和磁盘空间，防止代码失控耗尽系统资源。在实际生产中，最常见的场景是一个沙箱内的代码由于 bug 进入了死循环，如果没有资源限制，它会逐渐吃掉所有 CPU 资源，导致同一台服务器上的其他服务不可用。

3. **环境一致性** —— 每次代码执行都在干净的、可复现的环境中运行。你不用担心用户的代码污染了环境，影响下一次执行的结果。这对于需要多次运行同一段代码并期望得到相同结果的场景（如自动化测试、数据管道）至关重要。

4. **权限控制** —— 精细控制沙箱内的代码能访问哪些网络端点、哪些文件路径。你可以允许沙箱中的代码访问内部 API 但禁止访问外部网络，或者允许读取特定目录但禁止写入，实现细粒度的安全策略。

### 8.2 两种沙箱方案

**用一个类比来理解：**

> 想象你要租房子住。你有两个选择：租一套**独立公寓**（有独立的厨房、卫生间、客厅，完全属于你一个人），或者租一个**酒店单间**（只有一张床和一个小桌子，但价格便宜、拎包入住）。
>
> Daytona 沙箱就像独立公寓——给你一个完整的、隔离的"家"（Docker 容器），什么都有，但需要更多资源来"维护"。Deno 沙箱就像酒店单间——轻量、快速、够用就好，但空间有限，不能做大餐（运行大型应用）。
>
> 你的选择取决于需求：如果需要完整的环境来运行复杂的应用，选 Daytona（独立公寓）；如果只是需要一个安全的地方快速执行一段代码，选 Deno 沙箱（酒店单间）。

Deep Agents 支持两种沙箱实现，分别适用于不同的场景。理解它们的区别有助于你做出正确的选择：

| 特性 | Daytona | Deno Sandbox |
|------|---------|-------------|
| **运行环境** | Docker 容器（云端或本地） | Deno 运行时（进程级） |
| **启动速度** | 中等（需要拉取和启动容器） | 快速（进程级启动） |
| **隔离级别** | 系统级（完整容器隔离） | 进程级（沙箱 API 控制） |
| **网络控制** | ✅ 精细化控制 | ✅ 基本控制 |
| **文件系统** | ✅ 完全隔离 | ✅ 受限访问 |
| **适用场景** | 完整的代码运行环境，需要安装依赖包 | 轻量级脚本执行，快速验证 |

#### 8.2.1 Daytona 沙箱

Daytona 基于 Docker 容器，提供完整的系统级隔离。它最适合需要安装依赖包、运行完整项目、或者需要特定操作系统环境的场景。当你需要沙箱内运行完整的 Node.js 应用、Python 数据处理管道或编译大型项目时，Daytona 是首选方案。它的本质是在 Docker 容器中启动一个完整的操作系统环境，因此可以提供接近原生虚拟机的隔离级别：

```typescript
// Daytona 沙箱配置
// 前置条件：需要在系统上安装 Docker 和 Daytona 服务
// https://github.com/daytonaio/daytona

const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  sandbox: {
    type: "daytona",
    config: {
      image: "node:22",       // 使用 Node.js 22 的 Docker 镜像
      memory: "512m",          // 内存限制为 512MB
      disk: "2gb",             // 磁盘空间限制为 2GB
    },
  },
});
```

#### 8.2.2 Deno Sandbox

Deno 沙箱使用 Deno 运行时内置的沙箱机制，无需 Docker，启动更快，适合轻量级的脚本执行。与 Daytona 不同，Deno 沙箱不创建完整的操作系统级隔离，而是在 Deno 运行时层面通过权限 API 来限制代码的行为。它的优势在于启动速度快（通常在 300ms 以内），资源消耗小，适合低延迟场景：

```typescript
// Deno 沙箱 —— 轻量级代码执行
// 不需要 Docker，启动速度快
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  sandbox: {
    type: "deno",
    config: {
      permissions: {
        env: false,              // 禁止访问环境变量
        net: ["api.github.com"], // 只允许访问 GitHub API
        read: ["/tmp"],          // 只允许读取 /tmp 目录
        write: false,            // 禁止写入文件
      },
    },
  },
});
```

### 8.3 何时需要使用沙箱？

**用一个类比来理解：**

> 想象你开了一家咖啡馆。你不是每个顾客都需要"安保"服务——大多数顾客只是来喝杯咖啡就走。但是，如果有一个顾客说"我要在这里举办一场大型派对，可能会用到你的厨房设备"，那你就需要考虑如何保护你的厨房不被破坏。
>
> 沙箱也是同样的道理。如果 Agent 只是调用外部 API（比如查询天气、翻译文本），数据不经过 Agent 执行环境，风险很低。但如果 Agent 要**执行用户提交的代码**——无论是 JavaScript、Python 还是 Shell 脚本——你就需要沙箱来保护宿主系统。判断是否需要沙箱的核心标准是：**Agent 是否会在本地执行不可信代码**。

并不是所有 Agent 场景都需要沙箱。以下表格帮助判断：

| 场景 | 需要沙箱？ | 说明 |
|------|:---------:|------|
| Agent 只调用外部 API（如天气查询、翻译） | ❌ | Agent 不执行用户代码，没有安全风险 |
| Agent 运行用户提交的代码片段 | ✅ 强烈建议 | 防止恶意代码破坏宿主系统 |
| Agent 执行系统命令（如 ls、git） | ❌（但需要权限控制） | 使用第10章的 FilesystemPermission 做权限管理 |
| Agent 需要安装 npm 包或运行构建脚本 | ✅ 建议使用 | 沙箱提供干净的临时环境，避免污染全局 |
| Agent 处理敏感数据（密码、密钥等） | ✅ 强烈建议 | 沙箱防止数据在执行过程中泄露到外部 |

### 8.4 Daytona 与 Deno 的选择策略

**用一个类比来理解：**

> 想象你要出差。你有一个登机箱和一个双肩包。如果你只是去两天的短期出差，带双肩包就够了——轻便、快速、不用托运。如果你要去一个月，还需要带样品和设备，那登机箱甚至更大的行李箱才有足够的空间。
>
> 选择 Daytona 还是 Deno 沙箱也是类似的决策。核心问题是：**你的代码执行任务有多重？** 轻量级脚本用 Deno（就像双肩包），重量级应用用 Daytona（就像登机箱）。

在实际项目中，选择 Daytona 还是 Deno 沙箱取决于你的具体需求。以下是一些实用的选择策略：

**选择 Daytona 的场景：**
- 你的 Agent 需要运行完整的应用程序（如启动一个 Web 服务器、运行测试套件）
- 你需要安装特定的系统依赖（如 Python 的 numpy 包、Node.js 的 node-canvas）
- 你希望每次执行都在完全干净的系统中进行，不会有任何残留
- 你需要精细控制操作系统级别的资源（CPU 核数、内存上限、磁盘 IO）

**选择 Deno 沙箱的场景：**
- 你的 Agent 只需要执行简单的脚本（如数据清洗、格式转换、数学计算）
- 你对启动速度有要求（Deno 沙箱可以在几百毫秒内启动，通常在 200-500ms 之间）
- 你不需要安装额外的系统依赖
- 你在开发调试阶段，想快速验证沙箱配置是否正确

**混合使用策略：** 在一个需要兼顾开发效率和生产安全的项目中，一个被广泛采用的最佳实践是"开发用 Deno，生产用 Daytona"。在开发阶段，使用 Deno 沙箱快速迭代和调试沙箱配置；在 CI/CD 流程和生产环境中，切换到 Daytona 沙箱获得更强的隔离保障。这种方案可以在不牺牲开发体验的前提下，确保生产环境的最高安全标准。

> **💡 为什么沙箱要提供两种不同的方案？**
> 这是因为安全和效率之间需要权衡。Daytona 提供更强的隔离（系统级容器隔离），但启动速度较慢、资源消耗较大。Deno 沙箱提供较弱的隔离（进程级），但启动速度快、资源消耗小。Deep Agents 同时支持两种方案，让你可以根据实际场景选择合适的安全-效率平衡点。在不需要完全容器隔离的场景中使用 Deno 沙箱可以显著提升响应速度，同时仍然获得足够的安全保障。

> **💡 一个实用的建议：** 在开发阶段先用 Deno 沙箱快速验证功能，等到生产部署时再切换到 Daytona 沙箱获得更强的隔离保障。这样既能享受开发时的高效率，又能保证生产时的安全性。

## 🔨 实战演练

### 练习：配置安全执行环境

**场景描述：**
假设你要部署一个 Agent，允许用户提交 JavaScript 代码片段来运算数据。为了安全，这些代码不能访问文件系统、不能发送网络请求、不能读取环境变量。你需要配置一个 Deno 沙箱来实现这些限制。这是一个典型的"不可信代码执行"场景，在数据分析平台、在线编程教育、自动化工作流等应用中非常常见。

**你的任务：**
1. 使用 Deno 沙箱类型创建一个安全的代码执行环境
2. 配置权限：禁止网络、文件系统和环境变量访问
3. 创建一个执行代码的工具函数
4. 测试沙箱是否能正确阻止越权操作（如尝试读取 /etc/passwd 或发起网络请求）

<details>
<summary>🧑‍💻 先自己写，写完再展开看参考答案</summary>

```typescript
import { createDeepAgent } from "deepagents";
import { tool } from "langchain";
import { z } from "zod";

// 创建代码执行工具
const executeCode = tool(
  async ({ code }: { code: string }) => {
    try {
      // 在沙箱中执行代码（eval 仅作演示，实际使用沙箱运行）
      const result = eval(code);
      return `执行结果: ${JSON.stringify(result)}`;
    } catch (error) {
      return `执行错误: ${error.message}`;
    }
  },
  {
    name: "execute_code",
    description: "在沙箱中安全执行 JavaScript 代码",
    schema: z.object({
      code: z.string().describe("要执行的 JavaScript 代码"),
    }),
  }
);

// 配置带沙箱的 Agent
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  tools: [executeCode],
  sandbox: {
    type: "deno",
    config: {
      permissions: {
        env: false,   // 禁止读取环境变量
        net: false,   // 禁止网络请求
        read: false,  // 禁止读取文件
        write: false, // 禁止写入文件
      },
    },
  },
  systemPrompt: "你是一个安全的代码执行助手。所有代码都在沙箱中运行。",
});

// 测试安全执行
const result = await agent.invoke({
  messages: [
    { role: "user", content: "计算 1 到 100 的和" },
  ],
});
console.log(`Agent: ${result.messages.at(-1)?.content}`);
```

**预期输出：**
```
Agent: 我来帮你计算 1 到 100 的和。使用公式 n(n+1)/2：
1 + 2 + ... + 100 = 100 × 101 ÷ 2 = 5050
```

如果用户尝试恶意操作（如读取 `/etc/passwd`），Agent 会在沙箱中执行并返回权限错误。

</details>

## ⚡ 进阶技巧

### 技巧一：沙箱超时保护

沙箱内的代码如果陷入死循环，由沙箱机制自动超时终止，不会影响主进程。这是一个非常重要的安全特性——即使沙箱内的代码出了问题，你的主 Agent 和宿主系统也不会受到任何牵连。这对于生产环境中的 Agent 部署尤为关键：

```typescript
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  sandbox: {
    type: "deno",
    config: {
      timeout: 30_000,  // 30 秒超时，超时后自动终止执行
    },
  },
});
```

**为什么超时保护很重要？** 在执行不受信任的代码时，最常遇到的风险就是死循环或无限递归。如果没有超时机制，这样的代码会耗尽 CPU 资源，导致整个 Agent 进程无响应。沙箱的超时保护确保即使代码失控，也只是沙箱内部被终止，主进程不受影响。

### 技巧二：沙箱资源限制的配置

除了超时保护，你还可以限制沙箱使用的 CPU 和内存资源，防止代码执行占用过多系统资源。这对于多租户环境尤为重要——如果一个用户的代码占用了所有 CPU，其他用户的服务质量就会受到影响：

```typescript
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  sandbox: {
    type: "daytona",
    config: {
      image: "node:22",
      memory: "512m",    // 限制内存为 512MB，超出将触发 OOM 终止
      cpu: 2,            // 限制最多使用 2 个 CPU 核心
      disk: "2gb",       // 限制磁盘空间为 2GB
    },
  },
});
```

### 技巧三：Daytona 沙箱的自定义镜像

可以为 Daytona 沙箱准备预装依赖的自定义 Docker 镜像，加速启动和减少重复安装。这在团队中特别有用——你可以将团队常用的工具和依赖预先打包到镜像中，每个人使用时无需等待 npm install：

```dockerfile
# Dockerfile 示例 —— 预装常用的开发工具
FROM node:22
RUN npm install -g typescript ts-node prettier
COPY ./scripts /scripts
```

然后在配置中引用这个镜像：

```typescript
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  sandbox: {
    type: "daytona",
    config: {
      image: "my-custom-agent-image:latest",  // 使用自定义镜像，启动速度更快
    },
  },
});
```

### 技巧四：沙箱网络隔离策略

对于处理敏感数据的 Agent，网络隔离是一个重要的安全维度。Daytona 沙箱支持精细的网络访问控制，你可以指定沙箱内的代码只能访问哪些外部服务：

```typescript
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  sandbox: {
    type: "daytona",
    config: {
      image: "node:22",
      network: {
        allowedHosts: ["api.example.com", "database.internal:5432"],
        blockPublicNetworks: true,  // 禁止访问公共互联网
      },
    },
  },
});
```

> **💡 网络隔离的最佳实践：** 如果你的 Agent 只需要访问内部的 API 或数据库，强烈建议启用网络隔离并只开放必要的端点。这可以防止沙箱内的恶意代码将数据传输到外部服务器，是防止数据泄露的关键防线。

**Q1：沙箱的核心作用是什么？**
> A：提供安全隔离的执行环境，防止恶意代码影响宿主系统。同时限制资源使用（CPU、内存、磁盘），防止代码失控。

**Q2：Daytona 和 Deno 沙箱的主要区别是什么？**
> A：Daytona 基于 Docker 容器，提供完整的系统级隔离，适合需要完整运行环境的场景；Deno 沙箱是进程级轻量沙箱，启动速度快，适合轻量级脚本执行。

**Q3：什么情况下应该使用沙箱？**
> A：当 Agent 需要执行用户提交的代码时强烈建议使用沙箱。安装依赖包和处理敏感数据时也建议使用。纯 API 调用场景不需要。

**Q4：沙箱可以限制网络访问吗？**
> A：可以。Deno 沙箱通过 `permissions.net` 控制允许访问的网络端点，Daytona 沙箱通过容器网络配置控制，支持更精细的策略。

**Q5：沙箱超时有什么作用？**
> A：防止代码陷入死循环或执行时间过长耗尽系统资源。超时时间到达后沙箱会自动终止执行，不影响主进程。

**Q6：如何选择 Daytona 还是 Deno 沙箱？**
> A：如果你需要完整的应用运行环境、安装系统依赖或精细的 OS 级资源控制，选择 Daytona；如果只是执行轻量级脚本、对启动速度有要求，选择 Deno 沙箱。开发阶段可先用 Deno 快速验证，生产环境切换到 Daytona 获得更强隔离。

</details>

|------|------|----------|
| 沙箱执行超时 | 代码陷入死循环或执行时间过长 | 设置合理的 timeout 值，或者优化沙箱内代码的性能 |
| 沙箱网络请求被拒绝 | 沙箱网络权限配置了限制（permissions.net=false） | 在沙箱配置中开放需要的网络端点 |
| Daytona 沙箱启动失败 | Docker 服务未运行或沙箱配置错误 | 确认 Docker 已启动，检查 sandbox 配置中的镜像名 |
| Deno 沙箱初始化失败 | 系统未安装 Deno 运行时 | 安装 Deno (https://deno.com) |
| 沙箱内文件读写报错 | 沙箱文件系统权限未开放 | 在 permissions 中配置 read/write 路径白名单 |
| 沙箱内存溢出（OOM） | 代码使用了过多的内存 | 增加沙箱的 memory 配置值，或优化代码内存使用 |
| 自定义镜像拉取失败 | 镜像名错误或镜像仓库需要认证 | 检查镜像名称是否正确，私有仓库需配置认证信息 |

## 📝 本章小结

- ✅ 沙箱为 Agent 代码执行提供安全隔离环境，防止恶意代码破坏宿主系统
- ✅ Daytona 沙箱基于 Docker 容器，提供系统级隔离，适合完整项目运行
- ✅ Deno 沙箱基于进程级隔离，启动更快，适合轻量脚本执行
- ✅ 沙箱可以精细控制代码的 CPU、内存、网络、文件系统访问权限
- ✅ 执行用户提交的代码时强烈建议启用沙箱
- ✅ 沙箱超时机制可以防止死循环耗尽资源
- ✅ 通过自定义 Docker 镜像可以加速沙箱启动，减少重复依赖安装
