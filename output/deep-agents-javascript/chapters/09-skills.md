## 💡 核心概念

### 9.1 什么是技能？

**用一个类比来理解：**

> 你买了一台新手机。手机系统本身是通用的——它可以打电话、发短信、上网、装应用。但要让手机在特定场景下真正"好用"，你需要安装对应的应用：微信（社交）、高德地图（导航）、网易云音乐（娱乐）。每个应用赋予了手机在该领域的专业能力。
>
> 在 Deep Agents 中，**技能（Skill）** 就是 Agent 的"应用"。Agent 的核心能力是通用的——推理、调用工具、对话。但要让 Agent 在特定领域表现出色（比如代码审查、中文润色、客服接待），就需要安装对应的技能。每个技能定义了 Agent 在该领域的角色身份、行为规则和可用工具。

**技能的核心三要素：**

| 要素 | 说明 | 类比 |
|------|------|------|
| **系统提示（System Prompt）** | 定义 Agent 在该领域的角色身份和行为规则 | 应用的"使用说明书" |
| **工具配置（Tools）** | 注册该领域需要的专用工具集 | 应用的"功能模块" |
| **知识文件（Knowledge Files）** | 加载领域相关的参考文档和知识库 | 应用的"数据文件" |

### 9.2 技能文件结构

**用一个类比来理解：**

> 想象你去餐厅点餐。菜单上的每道菜都有三个要素：**菜名**（叫什么）、**图片和描述**（这道菜是什么）、**食材清单**（由什么做成）。你看到"番茄牛腩面"这个名字和图片，就知道这是一道什么样的菜，食材是否合你口味。
>
> 技能文件的 Frontmatter 就是技能"菜单"上的信息。`name` 是菜名、`description` 是菜品描述、`type` 和 `version` 是分类和版本标注。而正文部分（System Prompt、行为规则、Tools）则是完整的"菜谱"——告诉 Agent 这道技能"菜"应该怎么做。

一个技能文件是一个 Markdown 文件，使用 Frontmatter（YAML 格式的元数据头部）定义技能的元信息：

```markdown
---
name: code-reviewer              # 技能名称（唯一标识）
description: 代码审查技能         # 技能描述，说明这个技能的用途
type: skill                       # 类型固定为 skill
version: 1.0.0                    # 版本号，用于版本管理
---

# Code Reviewer Skill

## System Prompt

你是一个资深的代码审查专家。你的职责是审查代码并发现以下问题：

1. **安全性问题**：SQL 注入、XSS 攻击、敏感信息泄露
2. **性能问题**：不必要的循环、内存泄漏、N+1 查询
3. **代码质量**：可读性差、缺乏错误处理、魔法数字
4. **最佳实践**：不符合框架或语言惯例的代码

## 行为规则

- 每次审查按优先级排序问题：安全 > 性能 > 质量 > 风格
- 对每个问题给出三要素：文件位置 + 问题描述 + 修复建议
- 如果没有发现问题，明确说"未发现问题"
- 每个代码审查请求都要审查所有文件，不要遗漏

## Tools

- `search_codebase` —— 搜索代码库中相关的代码片段
- `fetch_file_content` —— 获取完整的文件内容用于审查
```

### 9.3 安装和使用技能

**用一个类比来理解：**

> 想象你买了一台新手机。手机自带了基础功能——打电话、发短信、上网。但如果你想用它来导航，需要安装高德地图；想用它来听歌，需要安装网易云音乐。安装的过程很简单——去应用商店搜索、点击安装、然后就可以使用了。
>
> Deep Agents 的技能安装也是同样的理念。框架本身提供了 Agent 的基础能力，而技能就像手机上的"应用"——通过 `install-skill` 命令安装后，就可以在任意 Agent 中通过 `skills` 参数引用了。

Deep Agents 支持从多种来源安装技能，安装后可以在任意 Agent 中引用：

```bash
# 从本地文件安装
npx deepagents install-skill ./skills/code-reviewer.md

# 从 URL 安装社区技能包
npx deepagents install-skill https://skills.deepagents.ai/code-reviewer

# 从 GitHub 仓库安装
npx deepagents install-skill gh:username/repo/skills/code-reviewer
```

安装后在创建 Agent 时引用技能：

```typescript
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  skills: ["code-reviewer"],  // 引用已安装的技能名称
});
```

### 9.4 技能 vs 工具 vs 中间件

**用一个类比来理解：**

> 想象你在经营一家餐厅：
> - **技能（Skill）** 就像是餐厅的**经营风格**——你做的是"高级法式料理"还是"快捷日式拉面"？这定义了整个餐厅的定位、服务方式和顾客群体。
> - **工具（Tool）** 就像是后厨的**单个设备**——烤箱、炉灶、搅拌机。每个设备完成一个特定的功能。
> - **中间件（Middleware）** 就像是餐厅的**管理流程**——顾客进门时登记、点餐时下单、结账时开发票。这些流程贯穿整个用餐体验。
>
> 在这个类比中，改变经营风格（技能）会影响整体；增加一个设备（工具）只增加单一功能；优化管理流程（中间件）则影响整个运营过程的效率。Deep Agents 的三个概念在本质上也是这样的层级关系。

这三个概念容易混淆。它们的区别在于作用的"层级"不同：

| 维度 | 技能（Skill） | 工具（Tool） | 中间件（Middleware） |
|------|-------------|-------------|-------------------|
| **作用级别** | 整体行为模式 | 单个操作能力 | 流程拦截和处理 |
| **包含内容** | 系统提示 + 工具 + 知识 | 函数 + Schema | 前置/后置逻辑 |
| **复用性** | 跨 Agent 共享 | 跨 Agent 共享 | 跨 Agent 共享 |
| **典型场景** | "让它变成一个代码审查专家" | "让它能搜索网络" | "记录所有调用日志" |

> **💡 一个简单的判断方法：**
> - 如果是在**改变 Agent 的身份和行为方式** → 用技能（Skill）
> - 如果是在**给 Agent 增加一个单一能力** → 用工具（Tool）
> - 如果是在**监控或拦截 Agent 的运行过程** → 用中间件（Middleware）

> **💡 为什么会设计这三个层次？**
> 这种分层设计的核心原因是**关注点分离（Separation of Concerns）**。如果 Agent 的所有行为定义、能力扩展和流程控制都混在一起，系统会变得难以维护和扩展。技能层负责"是什么"（Agent 的身份和角色）、工具层负责"能做什么"（具体的功能操作）、中间件层负责"怎么运行"（流程和监控）——每一层只关注自己的职责，层与层之间通过清晰的接口协作。这种架构让 Deep Agents 既能灵活扩展（通过添加工具），又能整体切换行为模式（通过更换技能），还能精细控制运行过程（通过配置中间件），三者各司其职，互不干扰。

### 9.5 组合多个技能

**用一个类比来理解：**

> 想象你是一个全能型选手——你既会写代码，又会写文章，还会做设计。当你接到一个"开发一个产品 landing page"的任务时，你会同时运用编程、文案和设计三种能力。它们不是互相排斥的，而是互补的——编程负责实现功能，文案负责传达信息，设计负责视觉呈现。
>
> 组合多个技能也是同样的道理。你可以给一个 Agent 同时配置"代码审查"和"中文润色"两个技能——当 Agent 审查完代码后，输出的审查报告会自动经过中文润色技能的优化，让报告读起来更流畅自然。

多个技能可以同时使用，系统提示会自动合并。这让你可以快速构建具备复合能力的 Agent：

```typescript
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  skills: ["code-reviewer", "chinese-polisher"],
  // 合并后，Agent 既能审查代码，又能在输出时优化中文表达
});
```

**技能合并的规则：**
- 多个技能的系统提示按 `skills` 数组的顺序拼接在一起
- 如果多个技能定义了相同的工具，以最后出现的技能为准
- 技能的系统提示在 Agent 内置的系统提示之后追加
- 合并后的总系统提示长度不应超出模型的最大上下文长度限制值

> **💡 组合技能时的注意事项：**
> 并非所有技能都能随意组合。如果两个技能定义了冲突的行为规则（例如一个说"回答要保持简洁"，另一个说"回答要非常详细"），Agent 可能会感到困惑。在组合技能时，建议选择互补的技能而非矛盾的技能。一个实用技巧是：检查每个技能的系统提示关键词，如果发现"简洁"和"详细"同时出现，说明可能存在冲突。此外，还要注意总系统提示长度——组合 3 个以上技能时，系统提示很容易超过 4000 token，建议使用 `list-skills` 查看每个技能的系统提示长度，评估组合后的总长度是否在模型的可接受范围内。

### 9.6 技能的查找和管理

**用一个类比来理解：**

> 想象你的手机上安装了 50 个应用。时间长了，有些应用你可能几个月都没打开过，但它们仍然占用着存储空间、在后台消耗着系统资源。所以你会定期清理——看看哪些应用还在用、哪些可以卸载、哪些需要更新到最新版本。
>
> 技能的管理也是类似的理念。安装多个技能后，定期检查已安装的技能列表是一个好习惯——移除不再使用的技能可以减少 Agent 的上下文负担（因为技能的系统提示会合并到总提示中），避免技能之间的行为冲突。

需要注意的是，技能的合并不是简单的字符串拼接——框架在合并时会自动去重相同的工具定义，避免 Agent 调用时产生歧义。同时，如果有两个技能定义了相同的工具名称但实现不同，后面的技能会覆盖前面的，这是为了保证行为的一致性。在设计技能时，尽量避免定义与其他流行技能同名的工具，以减少冲突的可能性。

安装了多个技能后，你可能需要查看哪些技能可用、检查版本、或者卸载不再需要的技能。Deep Agents 提供了一些管理命令：

```bash
# 列出所有已安装的技能
npx deepagents list-skills
# 输出示例：
# code-reviewer (v1.0.0) — 代码审查技能
# chinese-polisher (v1.0.0) — 中文润色技能

# 卸载不再需要的技能
npx deepagents uninstall-skill chinese-polisher

# 搜索可用的社区技能
npx deepagents search-skills database
```

> **💡 技能管理的最佳实践：**
> 建议定期检查已安装的技能列表，移除不再使用的技能。过多的技能会增加 Agent 的上下文长度（因为系统提示会合并），也可能导致行为规则之间的冲突。保持技能列表的精简，有助于 Agent 更准确地遵循指令。

## 🔨 实战演练

### 练习：创建中文润色技能

**场景描述：**
你的团队经常需要润色中文文档——修正语法错误、优化表达、保持风格一致。创建一个"中文润色"技能，让安装了这个技能的 Agent 具备专业中文编辑的能力。

**你的任务：**
1. 编写一个名为 `chinese-polisher` 的技能文件
2. 定义清晰的系统提示，包括润色原则和输出格式
3. 安装该技能并在 Agent 中使用

<details>
<summary>🧑‍💻 先自己写，写完再展开看参考答案</summary>

```markdown
---
name: chinese-polisher
description: 中文润色技能 —— 优化中文文本的语法、表达和风格
version: 1.0.0
---

# Chinese Polisher Skill

## System Prompt

你是一个专业的中文编辑。你的任务是根据用户提供的中文文本进行润色。

## 润色原则

1. **修正语法错误**：检查主谓宾搭配、标点使用、量词搭配是否正确
2. **优化表达**：去掉冗余词、合并重复句、提升整体可读性
3. **保持风格**：不改变原文的语气和风格（正式/口语/技术/文学）
4. **保留专有名词**：不修改人名、地名、产品名、代码片段和技术术语
5. **最少改动原则**：只修改必要的内容，不做过度润色

## 输出格式

每次润色请按以下格式输出：
- **原文**：`[引用的原文片段]`
- **润色后**：`[修改后的文本]`
- **修改说明**：简要解释每处改动的理由
```

安装并测试：
```bash
# 安装本地技能文件
npx deepagents install-skill ./chinese-polisher.md
```

```typescript
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  skills: ["chinese-polisher"],  // 通过技能名称引用已安装的技能
  // Agent 现在具备了中文润色能力，会自动优化所有输出的中文文本
});
```

**预期输出：**
```
当用户输入一段需要润色的中文文本（如"这个功能非常的好用，我非常的喜欢"），
Agent 会输出：
- 原文：这个功能非常的好用，我非常的喜欢
- 润色后：这个功能非常好用，我非常喜欢
- 修改说明：删除了冗余的"的"字，使表达更简洁流畅
```

</details>

## ⚡ 进阶技巧

### 技巧一：技能版本管理与升级策略

技能文件支持版本号，可以在不同版本之间切换。这在你需要迭代更新技能定义时非常有用，你可以安全地升级到新版本而不影响现有配置和其他依赖：

```bash
# 查看已安装的技能和版本
npx deepagents list-skills

# 安装特定版本
npx deepagents install-skill code-reviewer@2.0.0
```

**什么时候需要升级技能版本？** 当官方发布了技能的更新版本，修复了 bug、增加了新的工具、或者优化了系统提示时，你应该考虑升级到新版本。升级前建议查看详细的变更记录，确认新版本与你的现有配置是否兼容，避免升级后出现意外或不兼容的问题。

### 技巧二：分享你自己编写的技能

你编写的技能文件可以分享给团队或社区使用。技能文件是纯文本的 Markdown 格式，因此非常容易分享和传播——不需要编译、不需要打包、不需要复杂的安装步骤，一个文件即可搞定一切，团队成员拿到文件后即可直接安装使用：

```bash
# 分享给队友 —— 直接把文件发给队友，他安装即可
npx deepagents install-skill ./my-custom-skill.md

# 分享到社区
# 1. 确保 Frontmatter 中填写了 name、description、version 字段
# 2. 上传到 GitHub 公共仓库
# 3. 其他人可以通过 URL 直接安装
npx deepagents install-skill https://raw.githubusercontent.com/user/skills/main/my-skill.md
```

### 技巧三：从社区获取技能

Deep Agents 社区提供了丰富的技能包，可以直接安装使用：

```bash
# 搜索社区技能
npx deepagents search-skills code-review

# 安装社区技能
npx deepagents install-skill @community/code-reviewer
```

### 技巧四：编写高质量技能的最佳实践

要编写一个高质量的技能文件，除了遵循基本的 Frontmatter 格式外，还需要注意以下几点：

1. **系统提示要具体** —— 不要写"你是一个助手"这样模糊的提示。应该明确指定：你的角色身份是什么、你遵循什么规则、你输出的格式是什么。越具体，Agent 的行为越可控。
2. **工具选择要精准** —— 只给技能注册它真正需要的工具。多余的工不仅会增加 Agent 的决策负担，还可能引入不必要的安全风险。
3. **版本号要有语义** —— 使用 semver 语义化版本号（major.minor.patch），并在升级时遵循"大版本号不兼容时递增 major，添加功能时递增 minor，修复 bug 时递增 patch"的规则。
4. **技能描述要可搜索** —— `description` 字段会出现在 `search-skills` 的搜索结果中，使用包含关键技术的描述（如"React 组件开发技能"比"前端技能"更容易被找到）。

```typescript
// 一个高质量技能文件的代码引用示例
const agent = createDeepAgent({
  model: "anthropic:claude-sonnet-4-6",
  skills: ["code-reviewer"],  // 引用经过良好设计的技能
  // Agent 安装技能后，会自动获得该技能定义的系统提示、行为规则和工具集
});
```

## 🧠 知识检查点

<details>
<summary>点击展开答案</summary>

**Q1：技能的核心三要素是什么？**
> A：系统提示（定义角色和行为规则）、工具配置（注册该领域需要的专用工具）、知识文件（加载领域相关的参考文档）。

**Q2：技能和工具的区别是什么？**
> A：技能定义 Agent 在某一领域的整体行为模式（包含系统提示、工具和知识），工具是单个可调用的外部功能。技能建立在工具之上，为工具的组合使用提供上下文和规则。

**Q3：如何安装和使用社区技能？**
> A：使用 `npx deepagents install-skill` 命令从本地文件、URL 或 GitHub 安装，然后在创建 Agent 时通过 `skills: ["技能名称"]` 引用。

**Q4：多个技能同时使用时，系统提示如何工作？**
> A：多个技能的系统提示会自动按顺序拼接合并，Agent 会同时遵循所有技能的规则。如果出现冲突工具定义，以最后出现的技能为准。

**Q5：技能有版本管理吗？**
> A：有。技能文件支持 version 字段，可以通过 `install-skill name@version` 安装特定版本。

**Q6：技能和中间件可以互相替代吗？**
> A：不能。它们解决的是不同层次的问题：技能定义"Agent 是什么角色"（行为层面），中间件控制"Agent 如何运行"（流程层面）。一个改变行为方式，一个监控执行过程，两者是互补关系而非替代关系。

**Q7：如何选择技能的组合策略？**
> A：选择互补而非矛盾的技能。检查每个技能的系统提示是否定义了冲突的行为规则（如一个要求简洁、一个要求详细）。组合前建议先单独测试每个技能的效果，确认无误后再进行组合。

</details>

## 🐛 常见错误

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| 技能未生效 | 技能未正确安装或技能名称拼写错误 | 使用 `npx deepagents list-skills` 检查已安装的技能列表 |
| 多个技能行为冲突 | 多个技能定义了互相矛盾的行为规则 | 检查各技能的系统提示，避免冲突指令；必要时移除冲突技能 |
| 技能文件安装失败 | 路径错误或 Frontmatter 格式不正确 | 检查技能文件路径和 Frontmatter 格式（--- 包裹的 YAML） |
| 技能版本不兼容 | 安装的技能版本与当前 Deep Agents 版本不匹配 | 升级技能或回退到兼容版本 |
| 技能安装后找不到 | 技能安装路径未被正确识别或权限不足 | 使用 `list-skills` 查看安装状态，确认技能名称拼写正确 |

## 📝 本章小结

- ✅ 技能是 Agent 的"领域应用"，赋予 Agent 特定领域的专业能力
- ✅ 技能文件使用 Markdown + Frontmatter 格式定义系统提示、工具和知识
- ✅ 通过 `install-skill` 命令安装，在 Agent 中通过 `skills` 参数引用
- ✅ 技能是"行为模式"（角色+规则），工具是"单个功能"（操作）
- ✅ 多个技能可以组合使用，系统提示自动合并，框架自动去重工具定义
- ✅ 技能支持版本管理，可以安装特定版本
- ✅ 社区提供了丰富的技能包，可以通过 `search-skills` 发现和安装
- ✅ 设计技能时应遵循"提示具体、工具精准、描述可搜索"的最佳实践
