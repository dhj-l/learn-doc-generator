# doc-checker - 文档质量检查与修复器

用户希望检查的文档目录：**$ARGUMENTS**

如果 `$ARGUMENTS` 为空，请主动询问用户要检查哪个目录。通常路径格式为 `output/[技术名称]/`。

## 触发条件

当用户需要检查或修复已生成的学习文档时触发此 Skill。例如：
- "检查一下生成的文档"
- "doc-check output/xxx"
- "看看文档有没有问题"
- "修复文档"
- "/doc-check output/xxx"

## 核心职责

你是一个严格的文档质量审计员，依据 `learn-doc-generator.md` 中定义的**所有规范**，逐文件、逐规则地检查生成文档是否合规。只要发现**任何一条**不符合要求的地方，都必须记录并在修复阶段处理。

---

## 第一阶段：扫描与报告

### 1.0 预备工作

1. 确认目标目录存在且包含文档文件
2. 列出目录下所有 `.md` 文件，建立文件清单
3. 识别目录结构模式：单 topic（`output/[名称]/`）或多 topic（`output/[名称]/stage-N/X.N-topic/`）
4. 统计总文件数、章节数、附录数

### 1.1 检查规则清单

以下规则**全部来源于 `learn-doc-generator.md`** 的规范要求。每条规则标注了严重程度：

- **P0（必须修复）**：结构性缺失、功能不可用
- **P1（建议修复）**：内容深度不足、格式不规范
- **P2（可选优化）**：风格、措辞问题

---

#### 📂 A. 目录结构检查

对**每个 topic 目录**（包含 `chapters/` 子目录的那一层）执行以下检查：

| 编号 | 检查项 | 如何检查 | 严重程度 |
|------|--------|----------|----------|
| A-01 | `README.md` 是否存在 | 检查 topic 目录根下是否有 `README.md` | P0 |
| A-02 | `chapters/` 目录是否存在 | 检查是否有 `chapters/` 子目录 | P0 |
| A-03 | `appendix/` 目录是否存在 | 检查是否有 `appendix/` 子目录 | P0 |
| A-04 | `appendix/cheatsheet.md` 是否存在 | 检查文件是否存在且非空 | P0 |
| A-05 | `appendix/troubleshooting.md` 是否存在 | 检查文件是否存在且非空 | P0 |
| A-06 | 第一章文件名是否为 `01-introduction.md` | 检查 `chapters/` 目录下最小编号文件的名称 | P1 |
| A-07 | 最后一章文件名是否为 `NN-capstone-*.md` | 检查 `chapters/` 目录下最大编号文件的名称是否包含 `capstone` | P1 |
| A-08 | 章节文件编号是否连续（01, 02, 03...） | 检查文件名中的两位数字前缀是否连续 | P1 |

---

#### 📝 B. 章节模板检查（逐文件）

对 `chapters/` 下的**每个** `.md` 文件执行以下检查。检查顺序就是这些 H2 标题在文件中应有的顺序：

| 编号 | 检查项 | 如何检查 | 严重程度 |
|------|--------|----------|----------|
| B-01 | `# 第X章` 标题是否存在 | 检查文件是否以 `# 第` 开头的 H1 标题 | P0 |
| B-02 | `> 预计学习时间：` 是否存在 | 检查 H1 后是否有 blockquote 包含学习时间 | P1 |
| B-03 | `## 本章目标`（含 🎯）是否存在 | 检查是否有该 H2 标题，内容为无序列表 | P0 |
| B-04 | `## 前置知识`（含 📋）是否存在 | 检查是否有该 H2 标题（第一章除外） | P1 |
| B-05 | `## 核心概念`（含 💡）是否存在 | 检查是否有该 H2 标题 | P0 |
| B-06 | 核心概念下每个概念是否有类比引入 | 检查每个 `### 概念` 下是否包含类比描述（如"想象"、"好比"、"就像"等关键词，或明确的类比段落） | P1 |
| B-07 | 核心概念下是否有代码示例 | 检查核心概念部分是否包含 ` ``` ` 代码块 | P0 |
| B-08 | 核心概念下是否有"为什么这样做" | 检查是否有 `**为什么这样做**` 或 `💡 为什么这样做` 的 blockquote | P1 |
| B-09 | `## 实战演练`（含 🔨）是否存在 | 检查是否有该 H2 标题 | P0 |
| B-10 | 实战演练是否有 `<details>` 折叠 | 检查实战演练部分是否包含 `<details>` 标签 | P1 |
| B-11 | 实战演练是否有场景描述和任务步骤 | 检查是否有 `**场景描述**` 和 `**你的任务**` | P1 |
| B-12 | `## 进阶技巧`（含 ⚡）是否存在 | 检查是否有该 H2 标题 | P1 |
| B-13 | `## 知识检查点`（含 🧠）是否存在 | 检查是否有该 H2 标题 | P0 |
| B-14 | 知识检查点是否有 `<details>` 折叠答案 | 检查知识检查点部分是否包含 `<details>` 和 Q&A 格式 | P1 |
| B-15 | `## 常见错误`（含 🐛）是否存在 | 检查是否有该 H2 标题 | P0 |
| B-16 | 常见错误是否为三列表格（错误/原因/解决方案） | 检查是否包含 `| 错误 | 原因 | 解决方案 |` 表头 | P0 |
| B-17 | `## 本章小结`（含 📝）是否存在 | 检查是否有该 H2 标题 | P0 |
| B-18 | 本章小结是否使用 ✅ 勾选列表 | 检查本章小结部分是否包含 `- ✅` 列表项 | P1 |
| B-19 | `## 下一章预告`（含 ➡️）是否存在 | 检查是否有该 H2 标题（最后一章/ capstone 章除外） | P1 |
| B-20 | 下一章预告是否包含链接 | 检查下一章预告部分是否有指向下一章文件的链接 `[xxx](./xx-xxx.md)` | P0 |
| B-21 | H2 章节的顺序是否正确 | 检查各 H2 标题的出现顺序是否符合模板定义的顺序 | P1 |

#### 📖 C. 内容深度检查

| 编号 | 检查项 | 如何检查 | 严重程度 |
|------|--------|----------|----------|
| C-01 | 每章非代码部分字数 ≥ 2000 字 | 统计文件中非代码块（排除 ` ``` ` 之间的内容）的中文字符数 | P1 |
| C-02 | 核心概念中每个知识点至少 2-3 个代码示例 | 统计每个 `### 概念` 下的代码块数量 | P1 |
| C-03 | 代码是否完整可运行（含 import） | 检查 TypeScript/JavaScript 代码块是否包含 `import` 语句或完整声明 | P0 |
| C-04 | 代码是否有中文注释 | 检查代码块中是否包含 `//` 或 `/* */` 后跟中文内容 | P1 |
| C-05 | 代码是否有预期输出 | 检查代码块后是否有 `**预期输出**` 或 `输出` 的代码块 | P1 |
| C-06 | 是否包含无意义的 hello world 示例 | 检查代码中是否包含 "Hello World"、"hello world"、"打印 1-100" 等 | P0 |
| C-07 | 实战内容占比是否 ≥ 60% | 估算实战（实战演练 + 进阶技巧中的代码部分）与理论的比例 | P1 |

| 编号 | 检查项 | 如何检查 | 严重程度 |

#### 📑 E. 附录检查

| 编号 | 检查项 | 如何检查 | 严重程度 |
|------|--------|----------|----------|

| E-02 | 是否按使用频率排序（而非字母顺序） | 检查内容的组织逻辑（此项为软性检查，记录即可） | P2 |

#### ✍️ H. 写作原则检查

| 编号 | 检查项 | 如何检查 | 严重程度 |

| H-05 | 是否存在理论大量堆砌而缺乏实战 | 检查核心概念部分是否只是罗列概念而没有代码和实践 | P1 |

### 1.2 执行扫描

**按以下顺序逐目录、逐文件执行检查：**

1. **遍历所有 topic 目录**
   ```
   对每个包含 chapters/ 子目录的目录：
     1. 执行 A 类检查（目录结构）
     2. 执行 D 类检查（README）
     3. 执行 E 类检查（附录）
     4. 对 chapters/ 下每个 .md 文件：
        a. 如果文件名含 "capstone"：执行 B + C + F 类检查
        b. 否则：执行 B + C 类检查
     5. 执行 G 类检查（交叉引用）
   ```

2. **遍历所有章节文件执行 H 类检查**（写作原则）

3. **汇总所有问题**，按严重程度排序

## 第二阶段：用户确认修复策略

检查报告输出后，询问用户：

```
📋 检查完成！共发现 N 个问题（P0: X 个, P1: Y 个, P2: Z 个）

请选择修复范围：
  1. 🔧 修复全部问题（P0 + P1 + P2）
  2. 🔧 仅修复 P0 + P1 问题
  3. 🔧 仅修复 P0 问题（结构性缺失）
  4. 🔧 指定修复某些文件/Topic
  5. 📄 仅查看详细报告，暂不修复

请告诉我你的选择。
```

## 第三阶段：修复

### 3.0 修复原则

1. **增量修复**：不删除已有内容，在缺失位置插入符合规范的内容
2. **保持上下文**：修复内容必须与该章节的主题和技术栈一致
3. **风格一致**：修复内容的风格（emoji、格式、语气）必须与文件中已有的部分一致
4. **逐文件修复**：每修复一个文件后，立即重新检查该文件，确保修复有效且未引入新问题
5. **修复顺序**：先修复 P0 → 再修复 P1 → 最后修复 P2

### 3.1 各类问题的修复策略

---

#### 📂 A. 目录结构问题修复

**A-03/A-04/A-05 — appendix/ 目录或附录文件缺失**

1. 创建 `appendix/` 目录
2. 根据 `chapters/` 下所有章节的内容，提取关键技术点：
   - 生成 `cheatsheet.md`：以表格/卡片形式列出该 topic 中最常用的 API，按使用频率排序，每个 API 附带一行最简代码示例
   - 生成 `troubleshooting.md`：收集该技术最常见的 15-20 个错误/问题，每个包含错误信息、原因分析、解决方案
3. 格式严格遵循 `learn-doc-generator.md` 中 2.5 节的规范

**A-06 — 第一章文件名不规范**

将现有最小编号文件重命名为 `01-introduction.md`，更新所有引用该文件名的链接。

**A-07 — 最后一章文件名不含 capstone**

将现有最大编号文件重命名为 `NN-capstone-[主题名].md`，更新所有引用。

**A-08 — 章节编号不连续**

重命名文件使编号连续，更新所有内部链接引用。

---

#### 📝 B. 章节模板问题修复

**B-01 — 缺失 `# 第X章` 标题**

在文件最开头插入 `# 第X章 [根据文件内容推断的章节标题]`。

**B-02 — 缺失预计学习时间**

在 H1 标题后插入 `> 预计学习时间：XX 分钟`（根据内容长度估算）。

**B-03 — 缺失 `## 本章目标`**

在预计学习时间后、前置知识前插入：
```markdown
## 🎯 本章目标

学习完本章，你将能够：
- [根据章节内容提取 3-5 个具体能力目标]
```

**B-04 — 缺失 `## 前置知识`**（第一章除外）

在本章目标后、核心概念前插入：
```markdown
## 📋 前置知识

> 如果你还没有学习以下内容，建议先完成：
> - [上一章的标题和链接](./XX-xxx.md) 的核心概念部分
```

**B-05/B-06/B-07/B-08 — 核心概念相关问题**

- 如果完全缺失 `## 核心概念`：在前置知识后插入完整的 `## 💡 核心概念` 部分，包含 2-4 个概念，每个概念有类比引入、概念讲解、代码示例、"为什么这样做"
- 如果有核心概念但缺少类比引入：在每个 `### 概念` 开头插入一个生活化类比
- 如果缺少代码示例：在概念讲解后插入完整的、可运行的代码示例
- 如果缺少"为什么这样做"：在代码示例后插入 `> **💡 为什么这样做？**` blockquote

**B-09/B-10/B-11 — 实战演练相关问题**

- 如果完全缺失：在核心概念后插入完整的 `## 🔨 实战演练` 部分，包含真实业务场景、任务步骤、`<details>` 折叠的参考代码和预期输出
- 如果缺少折叠：用 `<details><summary>🧑‍💻 先自己写，写完再展开看参考答案</summary>` 包裹参考代码
- 如果缺少场景描述和任务步骤：补充 `**场景描述：**` 和 `**你的任务：**` 部分

**B-12 — 缺失 `## 进阶技巧`**

在实战演练后插入 `## ⚡ 进阶技巧`，包含 2-3 个 H3 子标题的进阶技巧。

**B-13/B-14 — 知识检查点相关问题**

- 如果完全缺失：在进阶技巧后插入 `## 🧠 知识检查点`，包含 3-5 个 Q&A，用 `<details>` 折叠答案
- 如果缺少折叠：用 `<details><summary>点击展开答案</summary>` 包裹

**B-15/B-16 — 常见错误相关问题**

- 如果完全缺失：在知识检查点后插入 `## 🐛 常见错误`，包含三列表格
- 如果格式不符：修改为标准的 `| 错误 | 原因 | 解决方案 |` 表格格式

**B-17/B-18 — 本章小结相关问题**

- 如果完全缺失：在常见错误后插入 `## 📝 本章小结`，使用 `- ✅ [要点]` 格式
- 如果未使用 ✅ 列表：修改为 `- ✅ [要点]` 格式

**B-19/B-20 — 下一章预告相关问题**

- 如果完全缺失：在本章小结后插入 `## ➡️ 下一章预告`
- 如果缺少链接：补充指向下一章文件的 Markdown 链接

**B-21 — H2 章节顺序不符**

按正确顺序重新排列各 H2 部分：本章目标 → 前置知识 → 核心概念 → 实战演练 → 进阶技巧 → 知识检查点 → 常见错误 → 本章小结 → 下一章预告

#### 📖 C. 内容深度问题修复

**C-01 — 字数不足（< 2000 字）**

对内容进行扩写：
- 在核心概念中增加更详细的解释和更多的类比
- 增加更多代码示例（每个知识点 2-3 个）
- 在实战演练中增加更多步骤和更详细的说明
- 在进阶技巧中增加更多实用技巧

**C-02 — 每个知识点代码示例不足**

为缺少代码的知识点补充完整的、可运行的代码示例。

**C-03 — 代码不完整（缺少 import）**

补充完整的 import 语句和必要的配置代码，确保代码可以直接复制运行。

**C-04 — 代码缺少中文注释**

在代码的关键行添加中文注释，解释"为什么这样做"。

**C-05 — 代码缺少预期输出**

在每个代码块后添加 `**预期输出：**` 代码块。

**C-06 — 包含无意义的 hello world 示例**

替换为真实业务场景的示例代码。

**C-07 — 实战内容占比不足 60%**

增加实战演练和进阶技巧的内容量，缩减纯理论讲解。

**D-02/D-03 — 导航表缺失或链接失效**

### 3.2 修复验证

每修复完一个文件后：

1. 重新读取该文件
2. 针对该文件之前报告的所有问题，逐一验证是否已修复
3. 检查修复操作是否引入了新的问题（如链接失效、格式破坏等）
4. 如果修复无效或引入了新问题，重新修复

## 典型工作流程

```
用户输入："检查 output/agent-development/"

→ Step 1: 确认目录存在，建立文件清单
→ Step 2: 按 A~H 类规则逐文件扫描
→ Step 3: 汇总问题，生成检查报告
→ Step 4: 展示报告，询问用户修复范围
→ Step 5: 用户确认后，按 P0 → P1 → P2 顺序逐文件修复
→ Step 6: 每修复一个文件后重新验证
→ Step 7: 输出修复完成报告
→ Step 8: 建议用户重新运行检查确认
```

> **注意**：修复时生成的内容必须基于该 topic 的实际技术主题。
> 如果修复过程中需要了解某个技术的最新 API，可以使用 `context7` MCP 或 `WebSearch` 工具获取。
